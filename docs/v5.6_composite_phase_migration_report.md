## 2. Phase 3 前后对比

### Before (V4.14)

```text
decay_and_recovery 是一个 composite black box。

运行时顺序执行 4 个 inline 操作：
  state.memories = memoriesBefore.map(decayMemory)
  state.proceduralRoutines = proceduralRoutinesBefore.map(decayProceduralRoutine)
  state.rewardState = recoverRewardBaseline(state.rewardState, daysElapsed)
  state.boundary = recoverBoundary(state.boundary, daysElapsed)

Trace 层面：
  phases[2] = {
    name: "decay_and_recovery",
    changedStates: ["memories", "proceduralRoutines", "rewardState", "boundary"],
    reasons: [3 条硬编码 reason]
  }

问题：
  - 4 个独立操作被压入 1 个 phase trace
  - 无法区分每个子操作的独立效果
  - reasons 手工拼接，不来自任何 V3 函数返回值
  - Benchmark System (V6) 无法测量各子操作的独立贡献
```

### After (V5.5)

```text
decay_and_recovery 仍是 1 个 V3 phase。

4 个子操作继续在 Phase 3 内部 inline 顺序执行。

Trace 层面新增 additive subProcesses 字段：
  phases[2] = {
    name: "decay_and_recovery",             // unchanged
    changedStates: [...],                    // unchanged
    reasons: [...],                          // unchanged
    subProcesses: [                          // NEW — additive
      MemoryDecaySubProcessTrace,            // V5.2
      ProceduralDecaySubProcessTrace,        // V5.3
      BoundaryRecoverySubProcessTrace,       // V5.4 (D10-aware)
      RewardRecoverySubProcessTrace          // V5.5 (D10-aware)
    ]
  }

每个 subprocess 包含：
  - id / kind / label
  - reads / writes / changedStates
  - reasons（包含 D10 notice if applicable）
  - metrics（kind-specific numeric measurements）
```

## 4. D10 Overwrite Semantics

```text
boundary_recovery 和 reward_recovery 是 D10-aware subprocesses。

Phase 3 执行 recover 后，Phase 4 homeostasis 会覆盖结果：

  Phase 3:  state.boundary   = recoverBoundary(state.boundary, daysElapsed)
            state.rewardState = recoverRewardBaseline(state.rewardState, daysElapsed)

            → boundaryRecoverySubProcess 记录 recoverBoundary 的 pre/post
            → rewardRecoverySubProcess   记录 recoverRewardBaseline 的 pre/post

  Phase 4:  state.boundary   = homeostasisResult.trace.regulatedBoundary
            state.rewardState = homeostasisResult.trace.regulatedRewardState

            → overwrites Phase 3 intermediate

最终 tick boundary 和 rewardState：
  state.boundary === homeostasisResult.trace.regulatedBoundary      (不是 Phase 3 intermediate)
  state.rewardState === homeostasisResult.trace.regulatedRewardState (不是 Phase 3 intermediate)

D10 验证测试：
  - V4.12: 4 D10 tests (homeostasis overwrite equivalence)
  - V5.4: 2 D10 tests (boundary_recovery subProcess + overwrite preservation)
  - V5.5: 2 D10 tests (reward_recovery subProcess + overwrite preservation)
```

## 8. V5 → V6 路线

### V6 Benchmark 入口建议

```text
V5 完成后的下一步是 V6 Benchmark System。

V6 应该从 Design Charter 开始（参考 V5.0 模式）。
不要马上写大量 benchmark 数据。

V6.0 Design Charter 应定义：

1. Benchmark Target
   - 每个 subprocess 的可见指标（V5 已提供）
   - e.g. memory_decay 对 trust drift 的独立贡献

2. Metric Schema
   - 每个 benchmark scenario 的输入/输出/expected direction
   - 基于 V5 subProcess metrics

3. Replay Fixture Format
   - 可复现的 event sequence
   - control vs experimental tick 的对比

4. Expected Direction Assertions
   - "memory decay 使 averageRecency 下降"
   - "boundary recovery 使 stressLoad 下降但 homeostasis 可覆盖"
   - "same events on different characters produce different subProcess deltas"

5. Tolerance Policy
   - 浮点精度容差
   - character 差异的合理范围
   - 不要求 exact equality between characters

6. Regression Workflow
   - benchmark 失败时的诊断路径
   - rollback vs investigate vs recalibrate
```

### 为什么不应跳过 V5 直接做 Benchmark

```text
如果跳过 V5，benchmark 面对的是 composite decay_and_recovery 黑箱：

  "decay_and_recovery 对 trust 的影响是 X"

但无法区分 X 中：
  - 多少来自 memory decay（记忆淡化）
  - 多少来自 boundary recovery（伤口愈合）
  - 多少来自 procedural decay（习惯消退）
  - 多少来自 reward recovery（奖励基线回归）

V5 的 subProcess decomposition 为 V6 提供了 granularity。
V6 可以在 subprocess level 做 targeted benchmark。
```
