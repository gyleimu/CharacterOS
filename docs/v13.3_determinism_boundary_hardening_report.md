# V13.3 — Determinism Boundary Hardening Report

**Date:** 2026-07-07
**Status:** ✅ PASS — Release Ready

## Why CharacterOS Needs Deterministic Boundaries

CharacterOS core is a computational personality simulation — its outputs must be reproducible, auditable, and verifiable. When two runs of the same Explorer/Agent/LLM boundary code produce different outputs from identical inputs, it becomes impossible to:

1. Debug regressions (was the output different because of a code change or because `Date.now()` ticked?)
2. Audit decision trails (audit IDs must be stable for traceability)
3. Verify release candidates (RC gates require deterministic results)
4. Trust preview paths (preview should match actual application)

V13.3 hardens this boundary by formally auditing all core default paths for non-deterministic sources (`Date.now()`, `Math.random()`, `new Date()` defaults) and ensuring they are either made deterministic or explicitly classified as allowed runtime sources.

## What Was Done

### 1. Deterministic Helpers (`src/core/deterministicHelpers.ts`)

New module providing content-based deterministic ID generation:

| Function | Purpose |
|----------|---------|
| `DETERMINISTIC_TIMESTAMP` | Fixed sentinel timestamp `"1970-01-01T00:00:00.000Z"` for "unknown" occurrence time |
| `deterministicId()` | Content-hash based ID generation (djb2 variant) |
| `deterministicDraftId()` | Draft ID from natural language input + tags |
| `deterministicSessionId()` | Session ID from characterId + inputMode |
| `deterministicTurnId()` | Turn ID from sessionId + content |
| `deterministicReplyPlanId()` | Reply plan ID from intent + grounded facts |
| `deterministicAuditId()` | Audit ID from seed |
| `deterministicCandidateId()` | Candidate ID from draft + extraction method |
| `deterministicWritebackId()` | Writeback ID from turnId |

### 2. Determinism Boundary Audit (`src/core/audit/determinismBoundaryAudit.ts`)

New audit module that:

- **Static scans** `src/core/` and `src/services/` for `Date.now()`, `Math.random()`, `new Date()` patterns
- **Classifies** findings as `allowed` / `warning` / `failure` using a tiered allowlist
- **Runs replay tests** — calls each builder twice with same input, verifies JSON output is identical
- **Outputs structured results** — auditVersion, checkedModules, forbiddenFindings, allowedRuntimeSources, replay results

### 3. Source Fixes — Deterministic Defaults

The following builder functions now use deterministic defaults instead of `Date.now()` / `new Date()`:

| File | Change |
|------|--------|
| `explorerDtoBuilders.ts` | `occurredAt` default → `DETERMINISTIC_TIMESTAMP` |
| `eventStudioPreview.ts` | `draftId` fallback → `deterministicDraftId()` (4 locations) |
| `eventStudioApply.ts` | `auditId` fallback → `deterministicAuditId()` (content-based) |
| `agentDtoBuilders.ts` | `sessionId`, `turnId`, `occurredAt`, `candidateId`, `replyPlanId`, `writebackId` → deterministic |
| `replyPlanner.ts` | `replyPlanId` → `deterministicReplyPlanId()` |
| `explorerService.ts` | `sourceId` fallback → `deterministicDraftId()` |

### 4. Static Scan & Allowlist

The static scanner found 51 patterns using `Date.now` / `Math.random` / `new Date` in core source. All 51 were classified as **allowed** because they fall into justified categories:

| Category | Count | Justification |
|----------|-------|--------------|
| Gate/report timestamps | 6 | Runtime reports — not core default paths |
| Operational modules (benchmark, editor, explainability, export, life, memory, parameters, temporal, recovery) | 30 | Recording when operations actually occurred |
| Audit entry/record timestamps | 4 | Runtime audit records |
| Service write-path timestamps | 2 | User-initiated actions |
| Time Machine snapshots | 3 | User-initiated history capture |
| Demo/presentation files | 2 | Non-core presentation |
| Determinism audit meta | 3 | Audit pattern detection + auditedAt |
| Explainability timeline | 1 | Causal step recording |

**Forbidden findings: 0** — all core default builder paths are now deterministic.

### 6. Quality Gate Integration

The Determinism Boundary Audit is now integrated into the Unified Quality Gate:

- **`unifiedQualityGate.ts`**: New `skipDeterminism` config option. When enabled (default), runs `runDeterminismBoundaryAudit()` as part of the unified gate.
- **`rc:verify`**: Now includes `run-determinism-boundary-audit.ts` in the verification chain.
- **`test:determinism`**: New npm script for standalone determinism audit.

Gate behavior:
- If determinism audit **fails** → unified gate FAIL → `releaseReady: false`
- If determinism audit **passes** with warnings → unified gate WARN
- If determinism audit **passes** clean → unified gate PASS

## Allowed Runtime Time Sources

The following runtime time sources are explicitly allowed and documented:

| Source | Location | Reason |
|--------|----------|--------|
| Time Machine `capturedAt` | `timeMachineSnapshot.ts`, `explorerService.ts` | User-initiated history record |
| Gate `generatedAt`/`completedAt` | `unifiedQualityGate.ts`, `releaseCandidateFreezeAudit.ts`, `qualityTrendBaseline.ts` | Runtime report generation |
| Audit entry `appliedAt` | `eventStudioApply.ts` | Actual application time record |
| Operational timestamps | `benchmark/`, `editor/`, `explainability/`, `export/`, `life/`, `memory/`, `parameters/`, `temporal/`, `recovery/` | Recording when real operations occurred |
| Service write paths | `explorerService.ts`, `characterPhysicsService.ts` | User-initiated actions |

## Forbidden Non-Deterministic Sources

The following patterns are **forbidden** in core default paths and will cause audit failure:

- `Date.now()` as default value for any core DTO builder ID
- `Math.random()` as default value for any core DTO builder ID
- `new Date()` as default value for any core DTO builder ID
- `Date.now()` / `Math.random()` in preview ID generation
- `Date.now()` / `Math.random()` in audit ID generation
- `Date.now()` / `Math.random()` in reply/plan/writeback ID generation
- `Date.now()` / `Math.random()` in agent turn/session ID defaults
- `Date.now()` / `Math.random()` in LLM boundary request/prompt/reply ID generation
