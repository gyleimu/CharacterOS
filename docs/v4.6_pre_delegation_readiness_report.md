## 1. 当前 V4 到底是什么

```text
V4 当前是 Temporal Homeostasis Observability Layer。

它提供了：
- 17 个 V3 tick phase 的结构化元数据
- 17 个 adapter shell（描述未来迁移计划，不执行）
- UnifiedTickTrace（将 V3 trace 映射为 per-process 结构）
- InternalStateFieldSnapshot（46 个 state variable 的快照 + baseline 偏离 + homeostatic pressure）
- 91 个 temporal 测试（全部通过，零 V3 行为改动）

它不是：
- Execution layer — 没有 TemporalProcess 在执行
- Replacement for runContinuousTick — V3 tick 仍然是唯一的执行路径
- Mutation pipeline — 所有 adapter 都是 shell，没有 delegation
```

**V4 是 V3 之上的一层透镜。透过它看 V3，V3 的时间行为变得结构化、可比较、可验证。但它不改变 V3 的任何行为。**

## 2. 17 个 Phase Delegation 评估

### 评分标准

| 维度 | low | medium | high |
|------|-----|--------|------|
| **Risk** | 单函数，单状态字段 | 多字段，有依赖 | 覆盖语义，复杂依赖 |
| **Isolation** | 无下游依赖 | 被 1-2 个 phase 读取 | 输出被后续 phase 覆盖或消费 |
| **Existing function** | 独立 V3 函数存在 | 部分 inline，部分函数 | 全部 inline，无独立函数 |
| **Mutation complexity** | 只写 1 个字段 | 写 2-4 个字段 | 写 5+ 个字段或覆盖前 phase |

## 3. 哪些 Phase 不适合 Delegation

### 3.1 全部 12 个 trace/context phase（不适合）

```text
Phase 1, 5-13, 16, 17

原因：
- 这些 phase 不改变 CharacterPhysicsState
- 它们构建 trace artifacts（recoveryTrace, parameterNetwork, boredomTrace 等）
- Delegation 的收益是"让 mutation 可追踪"——这些 phase 没有 mutation
- 包装为 TemporalProcess 的唯一价值是统一 trace 格式
- 但 V4.3 的 UnifiedTickTrace 已经做到了这一点（从 V3 trace 构建）
- 进一步 delegation 不会增加任何价值
```

### 3.2 decay_and_recovery（不适合第一批）

```text
Phase 3

原因：
- 这是最大的单体 composite phase
- 包含 4 个子操作：memory decay, procedural decay, reward recovery, boundary recovery
- 4 个子操作的顺序在当前代码中是隐式的（inline 顺序执行）
- 拆分需要决定：每个子操作是独立 TemporalProcess，还是保持 composite？
- 如果是独立的：需要 4 个 adapter，需要定义它们之间的依赖
- 如果是 composite：需要定义 sub-process ordering
- Phase 4（homeostasis）覆盖它的 boundary/reward/meta 输出
- 风险太高，不适合第一批
```

## 4. 第一批 Delegation 推荐

### 推荐：boredom（Phase 14）作为首选

```text
原因：

1. 最低风险
   - 只写 1 个字段：boredomState
   - 被覆盖/消费：无人覆盖。boredomState 只在 tick 结束后被 trace 读取。
   - 无下游依赖：没有其他 phase 依赖 boredomState 的新值。

2. 最高隔离
   - Phase 14 的输出不影响 Phase 15-17 的计算（它们读 meta/boundary/reward/memories）
   - 即使 boredom adapter 完全跳过，V3 的其他行为不变。

3. 已有纯函数
   - updateBoredomForTick(boredom, meta, reward, boundary, daysElapsed)
   - 返回值包含 before/after/delta/explorationDrive/inspirationChance/reasons
   - 不需要访问完整的 CharacterPhysicsState，只需要 4 个字段

4. 易于验证
   - 调用 V3 函数 → 获得结果 → 写回 state.boredomState
   - before/after 对比：boredomLevel 的差值应该等于 adapter 返回的 delta
```

### 备选：belief_evolution（Phase 15）

```text
原因：

1. 低风险
   - 只写 1 个字段：beliefStates
   - 被覆盖/消费：无人覆盖。DerivedCharacterState 读取 beliefStates，但那是 Phase 2 的事。
   - 无下游依赖：是 V3 tick 的最后一个 mutation phase（之后是 trace-only phases）

2. 高隔离
   - 输出是 tick 的最终状态的一部分
   - 不被任何后续 phase 修改

3. 已有纯函数
   - evolveBeliefsForTick(beliefs, memories, meta, daysElapsed)
   - 返回值包含 before/after/reasons

4. 一个注意事项
   - 读 memories（在 Phase 3 已 decay）
   - 只要 delegation 在 V3 Phase 3 之后执行，memories 就已经是正确的
```

### 不作为第一批：homeostasis（Phase 4）

```text
虽然 homeostasis 是 V4 的第一公民原则，但它不适合第一批 delegation。

原因：

1. 覆盖语义
   - Phase 4 的 regulatedBoundary/regulatedMetaState/regulatedRewardState
     有意覆盖 Phase 3 的 boundary/metaState/rewardState 输出
   - 如果在 delegation 中丢失了这个覆盖语义，会导致"double recovery"
     （Phase 3 恢复了一次，Phase 4 应该在此基础上做 homeostatic 精调）
   - 验证"覆盖是否正确"比验证"单字段写入"复杂得多

2. 读/写复杂
   - 读：homeostasisState, metaState, boundary, rewardState（4 个对象）
   - 写：homeostasisState, metaState, boundary, rewardState（4 个对象）
   - 其中 3 个（meta, boundary, reward）刚刚被 Phase 3 修改过
   - 需要确保 adapter 读取的是 Phase 3 之后的值

3. 建议
   - 在 boredom 和 belief_evolution 的 delegation 验证成功后
   - 以它们为模板，再迁移 homeostasis
   - 届时迁移经验更丰富，风险更低
```

## 5. Delegation 成功标准

### 必须达成

```text
S1: V3 ContinuousTickTrace shape unchanged
     Delegation 后，runContinuousTick 返回的 trace 的 phases 数组
     仍然是 17 个元素，每个 phase 的 name/changedStates/reasons 不变。

S2: All 347 tests continue passing
     V3 的 256 tests + V4.1-V4.5 的 91 tests 全部通过。

S3: Before/after state equivalent to V3
     对同一个 state，tick(7 days) 后的 CharacterPhysicsState
     在 delegation 前后完全相同（逐字段比较）。

S4: UnifiedTickTrace can mark adapter as "delegated"
     delegation 后，对应 process 的 implementationStatus 从
     "adapter_shell" 变为 "delegated"。
     observedOnly 从 true 变为 false。

S5: No extra mutation
     Delegation 不应该产生 V3 中没有的额外状态修改。
     可以通过 InternalStateFieldSnapshot before/after 对比验证。

S6: No phase order change
     Delegation 的 adapter 在 V3 tick pipeline 中的执行位置
     必须与其 sourcePhase 一致。不能提前或延后执行。
```

### 可选达成

```text
S7: Adapter 的执行时间可测量（overhead < 1ms）
S8: Adapter 的 trace 包含 before/after snapshot
S9: Adapter 的错误不会静默吞掉
```

## 6. Delegation 风险

### R1: Duplicated mutation

```text
如果 adapter 写回了 state 字段，但 V3 tick 的内联代码仍然在写同一个字段，
会导致 double-mutation。

缓解：
- Delegation 时注释掉 V3 tick 中的对应内联代码
- 不是删除，是注释（保留回退路径）
- 加注释标记 "// V4 delegated — see boredomAdapter"
```

### R2: Trace mismatch

```text
V3 tick 的 phases 数组是手动 push 的。
如果注释掉了 V3 内联代码，但没有删除对应的 phases.push()，
trace 中会出现 changedStates 与实际不符的情况。

缓解：
- Adapter 需要返回自己的 trace（changedStates, reasons）
- V3 tick 需要从 adapter 结果中读取 trace，而不是手动构建
```

### R3: Phase order drift

```text
如果 adapter 注册到 registry 中的 priority 与 V3 执行顺序不一致，
会导致 mutation 发生在错误的时机。

缓解：
- V4.1 metadata 已锁定 1-17 的 phase 顺序
- Delegation 必须严格按 phase number 顺序执行
- 不允许重新排序
```

### R4: State reference mutation

```text
如果 adapter 接收 state 对象引用并直接修改（而不是返回新值），
可能导致引用共享问题。

缓解：
- V3 函数已经设计为接收值并返回新值（或直接修改传入的 state 字段）
- updateBoredomForTick 和 evolveBeliefsForTick 都返回 { before, after, ... }
- Adapter 只需要写 state.field = result.after
```

### R5: Brittle tests

```text
如果 delegation 测试过于依赖精确数值，
未来 V3 参数调优会导致 delegation 测试脆断。

缓解：
- Delegation 测试应该验证"行为一致"（before/after 等价），
  而不是"数值精确"
- 使用 snapshot comparison 而非精确断言
```

## 7. 不建议现在做什么

```text
不重写 runContinuousTick。
  当前单体函数是稳定的。只注释 1-2 个 phase 的内联代码，
  委托给 adapter。不拆函数。

不实现 full registry execution。
  不让 TemporalProcessRegistry 执行所有 17 个 phase。
  只让 1-2 个 adapter 在 V3 tick 中被调用。

不做 Process Orchestrator (tickScheduler)。
  V4 Design Charter §3.1 描述的 tickScheduler.tick() 是 V5 的目标。
  V4 只做单个 adapter 的验证。

不恢复前端 / 不做 3D / 不做多角色 / 不做世界模拟。
  延续 V3 的所有限制。

不做 graph / benchmark / multi-character。
  这是 V10-V20 的范围。
```
