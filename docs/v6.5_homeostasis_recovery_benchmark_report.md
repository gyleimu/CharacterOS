### `src/core/benchmark/fixtures/firstReplayFixtures.ts`（修复）

**cracks 断言从 `bounded_above` (bound=1) 改为 `decrease` (directional)：**

原因：boundary 系统的 cracks 使用 `Math.max(0, value)` 只保底 0，不封顶 1。连续 overflow events 会导致 cracks 累计超过 1。`bounded_above` with `bound=1` 是不正确的约束。`decrease` 反映真实的 recovery 方向。

## 2. 为什么 homeostasis_recovery 需要 D10-aware

```text
benchmark 评估的是 subprocess boundary_recovery 的 Phase 3 intermediate。

Phase 4 homeostasis 通过 regulatedBoundary 覆盖 Phase 3 结果：
  final state.boundary === homeostasis.regulatedBoundary

benchmark 应该知道这个关系：
  - subprocess metrics 反映 recoverBoundary() 的效果（stress↓, cracks↓, integrity↑）
  - 最终 tick boundary 是 homeostasis.regulatedBoundary
  - 两者可能不同——subprocess 是 intermediate

D10 warning 注入 result.warnings：
  "D10 note: boundary_recovery subprocess metrics reflect Phase 3 intermediate.
   Final boundary is homeostasis.regulatedBoundary — the subprocess values
   may differ from the tick's final boundary."

这使得 benchmark 结果不会误将 subprocess metrics 误读为 final boundary。
```

## 3. 边界系统设计笔记

```text
cracks 和 stressLoad 不是 [0, 1] clamped 值：
  - cracks = Math.max(0, round4(value)) — 只保底，不封顶
  - stressLoad = Math.max(0, round4(value)) — 同上
  - recoverBoundary 和 applyBoundaryImpact 都不 clamp 到 1

多次 overflow events 会累计 cracks > 1 和 stressLoad > 1。

benchmark fixtures 应该使用 directional assertions（decrease/increase）
而不是 bounded assertions（bounded_above with bound=1）。
如果必须使用 bounded，bound 应设为安全上限（如 maxBound: 10 或 maxBound: 20）。

V6.5 的 fixture 修复反映了这个设计发现。
```
