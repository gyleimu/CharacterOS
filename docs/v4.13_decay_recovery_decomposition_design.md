## 1. Phase 3 Decay and Recovery — 当前行为

### 代码（continuousTick.ts lines 203-220）

```typescript
// Phase 3: decay_and_recovery
state.memories = memoriesBefore.map((memory) =>
  decayMemory(memory, daysElapsed, memoryDecayRate));
state.proceduralRoutines = proceduralRoutinesBefore.map((routine) =>
  decayProceduralRoutine(routine, daysElapsed));
state.rewardState = recoverRewardBaseline(state.rewardState, daysElapsed);
state.boundary = recoverBoundary(state.boundary, daysElapsed);
state.metaState = metaDriftResult.trace.after;  // ← Phase 2 的 after，不是 recovery
phases.push({
  name: "decay_and_recovery",
  changedStates: ["memories", "proceduralRoutines", "rewardState", "boundary"],
  reasons: [
    "Memory recency and effective weight decay with elapsed time.",
    "Procedural routines lose strength when unused.",
    "Reward state and psychological boundary recover toward baseline."
  ]
});
```

### 4 个子操作

| # | 操作 | V3 函数 | 作用对象 | 语义 |
|---|------|---------|---------|------|
| 1 | Memory decay | `decayMemory(memory, daysElapsed, decayRate)` | `memories[]` 每个元素 | **衰减** — recency 下降，weight 降低 |
| 2 | Procedural decay | `decayProceduralRoutine(routine, daysElapsed)` | `proceduralRoutines[]` 每个元素 | **衰减** — strength 下降 |
| 3 | Reward recovery | `recoverRewardBaseline(rewardState, daysElapsed)` | `rewardState` 整体 | **恢复** — dopamine 向 baseline 回归 |
| 4 | Boundary recovery | `recoverBoundary(boundary, daysElapsed)` | `boundary` 整体 | **恢复** — stressLoad 下降，integrity 修复 |

### Reads（5 个域）

| 域 | 来源 | 被哪个子操作读取 |
|----|------|----------------|
| `memoriesBefore` | Phase 1 snapshot | #1 |
| `proceduralRoutinesBefore` | Phase 1 snapshot | #2 |
| `state.rewardState` | 当前值 | #3 |
| `state.boundary` | 当前值 | #4 |
| `memoryDecayRate` | Phase 1 计算（含 metaState.forgettingSpeed 影响） | #1 |

### Writes（5 个 state 字段）

| 字段 | 写入者 | 后继覆盖？ |
|------|--------|----------|
| `state.memories` | #1 (decay) | 否 |
| `state.proceduralRoutines` | #2 (decay) | 否 |
| `state.rewardState` | #3 (recovery) | **是 — Phase 4 regulatedRewardState** |
| `state.boundary` | #4 (recovery) | **是 — Phase 4 regulatedBoundary** |
| `state.metaState` | Phase 2 after | **是 — Phase 4 regulatedMetaState** |

### ChangedStates

```text
["memories", "proceduralRoutines", "rewardState", "boundary"]
```

注意：虽然 `state.metaState` 也被赋值，但它不在 Phase 3 的 changedStates 中。它在 Phase 2 (meta_drift) 的 changedStates 中。

### Reasons

3 条硬编码的 reason 字符串。不是从 V3 函数返回的 — 因为每个子函数的返回值不包含 reasons。

## 2. 为什么它是 Composite Phase

### 与前 4 个 Delegated Phase 的对比

| 维度 | Boredom | Belief | Meta Drift | Homeostasis | **Decay & Recovery** |
|------|---------|--------|-----------|-------------|---------------------|
| V3 函数数 | 1 | 1 | 1 | 1 | **4** |
| State 字段数 | 1 | 1 | 1 | 4 (via 1 fn) | **5 (via 4 fns)** |
| 集合操作 | 否 | 否 | 否 | 否 | **是 — `.map()` 遍历数组** |
| 语义方向 | 单向 | 单向 | 单向 | 覆盖 | **混合（2 衰减 + 2 恢复）** |
| Reasons 来源 | 函数返回 | 函数返回 | 函数返回 | 函数返回 | **硬编码字符串** |
| 适配 adapter 模板 | ✓ | ✓ | ✓ | ✓ | **✗ 模板不匹配** |

### 核心困难

```text
前 4 次 delegation 都遵循同一模式：
  adapter 调用 1 个 V3 函数 → 返回 trace → 写 state → push phase

decay_and_recovery 不遵循此模式：
  - 4 个独立函数调用（非 1 个）
  - 2 个是 .map() 迭代（非整体替换）
  - reasons 是硬编码（非函数返回）
  - 3/5 的写入被 Phase 4 覆盖

直接套用 adapter 模板是强行适配。
```

## 3. 方案 A：整块 Composite Adapter

### 设计

```typescript
interface DecayRecoveryCompositeInput {
  memories: MemoryNode[];
  proceduralRoutines: ProceduralRoutine[];
  boundary: PsychologicalBoundary;
  rewardState: RewardState;
  memoryDecayRate: number;
  daysElapsed: number;
}

interface DecayRecoveryCompositeResult {
  trace: {
    memoriesAfter: MemoryNode[];
    proceduralRoutinesAfter: ProceduralRoutine[];
    boundaryAfter: PsychologicalBoundary;
    rewardStateAfter: RewardState;
    subResults: {
      memoryDecay: { count: number };
      proceduralDecay: { count: number };
      boundaryRecovery: { stressDelta: number };
      rewardRecovery: { dopamineDelta: number };
    };
  };
  phase: { name; changedStates; reasons };
  temporalResult: TemporalProcessResult;
}
```

### 评估

| 维度 | 评价 |
|------|------|
| 改动量 | 小 — 整块替换，类似 homeostasis |
| Phase shape | 完全保留（changedStates 不变，reasons 不变） |
| V4 抽象价值 | **低** — 仍是黑箱 composite，没有暴露内部结构 |
| D10 充分性 | **不足** — D10 只覆盖 overwrite semantics |
| 测试复杂度 | 中等 — 4 个子操作结果需逐个验证 |
| 回滚复杂度 | 低 — 恢复 5 行赋值代码 |

### 需要新增 Checklist

如果选择此方案，需要 **D11: Composite Subprocess Equivalence**：
```text
□ 每个子操作的 before/after 值等价于直接调用 V3 函数
□ 子操作的顺序与 V3 inline 顺序一致
□ subResults 完整反映 4 个子操作的输出
□ changedStates 仍为 ["memories", "proceduralRoutines", "rewardState", "boundary"]
```

## 4. 方案 B：拆分为多个独立 Adapter

### 设计

```text
Phase 3a: memory_decay adapter
Phase 3b: procedural_decay adapter
Phase 3c: reward_baseline_recovery adapter
Phase 3d: boundary_recovery adapter
```

### 评估

| 维度 | 评价 |
|------|------|
| 符合 TemporalProcess 设计 | **高** — 每个 adapter 是单一职责 |
| 改动量 | **大** — 需要改 phases 数组（17 → 20+） |
| V3 trace 兼容性 | **破坏性** — ContinuousTickTrace 假设 17 phases |
| phase order | 3a-3d 之间的顺序需要明确定义 |
| 测试复杂度 | 高 — 4 个独立 adapter 测试 + 组合测试 |
| 回滚复杂度 | 高 — 需要恢复 4 个 adapter + 合并 phases |

### 兼容性问题

```text
1. V3 ContinuousTickTrace.phases 长度为 17。
   拆分为 4 个 adapter 后 phases 变为 20。
   API consumers / Dashboard / trace replay 都假设 17。

2. Phase 3 的 changedStates = ["memories", "proceduralRoutines",
   "rewardState", "boundary"]。
   拆分后每个 sub-phase 的 changedStates 只包含 1 个字段。

3. Phase 3 的 reasons 是 3 条硬编码字符串。
   拆分后每个 sub-phase 需要生成自己的 reasons。

这些问题不是无法解决，但需要 Phase 3 拆分成为一个 V4→V5 migration，
而不是 V4 adapter delegation。

建议：V5 做这个迁移。V4 保持 17-phase trace 兼容。
```

## 5. 推荐方案

### 明确建议：V4 不做 decay_and_recovery delegation。

```text
理由：

1. 前 4 次 delegation 已经覆盖了 4/5 mutation phases。
   decay_and_recovery 是唯一剩余的。它的 composite 特性
   说明它需要的不只是一个 adapter，而是一个架构决策。

2. 两个方案都有显著缺陷：
   - 方案 A（composite）→ V4 抽象价值低，黑箱包装
   - 方案 B（split）→ 破坏 V3 trace 兼容性，V5 再做

3. 当前 Phase 3 代码是稳定的。4 个 V3 函数都经过充分测试。
   没有 urgent 需要 delegating 的理由。

4. V4 的 Execution Layer 目标已经达成：
   - 4/5 mutation phases delegated
   - D1-D10 checklist 完整
   - 154 temporal tests
   - 407 total tests

5. V5 可以在 V4 的经验基础上做这个迁移：
   - 决定 composite vs split
   - 设计 V5 tick trace format（不再绑定 17 phases）
   - 一次性迁移所有剩余 phase
```

### 如果未来必须做，推荐方案 A

```text
方案 A（整块 composite adapter）是短期可行的路径：
- 保持 17-phase trace
- 4 个子操作结果在 subResults 中暴露
- D11 checklist 确保等价性
- V5 再考虑拆分
```

## 6. 如果需要，新增 Checklist

### D11: Composite Subprocess Equivalence

```text
适用条件：adapter 内部包含多个独立 V3 函数调用。

□ 每个 subprocess 的 before/after 等价于直接调用对应 V3 函数
□ subprocess 执行顺序与 V3 inline 顺序一致
□ subResults 包含每个 subprocess 的关键输出
□ changedStates 仍与 V3 原 phase 一致
□ 如果有 subprocess 的输出被后续 subprocess 读取，顺序不能改变
```

### D12: Downstream Overwrite Preservation

```text
适用条件：adapter 的输出被后续 phase 覆盖。

□ Phase N 的输出值被 Phase N+1 正确读取
□ Phase N+1 的 overwrite 仍然是基于 Phase N 的输出计算的
□ 如果 adapter 改变了 Phase N 输出的顺序/格式，Phase N+1 的行为不变
```

D12 对 homeostasis + decay_and_recovery 的关系尤其重要。
当前 D10 已验证 homeostasis 正确覆盖 Phase 3。
如果 Phase 3 被 adapter 化，D12 确保覆盖链路没有断裂。

## 7. 与 Homeostasis 的关系

### 覆盖链路

```text
Phase 3 (decay_and_recovery):
  state.boundary = recoverBoundary(state.boundary, daysElapsed)
  state.rewardState = recoverRewardBaseline(state.rewardState, daysElapsed)
  state.metaState = metaDriftResult.trace.after  (← Phase 2 的输出)

       ↓ 这些值被 Phase 4 读取 ↓

Phase 4 (homeostasis):
  applyHomeostasis({
    meta: state.metaState,      ← 可能是 Phase 3 写入的 metaDrift.after
    boundary: state.boundary,    ← Phase 3 recovery 后的值
    reward: state.rewardState,   ← Phase 3 recovery 后的值
  })
  → 在 Phase 3 的基础上做 homeostatic 精调
  → 输出 regulatedBoundary / regulatedMetaState / regulatedRewardState
  → 覆盖 Phase 3 的值
```

### 如果 Phase 3 被 adapter 化

```text
如果使用方案 A (composite)：
  adapter 输出的 boundary/reward/meta 与 V3 inline 代码完全一致
  → Phase 4 读取的值不变
  → D10 覆盖语义不变
  → D12 自动满足

如果使用方案 B (split)：
  需要确保 sub-phase 3d (boundary_recovery) 在 3c (reward_recovery) 之后
  且都在 Phase 4 之前
  → Phase 4 读取的值不变
  → D10 覆盖语义不变
  → D12 需要显式验证 Phase 3d → Phase 4 的值传递
```
