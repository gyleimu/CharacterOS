# Temporal Semantics 核心阶段报告

## 阶段结论

本阶段将事件时间从展示字段提升为 Character Physics Core 的正式输入。

此前一次 `processEvent()` 调用等价于一个事件步，但事件步没有回答两个关键问题：事件之间过去了多久，以及同一事件在短时间内被重复记录了多少次。现在核心积分明确区分：

```text
raw event impact
-> elapsed-time recovery
-> event-density saturation
-> effective event impact
-> memory / belief / boundary / personality integration
-> temporal audit trace
```

因此，五次事件发生在数小时内与五次事件分散在数周内，不再得到相同的有效剂量、恢复过程和状态轨迹。

## 核心实现

### CharacterTemporalState

`CharacterPhysicsState` 新增持久化时间状态：

- `lastProcessedAt`: 当前逻辑时钟；
- `totalElapsedDays`: 已应用的恢复时间；
- `processedEventCount`: 所有处理事件数量；
- `timedEventCount`: 进入时间窗口的事件数量；
- `recentEvents`: 90 天内、最多 256 条的密度审计记录。

序列化合同对 `temporal` 保持可选读取，因此旧状态文件仍可加载；加载后会补齐默认时间状态。

### 事件时间模式

每次事件先得到一个 `EventTemporalPlan`：

| Mode | 含义 | 行为 |
|------|------|------|
| `legacy_untimed` | 旧调用且尚无逻辑时钟 | 保持旧影响语义 |
| `first_timed_event` | 第一个有效时间戳 | 建立时钟，不虚构恢复 |
| `forward_time` | 时间向前 | 先恢复，再处理事件 |
| `same_time` | 同时事件 | 不恢复，进入密度饱和 |
| `inherited_clock` | 时钟已建立，但新事件缺少时间 | 继承当前时钟，不能绕过密度控制 |
| `invalid_timestamp` | 时间格式无效 | 产生审计 warning，不推算时间 |
| `out_of_order` | 事件早于当前时钟 | 不倒放恢复，不回拨时钟，显式 warning |

输入时间统一规范化为 ISO-8601 UTC。未知时间哨兵不会再被写成 1970 年记忆。

### 事件密度饱和

24 小时窗口内，以事件内容、标签与类别构造语义签名，而不是依赖 event ID。不同 ID 的同一事件仍会被识别为重复输入。

当前工程公式：

```text
densityPressure = sameEventCount * 0.9 + otherSameCategoryCount * 0.15
densityScale = max(0.35, 1 / sqrt(1 + densityPressure))
effectiveImpact = rawImpact * densityScale
```

这个机制不是删除重复事件。每次事件仍产生独立 MemoryNode、Particle 和审计记录，只是短时间内的边际影响递减，并保留 `0.35` 的影响下限。

### 事件间恢复

有正向时间间隔时，`processEvent()` 在新事件进入前复用既有 `runContinuousTick()`：

- Memory recency 与有效权重衰减；
- Procedural routine 衰减；
- Boundary 与 Reward 向基线恢复；
- Homeostasis、Meta drift、Belief evolution 正常执行；
- Cluster mass 按衰减后的 Memory 重新同步；
- Personality velocity 按 14 天半衰期衰减。

人格动量保留率：

```text
retention = exp(-ln(2) * elapsedDays / 14)
```

连续 tick 在逻辑时钟已建立时也会推进时钟，避免手工恢复后下一次事件重复计算同一段时间。

### Trace 与边界

`PhysicsStepResult`、API DTO、simulation snapshot 和 galaxy step trace 均新增 `temporalSemantics`：

- 原始与有效 impact；
- `instantImpact`、窗口内 `accumulatedImpact` 与 `recoverySinceLastEvent`；
- 事件密度与缩放；
- 原始与实际恢复天数；
- Boundary、Memory recency、Cluster mass、Velocity 的恢复前后值；
- warning、处理后时钟和事件计数。

Event Studio Preview 与 Apply 现在会传递 `occurredAt`。Apply 后的 Memory 时间、Temporal State 和审计窗口使用同一规范化时间。

Draft ID、Apply audit ID 与状态 fingerprint 也纳入事件时间和 Temporal State，避免“同一文本、不同发生时间”被错误识别为同一次写入。二次 review 同时修复了 `allowMutation=true` 时 before state 可能被同一引用覆盖、导致审计摘要失真的旧问题。

## 兼容策略

- 完全无时间戳且尚未建立时钟的旧调用继续使用 full impact，不自动增加恢复；
- 一旦状态建立逻辑时钟，后续无时间戳事件继承当前时钟，防止调用方通过省略时间绕过饱和；
- 旧序列化状态缺少 `temporal` 时自动补齐，不要求迁移脚本；
- `runEventSequence(daysPerStep)` 保留旧的 memory-only 行为，但显式时间戳存在时不再重复衰减；
- 时间最大恢复区间限制为 3650 天，异常超长间隔会产生 warning。

## 仍然存在的限制

1. 24 小时窗口、0.35 下限和 14 天 velocity 半衰期是工程先验，不是心理学实证参数。
2. 乱序事件当前被安全接收和审计，但不会自动从历史时间点重放完整因果链；这需要 Durable State / Event Store。
3. Legacy untimed 路径只为兼容保留，无法表达真实恢复时间。
4. Memory recency 的长期 active mass 下限仍需通过 Golden Trajectory 校准。
5. 当前测试证明系统内部一致和可复现，不代表对现实人格具有临床或测量效度。
