## 1. Summary

V13.2 implements the **LLM Boundary Prompt Builder** — the component that converts
a `LlmBoundaryRequest` (V13.1) into a safe, bounded `LlmBoundaryPrompt` (V13.1).

The prompt builder is the **gatekeeper** between CharacterOS's deterministic pipeline
and any future LLM provider. It constructs exactly what the LLM may see — and nothing more.

## 3. Functions

### 3.1 `buildLlmBoundaryPromptFromRequest(request, options?)`

**Main entry point.** Orchestrates all sub-builders and produces the final prompt.

Options:
- `outputFormat`: `"plain_text"` (default) or `"markdown"`
- `extraForbiddenClaims`: additional forbidden claims beyond defaults
- `extraResponseConstraints`: additional response constraints

### 3.2 `buildSystemInstructions(request)`

Builds the **system-level** instructions that define the LLM's role as a language realization adapter.

Contains:
- Role definition: "你是 CharacterOS 的语言实现适配器"
- Core prohibitions: no mutation, no diagnosis, no invention, no writeback claims, no raw variables
- Positive guidance: use only grounded facts, preserve uncertainty, include safety notices
- `allowLlm=false` warning when LLM path is disabled
- Grounding weakness warnings when facts are few or absent
- Strict safety mode elevation
- Locale hint (zh-CN → 简体中文)

### 3.3 `buildDeveloperInstructions(request)`

Builds **developer-level** instructions with specific guidance for this turn.

Contains:
- Reply intent and tone (from AgentReplyPlan)
- Character context: headline, emotional state label, stress level/phase
- Dominant beliefs (top 2, with strength)
- Behavior tendency
- Suggested response outline
- LLM boundary instructions (if set on reply plan)
- Allowed response boundaries
- Output format expectation
- First-person roleplay prohibition

### 3.4 `extractGroundingFacts(request)`

Extracts grounding facts from three sources:

1. `replyPlan.groundedFacts` — tagged `[grounded]`
2. `bundle.evidenceRefs` — tagged `[evidence:<source>]`
3. `bundle.explainabilityTimeline.causalSteps` — tagged `[causal:<confidence>]`

If all sources are empty, includes a `[warning]` entry.

### 3.5 `buildForbiddenClaims(request)`

Builds the **negative constraints** — what the LLM MUST NOT claim.

Categories:
- Medical/psychological diagnosis
- State mutation claims
- Writeback execution claims
- Unsupported emotion/personality/belief claims
- Multi-character relationship expansion
- Raw variable disclosure (trust score, fear level, etc.)
- Numerical state values
- False certainty when evidence is weak
- Authority overreach
- Strict-mode extras: self-harm, clinical judgment

### 3.6 `buildResponseConstraints(request)`

Builds the **positive constraints** — what the LLM SHOULD do.

Categories:
- Conciseness
- Grounding
- Safety notice preservation
- Uncertainty acknowledgment (elevated when notes exist or facts are few)
- Intent-specific guidance (different for explain_state, preview_event, etc.)
- No diagnosis
- No mutation/writeback claims
- No first-person roleplay
- `allowLlm=false` disabled warning

### 3.7 `redactPromptUnsafeFields(prompt)`

**Defense-in-depth.** Redacts any unsafe content that may have leaked into the prompt.

Redacts:
- Secret patterns: `apiKey`, `access_token`, `bearer token`, `authorization`, `password`, `sk-*`, `pk-*`
- Raw state key patterns: `particleIds`, `driftMultiplier`, `biologicalNature`, `rewardState`, `homeostasisState`, `coordinate`, `memoryPayload`, etc.
- Embedded JSON: `"coordinate": {...}`, `"biologicalNature": {...}`, `"memories": [...]`

Returns a new prompt object (does not mutate the input).

## 4. System / Developer Instruction Boundaries

| Layer | Audience | Content | Authority |
|-------|----------|---------|-----------|
| System Instructions | LLM system prompt | Role, absolute prohibitions, safety mode | Always included, cannot be overridden |
| Developer Instructions | LLM developer/assistant prompt | Intent, tone, character context, reply outline | Derived from reply plan, always included |

The system instructions define **what the LLM is** (language adapter).
The developer instructions define **what to do this turn** (specific intent + context).

---

## 5. Grounding Fact Extraction

Three-tier extraction ensures the prompt includes all available evidence:

| Tier | Source | Tag |
|------|--------|-----|
| 1 | `replyPlan.groundedFacts` | `[grounded]` |
| 2 | `bundle.evidenceRefs[]` | `[evidence:<source>]` |
| 3 | `bundle.explainabilityTimeline.causalSteps[]` | `[causal:<confidence>]` |

Tags give the LLM provenance awareness without exposing raw state.

## 8. Redaction

The redaction layer provides defense-in-depth. Even if a builder accidentally
includes forbidden content, the redactor strips it before the prompt reaches the LLM.

| Pattern Type | Detection | Response |
|-------------|-----------|----------|
| Secret keys | Regex: `apiKey`, `access_token`, `bearer`, `sk-*`, etc. | Replace key+value with `[已编辑 — 内容违反边界策略]` |
| Raw state keys | Regex: `particleIds`, `coordinate`, `biologicalNature`, etc. | Replace key word |
| Embedded JSON | Regex: `"coordinate":{...}`, `"memories":[...]` | Replace entire JSON block |

---

## 9. `allowLlm=false` Behavior

When `request.allowLlm` is `false`:

1. **System instructions** include an explicit `⚠️ LLM 路径已禁用` section stating the prompt is structural preview only and actual output must use deterministic fallback
2. **Response constraints** include a matching disabled warning
3. The prompt is **still built normally** — all grounding facts, safety notices, etc. are included
4. No LLM is called (the prompt builder has no network capability)
5. The prompt can be used for testing, inspection, or documentation

---

## 10. Why No Provider Call Yet

The prompt builder is a **pure construction layer**. It:

- Does not call any LLM provider
- Does not make network requests
- Does not depend on any external service
- Is fully synchronous
- Is fully deterministic

Provider calls belong in V13.3 (Mock Provider Adapter) and V13.x (real providers).
The prompt builder's output (`LlmBoundaryPrompt`) is the **input** to those adapters.
