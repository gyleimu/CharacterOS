## 2. Flaky Test: Located & Fixed

### Root Cause

**File:** `tests/core/benchmark/benchmarkRunner.test.ts`  
**Tests affected:** 2 assertions across 2 test cases

```typescript
// BEFORE (flaky):
expect(result.durationMs).toBeGreaterThan(0);
```

`durationMs` is a benchmark timer with millisecond resolution. On fast runs, the benchmark completes in < 1ms, producing `durationMs === 0`. The assertion `toBeGreaterThan(0)` would intermittently fail depending on machine load and timing variance.

### Fix

```typescript
// AFTER (deterministic):
expect(result.durationMs).toBeGreaterThanOrEqual(0);
```

A `durationMs` of 0 is a valid measurement — it means the benchmark ran faster than the timer resolution. It does not indicate a problem with the benchmark or the system.

### Verification

- **10/10** isolated runs of `benchmarkRunner.test.ts` — all pass
- **5/5** full test suite runs — all 792 tests pass
- **0 failures** across all verification runs

## 5. Integrity Policy Audit

### Current Implementation

The `evaluatePatchIntegrityPolicy()` function returns three possible statuses:
- `pass` — no issues
- `soft_warning` — warnings present, apply allowed
- `block` — errors present

### Enforcement Status

| Gate | Enforced? | Where |
|---|---|---|
| `validation.valid === false` | ✅ Yes | Preview, Apply, Rollback |
| `riskSummary.severity === "error"` | ✅ Yes | Apply, Rollback |
| `integrityPolicy.status === "block"` | ❌ No | Not enforced |

The `block` status from integrity policy is **observed but not enforced**. The policy result is returned in API responses for transparency, but does not prevent state writes.

### Why Block Is Not Enforced

The integrity check (`inspectCharacterStateIntegrity`) performs deep structural validation:
- ID uniqueness, referential integrity (memory ↔ cluster, belief ↔ memory)
- Range checks on coordinates, scalars, velocities
- Cluster map consistency

These checks target **structural corruption**, not **patch validity**. A patch edit that changes scalar values through the standard pipeline (with clamping) cannot cause structural corruption. However:

1. `cloneStateForEditing` uses `JSON.parse(JSON.stringify(state))` — Maps are serialized to arrays and reconstructed, which could trigger false positives if the reconstruction is imperfect
2. Pre-existing blueprint state may have warnings/errors unrelated to the patch
3. The integrity check was designed as a diagnostic tool, not a write gate

### Decision

**Do not enforce block gate in V9.8.** The existing gates (validation + risk severity) provide sufficient protection. The integrity policy remains an observable signal for diagnosis.

**Recommendation for V10:** If snapshot storage is added (to support multi-step rollback), re-evaluate integrity enforcement as part of a broader state snapshot + repair pipeline.

### History Ordering

Audit entries are stored in insertion order in the `patchAuditLog` Map. The history route returns them in insertion order (oldest first). This is deterministic because:
- Entries are appended, never reordered
- Tests use unique characterIds (no interference)
- No parallel mutation to the same character

## 10. Known Limitations

| Limitation | Details |
|---|---|
| Integrity block not enforced | `block` status from `evaluatePatchIntegrityPolicy` is observed but does not prevent writes. Existing gates (validation, risk severity) provide sufficient protection for scalar edits. |
| Single-step rollback only | Multi-step undo requires full snapshot storage — out of scope for V9. |
| `patchAuditLog` not cleared on `resetCharacter` | Dormant entries accumulate in the in-memory Map but are harmless (benign memory leak in test-only scenarios). |
| No UI | All editor functionality is API-only; UI deferred to V10+. |
| No concurrent edit protection beyond stale-write | `validatePatchAgainstState` catches path-level races; no optimistic locking or version vectors. |
