# V13.8 Mock Provider Harness

## 目标

V13.8 将既有的 Agent Reply Plan、LLM Boundary DTO 与安全 Prompt 从“只生成预览”推进为完整的离线语言边界闭环：

```text
AgentReplyPlan + AgentGroundingBundle
-> Safe LlmBoundaryPrompt
-> Deterministic Mock Provider
-> Output Validation
-> Grounding Check
-> Validated Reply or Deterministic Fallback
```

本版本不连接真实模型、不使用网络、不读取 CharacterPhysicsState 原始状态，也没有写回权限。

## 实现模块

- `mockLlmProvider.ts`：确定性 Mock Provider，覆盖成功、空输出、超时、错误、诊断、伪写回、无接地事实与缺安全声明。
- `llmOutputValidator.ts`：检查 provider 错误、空输出、诊断、状态修改声明、内部字段、密钥、过度确定与安全声明缺失。
- `llmGroundingChecker.ts`：把输出事实逐条映射到 Prompt facts、Reply Plan facts 与 evidence refs，输出支持证据和置信度。
- `llmFallbackReplyGenerator.ts`：根据 intent、grounded facts、uncertainty notes 和 safety notices 生成确定性回退文本。
- `llmBoundaryService.ts`：编排完整链路；任一安全步骤失败即回退，并对最终交付文本再次 validation 和 grounding。

最终复审额外收紧了三项执行边界：

- Provider 只接收深拷贝且递归冻结的 DTO 快照，无法通过对象引用修改原请求。
- Provider response ID 使用完整响应内容、结束原因和脱敏错误生成；execution ID 同时纳入 Provider 与 fallback provenance，避免不同失败共享审计 ID。
- 确定性 fallback 若未通过最终 validation 或 grounding，会抛出 `LlmBoundaryDeliveryError` 并拒绝交付，不会把复检失败的文本继续返回。

## 关键边界

```text
providerType = mock
networkAllowed = false
networkUsed = false
noMutation = true
noWritebackAuthority = true
writebackPerformed = false
```

LLM 成功路径必须同时满足：显式启用、V12 session 允许、policy 未阻止、输入预检通过、provider 输出有效、所有事实接地。只改变自然语言不能绕过任何条件。

## Fallback 原因

- `safety_preflight_failed`
- `llm_disabled`
- `provider_not_allowed`
- `llm_unavailable`
- `validation_failed`
- `grounding_failed`

Fallback 不是未经检查的兜底字符串。最终 fallback 文本也会再次经过 Output Validator 与 Grounding Checker。

## 静态 Harness

`outputs/llm-boundary-harness/index.html` 可直接离线打开，包含七类审计 case：

1. Grounded mock reply
2. LLM disabled
3. Provider timeout
4. Diagnosis blocked
5. Mutation claim blocked
6. Ungrounded claim blocked
7. Missing safety notice

当前结果为 1 条安全 Mock 回复、6 条确定性回退、0 条最终交付失败。

专项测试覆盖 221 条 LLM Boundary 测试；全项目验证为 180 个测试文件、2447 个测试、0 failures。

## 路线说明

V13.0 Charter 原计划把 Mock Provider、Validator、Grounding、Fallback 和 Service 分配到 V13.3–V13.7。实际仓库在这些编号中完成了 Determinism Boundary、指标协调、AST 扫描与分支整合，因此 V13.8 以能力闭环为准补齐缺失实现，不重写历史版本号。
