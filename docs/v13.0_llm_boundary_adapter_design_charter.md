## 1. V13 Definition

**V13 is the LLM Boundary Adapter.**

It is the **thin outermost layer** of the CharacterOS stack. Its sole responsibility is
to convert structured Agent SDK output (`AgentReplyPlan` + `AgentGroundingBundle`)
into natural-language replies that a human user can read.

V13 is:

- An **adapter** — it translates structured plans into prose
- A **boundary** — it sits between the deterministic CharacterOS core and any LLM provider
- **Read-only** — it cannot mutate character state, create memories, or execute writebacks
- **Safe by default** — it validates every LLM output before it reaches the user

V13 is NOT:

- A chat product
- An LLM integration into the core engine
- A multi-character system
- An autonomous scheduler
- A replacement for the deterministic reply planner

### V13 is language realization only.

CharacterOS already decides **what** to say (via AgentReplyPlan).
V13 decides **how** to say it in natural language — and only when safe to do so.

```
┌─────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────┤
│  V12  Agent SDK                                             │
│       Input → Candidates → Policy → Context → Reply Plan    │
│       → Writeback Plan. Deterministic pipeline.             │
│       Session state continuity (V12.12).                    │
├─────────────────────────────────────────────────────────────┤

│       CharacterStateSurface, EventStudio (preview/apply),   │
│       ExplainabilityTimeline, TimeMachine, RealityAudit.    │
├─────────────────────────────────────────────────────────────┤
│  V10  Core Kernel                                           │
│       CharacterPhysicsEngine, PersonalityCoordinate,        │
│       PsychologicalBoundary, Memory, Belief, Homeostasis.   │
│       Deterministic simulation engine.                      │
└─────────────────────────────────────────────────────────────┘
```

| Layer | Role | V13 interacts by… |
|-------|------|-------------------|
| V10 Core | Deterministic personality engine | **Never** — V13 cannot see or touch raw state |
| V11 Explorer | State/explainability/time-machine surfaces | Reading `CharacterStateSurface` via `AgentGroundingBundle` (already mediated by V12) |
| V12 Agent SDK | Input/policy/context/reply/writeback plans | Receiving `AgentReplyPlan` + `AgentGroundingBundle` as input |
| V13 LLM Boundary | Language realization only | Producing validated natural-language replies |

**Key principle:** V13 extends the surface, not the depth. It adds a language layer on top
of what V12 already produces — it does not change how V10/V11/V12 work.

(V13) and, separately, the structured writeback plan (V12). Confirmation flows through
V12's existing `applyWriteback()` gate — not through the LLM.

## 4. Strict Boundaries

The LLM is confined to a **sandbox at the boundary**. It operates on a curated subset of
information and cannot reach into the system.

### LLM CANNOT:

| Category | Specific Prohibition |
|----------|---------------------|
| **State mutation** | Cannot mutate `CharacterPhysicsState` |
| **Event application** | Cannot call `EventStudioApply` or `applyWriteback` |
| **Memory creation** | Cannot create memory nodes directly |
| **Policy alteration** | Cannot change `AgentPolicyDecision` |
| **Writeback execution** | Cannot alter or execute `AgentWritebackPlan` |
| **Raw state access** | Cannot see `CharacterPhysicsState` internals (coordinates, particles, boundary internals) |
| **Secrets** | Cannot see API keys, environment variables, auth tokens |
| **Diagnosis** | Cannot produce medical or psychological diagnosis language |
| **Fabrication** | Cannot claim facts not provided in `groundedFacts` or `evidenceRefs` |
| **Emotion invention** | Cannot invent hidden emotions or beliefs beyond the provided context |
| **Truth claims** | Cannot represent simulation output as objective truth about a real person |

### LLM CAN:

| Category | What is Allowed |
|----------|-----------------|
| **Naturalization** | Convert structured reply outline into natural language |
| **Tone adaptation** | Match the tone specified in `AgentReplyPlan.tone` |
| **Fact restatement** | Restate facts from `groundedFacts` in user-friendly language |
| **Uncertainty expression** | Express uncertainty where `uncertaintyNotes` indicate it |
| **Safety notice inclusion** | Include or preserve `safetyNotices` in the reply |
| **Question asking** | Ask clarifying questions when intent is `clarify_input` |
| **Refusal** | Refuse unsafe requests when intent is `refuse_unsafe` |

### The Boundary Contract

```
┌──────────────────────────────────────────────────────┐
│                   LLM CAN SEE                        │
│                                                      │
│  • AgentReplyPlan (intent, tone, groundedFacts,      │
│    uncertaintyNotes, safetyNotices,                  │
│    suggestedResponseOutline)                         │
│  • AgentGroundingBundle.characterStateSurface        │
│    (headline, emotionalState.label,                  │
│     stressState.label, dominantBeliefs,              │
│     behaviorTendencies)                              │
│  • evidenceRefs (source + excerpt only)              │
│  • explainabilityTimeline (causal steps, confidence) │
│                                                      │
│  ALL OF THESE ARE ALREADY MEDIATED BY V12.           │
│  RAW STATE NEVER APPEARS HERE.                       │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│                 LLM CANNOT SEE                        │
│                                                      │
│  • CharacterPhysicsState (coordinates, particles)    │
│  • Raw personality coordinate values                 │
│  • Full memory dump (all memory nodes)               │
│  • Boundary internals (stressLoad, integrity)        │
│  • BiologicalNature                                  │
│  • RewardState                                       │
│  • HomeostasisState                                  │
│  • MetaState internals                               │
│  • WritebackPlan                                     │
│  • PolicyDecision internals                          │
│  • API keys / secrets / env vars                     │
│  • Any write/apply function                          │
└──────────────────────────────────────────────────────┘
```

## 6. Prompt Builder Rules

The Prompt Builder is the **only** component that constructs what the LLM sees.
It must enforce strict information boundaries.

### 6.1 What the Prompt MAY Include

| Input | Source | Purpose |
|-------|--------|---------|
| Reply intent | `AgentReplyPlan.intent` | Context for the LLM about what kind of reply to produce |
| Reply tone | `AgentReplyPlan.tone` | Style guidance |
| Grounded facts | `AgentReplyPlan.groundedFacts` | The **only** facts the LLM may state |
| Uncertainty notes | `AgentReplyPlan.uncertaintyNotes` | Must be preserved in output |
| Safety notices | `AgentReplyPlan.safetyNotices` | Must appear in or with the output |
| Suggested response outline | `AgentReplyPlan.suggestedResponseOutline` | Structural guidance |
| Character state surface headline | `AgentGroundingBundle.characterStateSurface.headline` | High-level context |
| Emotional state label | `AgentGroundingBundle.characterStateSurface.emotionalState.label` | Emotional context |
| Stress state label + phase | `AgentGroundingBundle.characterStateSurface.stressState` | Stress context |
| Dominant beliefs (content + strength) | `AgentGroundingBundle.characterStateSurface.dominantBeliefs` | Belief context |
| Behavior tendency | `AgentGroundingBundle.characterStateSurface.behaviorTendencies` | Behavioral context |
| Evidence refs (source + excerpt) | `AgentGroundingBundle.evidenceRefs` | Citation support |
| Causal steps | `AgentGroundingBundle.explainabilityTimeline` (if present) | Why the state is what it is |
| Boundary instructions | `AgentReplyPlan.llmBoundaryInstructions` | LLM behavior constraints |

### 6.2 What the Prompt MUST NOT Include

| Forbidden Content | Reason |
|-------------------|--------|
| Raw `CharacterPhysicsState` | Security boundary |
| Raw coordinate values (trust, fear, openness, etc.) | Too low-level for LLM, risk of hallucination |
| Full memory dump | Privacy + context pollution |
| `BiologicalNature` internals | Not relevant to language |
| `RewardState` internals | Not relevant to language |
| `HomeostasisState` internals | Not relevant to language |
| `MetaState` internals | Not relevant to language |
| API keys / secrets / env vars | Security |
| Policy override commands | LLM must not control policy |
| Writeback execution authority | LLM must not trigger state changes |
| Any instruction to "bypass" or "override" | Safety |
| `applyWriteback` function reference | Boundary enforcement |

### 6.3 System Prompt Template (Conceptual)

```text
You are a character simulation narrator for CharacterOS.
You speak FOR the simulated character based on provided state data.

RULES (violation = invalid output):
1. Only state facts from the GROUNDED FACTS section below.
2. If a fact is marked UNCERTAIN, preserve that uncertainty.
3. Include ALL safety notices in your response.
4. NEVER diagnose, prescribe, or give medical advice.
5. NEVER claim the simulation is real or factual.
6. NEVER mention internal variable names (trust score, fear level, etc.)
7. NEVER promise that events have been applied or state has changed.
8. If you don't have enough information, SAY SO — don't fabricate.
9. Match the specified TONE and INTENT.
10. Output ONLY the reply text. No meta-commentary.

[GROUNDED FACTS]
...

[UNCERTAINTY NOTES]
...

[SAFETY NOTICES]
...

[RESPONSE OUTLINE]
...

[TONE] ...
[INTENT] ...
```

## 7. Output Validation

Every LLM output must pass validation before it reaches the user. Validation is
**deterministic**, **rule-based**, and **runs without LLM involvement**.

### 7.1 Validation Rules

| Rule ID | Check | Severity | Description |
|---------|-------|----------|-------------|
| `no_diagnosis` | Regex + keyword | error | No medical/psychological diagnosis language |
| `no_unsupported_claim` | Grounding check | error | Every factual claim maps to a groundedFact or evidenceRef |
| `no_mutation_promise` | Regex | error | No "I have updated", "state changed", "memory saved" |
| `no_writeback_claim` | Regex | error | No claim that writeback happened or was applied |
| `no_multi_character` | Regex | warn | No expansion into multi-character relationship dynamics |
| `safety_notices_present` | Substring | error | Safety notices appear in or alongside the reply |
| `uncertainty_preserved` | Heuristic | warn | Uncertainty language preserved when evidence is weak |
| `intent_alignment` | Heuristic | warn | Response aligns with replyPlan.intent |
| `no_raw_internals` | Regex | error | No internal variable names leaked (particleIds, driftMultiplier, etc.) |
| `no_diagnosis_disclaimer` | Substring | error | Disclaimer about simulation present |

### 7.2 Validation Flow

```
raw LLM output
    │
    ▼
┌─────────────────┐
│ Regex-based      │
│ checks           │  (no_diagnosis, no_mutation_promise, no_raw_internals, etc.)
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Grounding check  │  (see Section 8)
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Heuristic checks │  (uncertainty_preserved, intent_alignment)
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Decision         │
│                   │
│ all errors pass?  │── YES ──► AgentNaturalLanguageReply
│                   │
│ any error fail?   │── YES ──► FallbackReply
└──────────────────┘
```

## 8. Grounding Checker

The Grounding Checker ensures every factual statement in the LLM output is
supported by the `groundedFacts` or `evidenceRefs` that were provided in the prompt.

### 8.1 Algorithm (Conceptual)

```
Input: LLM output text, AgentGroundingBundle, AgentReplyPlan

1. EXTRACT claims from LLM output:
   - Split into sentences
   - Classify each sentence as: factual claim, emotional expression,
     safety notice, conversational filler, question

2. For each FACTUAL CLAIM:
   a. Search for matching support in groundedFacts
   b. Search for matching support in evidenceRefs[].excerpt
   c. If NO match found:
      → record UngroundedClaim with reason "no matching groundedFact"
   d. If claim CONTRADICTS a groundedFact:
      → record UngroundedClaim with reason "contradicts evidence"

3. For each EMOTIONAL EXPRESSION:
   a. Check against emotionalState.label from characterStateSurface
   b. If expression exceeds provided emotional range → warn

4. Check UNCERTAINTY:
   a. If uncertaintyNotes are non-empty but LLM output contains
      no uncertainty language → warn "uncertainty not preserved"

5. Return GroundingCheckResult
```

### 8.2 Grounding Failure → Fallback

If the Grounding Checker returns `grounded: false` with any error-severity
ungrounded claims, the LLM output is rejected and the system falls back to
the deterministic reply outline.

## 9. Fallback Strategy

The fallback is **always available** and **always deterministic**. It does not
require an LLM.

### 9.1 When Fallback Triggers

| Condition | Fallback Reason |
|-----------|----------------|
| LLM provider unavailable (timeout, network error) | `llm_unavailable` |
| LLM provider returns error | `llm_unavailable` |
| Output validation fails (any error rule) | `validation_failed` |
| Grounding check fails (any error-severity ungrounded claim) | `grounding_failed` |
| LLM response is empty or whitespace-only | `validation_failed` |

### 9.2 Fallback Reply Generator

The fallback generator produces a reply **directly from AgentReplyPlan** without any LLM:

```
Input: AgentReplyPlan

1. Select template based on intent:
   explain_state    → "Current state: {headline}. Emotional state: {label}."
   preview_event    → "Event preview: {candidate summary}. Impact: {delta}."
   ask_confirmation → "Confirm to apply: {event}. Reply 'apply' to confirm."
   refuse_unsafe    → "Cannot process: {reasons}. Please revise input."
   summarize_history → "Recent changes: {timeline summary}."
   neutral_ack      → "Input received. No significant state changes detected."
   clarify_input    → "More detail needed: {suggestions}."

2. Fill template with data from AgentReplyPlan + AgentGroundingBundle

3. Append ALL safetyNotices

4. Mark source = "fallback"

5. Return LlmFallbackReply
```

### 9.3 Fallback Properties

- Deterministic: same plan → same fallback reply
- No network calls
- No LLM dependency
- Always includes safety notices
- Clearly marked as `source: "fallback"`
- Does not block the writeback pipeline (writeback is separate)

## 10. Provider Boundary

LLM providers are **injected** — never hardcoded.

### 10.1 Provider Interface (Conceptual)

```typescript
interface LlmProvider {
  readonly providerId: string;
  readonly model: string;
  complete(prompt: LlmBoundaryPrompt): Promise<LlmProviderResponse>;
  healthCheck(): Promise<boolean>;
}
```

### 10.2 Planned Provider Implementations

| Provider | Purpose | Timeline |
|----------|---------|----------|
| `MockLlmProvider` | Deterministic test provider; returns predefined responses | V13.3 |
| `OpenAiCompatibleProvider` | OpenAI-compatible API (OpenAI, local LLM, etc.) | V13.x (future) |
| `LocalModelProvider` | Local inference (e.g., Ollama) | V13.x (future) |

### 10.3 Provider Injection

```typescript
// Conceptual — NOT implemented in V13.0
const llmBoundary = createLlmBoundaryService({
  provider: new MockLlmProvider(),
  // provider: new OpenAiCompatibleProvider({ baseUrl: "...", apiKey: "..." }),
});
```

Provider configuration is NEVER embedded in prompt text. API keys are NEVER
included in any log, prompt, or DTO.

### 10.4 Testing

All core tests use `MockLlmProvider` only. No network calls in test suites.
The mock provider returns deterministic responses based on input hash, ensuring
reproducible test results.

## 11. Safety

### 11.1 Safety Principles (inherited from V10/V11/V12)

| Principle | How V13 Preserves It |
|-----------|---------------------|
| Simulation, not diagnosis | Validator rejects diagnosis language; safety notices always present |
| Not medical advice | Validator rejects prescriptive language |
| No autonomous writeback | Writeback plan never reaches LLM; confirmation gate unchanged |
| User confirmation required | Apply requires explicit confirmation via V12 gate |
| LLM does not decide truth | Grounding checker enforces fact-to-evidence mapping |

### 11.2 Defense in Depth

```
Layer 1: Prompt Builder
  → Curates what LLM can see; excludes raw state, secrets, writeback

Layer 2: LLM Output
  → Raw text from external provider

Layer 3: Output Validator
  → Regex + heuristic checks before output reaches user

Layer 4: Grounding Checker
  → Factual claim verification against provided evidence

Layer 5: Fallback
  → Deterministic reply when any layer fails
```

### 11.3 Safety Notice Preservation

The following notices MUST appear in or alongside every reply (LLM or fallback):

1. "This is CharacterOS simulation output, not medical or psychological diagnosis."
2. "Character state is a model calculation and does not represent real personality or clinical condition."
3. If writeback is allowed: "Event application requires explicit user confirmation."
4. If writeback is blocked: "This interaction will not modify character state."

## 13. Explicit Non-Goals

| Non-Goal | Reason |
|----------|--------|
| Production chat UI | Out of scope for V13; V13 is a service boundary |
| Direct OpenAI/anthropic integration | Provider is abstracted; first implementation is mock |
| Multi-character conversation | Single character only (V12 constraint) |
| Relationship engine | V12 `noMultiCharacter: true` |
| Hidden/implicit writeback | All writeback stays in V12 deterministic path |
| Medical/psychological diagnosis | Explicitly prohibited at all layers |
| Autonomous loop/scheduler | No agent autonomy; turn-driven only |
| User account system | Not a product; no auth beyond existing API auth |
| Streaming responses | V13.0–V13.9: synchronous request/response only |
| Conversation history/memory | Single-turn only; multi-turn continuity is V12's session state, not V13's concern |
| Prompt caching optimization | Premature optimization for design phase |
| Token counting/billing | Out of scope for V13 design |

## 14. Final Position

```
┌────────────────────────────────────────────────────────────┐
│                                                            │
│   V13 makes CharacterOS speak safely.                      │
│   It does NOT make CharacterOS think differently.          │
│                                                            │
│   Thinking = V10 Core + V11 Explorer + V12 Agent SDK       │
│   Speaking = V13 LLM Boundary Adapter                      │
│                                                            │
│   The boundary is thin.                                    │
│   The boundary is validated.                               │
│   The boundary is fallback-guaranteed.                     │
│   The boundary never mutates.                              │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### V13's Contract

1. **Input:** `AgentReplyPlan` + `AgentGroundingBundle` (both V12 output)
2. **Process:** Prompt construction → provider call → validation → grounding → output
3. **Output:** `AgentNaturalLanguageReply` or `LlmFallbackReply`
4. **Side effects:** NONE
5. **Failure mode:** Deterministic fallback, always safe, always available

### What Changes and What Doesn't

## Appendix A: Terminology

| Term | Definition |
|------|-----------|
| **LLM Boundary** | The outermost layer where natural language is produced from structured plans |
| **Prompt Builder** | Constructs a safe, bounded prompt for the LLM |
| **Provider Adapter** | Abstraction over a specific LLM provider (mock, OpenAI, local) |
| **Output Validator** | Rule-based checker for LLM output safety and correctness |
| **Grounding Checker** | Verifies every factual claim maps to provided evidence |
| **Fallback Reply** | Deterministic reply generated without LLM when validation/grounding fails |
| **AgentNaturalLanguageReply** | Validated, grounded natural-language reply (V13 output type) |
| **Language Realization** | The act of converting structured semantic content into natural language text |
