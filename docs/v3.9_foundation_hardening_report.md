
## 已完成修复

### 1. API 认证 / 授权边界

**新增文件：**
- `src/app/api/_shared/auth.ts` — 最小 API Key 认证中间件
- `tests/app/apiAuth.test.ts` — 16 个测试覆盖 auth 场景

**实现方案：**
- 通过环境变量 `CHARACTEROS_API_KEY` 控制
- 环境变量未设置时（本地开发），所有请求直接通过，不需要 header
- 环境变量设置后：
  - 所有 POST / PUT / DELETE route 需要 `x-api-key` header 匹配
  - GET route 当前保持开放（state/export/history 等只读端点）
  - `/decision` GET route 也受保护（因为会调用 LLM，消耗资源）
- 使用了 timing-safe 字符串比较减少时序泄露

**已接入 auth 的 route：**
- `characters/[characterId]/physics` — POST, DELETE
- `characters/[characterId]/physics/parse` — POST
- `characters/[characterId]/physics/simulate` — POST
- `characters/[characterId]/physics/tick` — POST
- `characters/[characterId]/physics/adjustment/apply` — POST
- `characters/[characterId]/physics/adjustment/rollback` — POST
- `characters/[characterId]/import/validate` — POST
- `characters/[characterId]/import/apply` — POST
- `characters/[characterId]/decision` — GET (LLM cost protection)

**未做：**
- 不做 OAuth / JWT / Session
- 不做用户管理
- 不做 RBAC
- 不做 rate limiting

### 2. 导入事务边界增强

**修改文件：**
- `src/core/export/characterImportApply.ts` — 新增类型和步骤
- `src/services/characterPhysicsService.ts` — 增强导入流程

**改进内容：**

a) 新增 `pre_mutation_snapshot_captured` transaction step：
   - 在修改任何仓库数据之前，捕获当前状态的完整序列化快照
   - 这个快照存入 trace 的 `stateRollbackSnapshot` 字段
   - 不是自动回滚机制 — 是给人工操作员的审计产物

b) 新增 `CharacterImportMutationOutcome` 类型：
   - `stateReplaced` — 状态是否已替换
   - `adjustmentHistoryReplaced` — 调整历史是否已替换
   - `historyRecorded` — 导入历史是否已记录
   - `fullyApplied` — 三个 mutation 是否全部成功
   - `description` — 人类可读的诊断信息

c) 逐步骤 mutation tracking：
   - `state_replaced` 现在被 try/catch 独立包裹
   - 如果状态替换成功但调整历史替换失败，trace 会清楚表达 "partial mutation"
   - `adjustment_history_replaced` 失败时，trace 会标记 `stateRollbackSnapshot` 可用

d) Transaction summary 现在包含 partial mutation warning：
   - 当 stateMutated=true 但 adjustmentHistoryMutated=false 时，summary reasons 会追加 WARNING

**未做：**
- 不实现真正 ACID 事务（需要 SQLite 或类似数据库支持跨表原子提交）
- 不实现自动 rollback（当前文件仓库不支持）
- 回滚策略：当前仅保留 pre-mutation snapshot 供人工恢复，文档已明确说明

### 3. Export / Import 验证深度

**修改文件：**
- `src/core/export/characterExportValidation.ts` — `validateSerializedState` 加深

**改进内容：**

之前的 state 验证只检查顶层字段类型（isRecord / isArray / typeof number）。
现在增加了：

a) Coordinate 深度验证：
   - 检查 `coordinate.values` 存在且为 object
   - 验证 Big Five 核心维度（openness, conscientiousness, extroversion, agreeableness, neuroticism）为 number
   - 验证扩展维度（trust, attachment, fear, control）如果存在则为 number（optional check）

b) Personality (BigFiveVector) 深度验证：
   - 验证五个核心维度均为 number

c) Cluster 元素验证：
   - 每个 cluster 必须有非空 string id
   - 必须有 string category
   - 必须有 number mass

d) Memory 元素验证：
   - 每个 memory 必须有非空 string id
   - 必须有 number recency
   - 必须有 number importance

**已有的深度验证（V3.8 已做，本次未改）：**
- History entry 逐条验证（id, characterId, action, status, snapshotId, operationCount, targetPaths, createdAt, reasons, governanceOverride）
- History summary 验证（totalEntries, appliedCount, rollbackCount, blockedCount, overrideCount, totalOperations, uniqueTargetPaths, latestTargetPaths, frequentTargetPaths, stabilityRisk, reasons）
- Governance 验证（recommendation, cooldownDays, cooldownActive, reasons）
- Package digest 验证（algorithm, canonicalization, excludedTopLevelFields, value sha256 format）
- State integrity report 验证（valid, errorCount, warningCount, issues, summary）

### 4. API 错误处理一致性

**修改文件：**
- `src/app/api/characters/[characterId]/physics/parse/route.ts` — 从 `request.json()` 改为 `readJsonBody`

**改进内容：**
- parse route 之前直接用 `await request.json()` 解析 body，非法 JSON 会抛出 500
- 现在使用 `readJsonBody`，非法 JSON 返回 `400 Invalid JSON body`
- 所有 character route 现在统一使用 `readJsonBody` + `resolveCharacterRouteParams`

**当前所有 route 的错误处理状态：**
- 所有 POST route 使用 `readJsonBody` → 非法 JSON 返回 400 ✓
- 所有 character route 使用 `resolveCharacterRouteParams` → 统一的 params 解析 ✓
- 所有 mutation route 使用 `requireAuth` → 未授权返回 401 ✓
- 缺失必填字段返回 400（如 "Missing event", "Missing description", "patch and snapshot are required" 等）
- Import apply 成功返回 200 / 冲突返回 409
- Import validate 成功返回 200 / 不可导入返回 422

## 未修复的问题及暂缓原因

### 暂缓：真正 ACID 导入事务

当前导入仍然是顺序 repo 操作。虽然新增了 pre-mutation snapshot 和 per-step mutation tracking，但这不是 ACID。

暂缓原因：
- 文件 JSON 仓库不支持跨文件原子提交
- 需要先决定是否迁移到 SQLite 或类似存储
- 当前 pre-mutation snapshot 提供了人工恢复的途径
- 建议在 V4.0 讨论 repository 迁移策略时一起解决

### 暂缓：runContinuousTick 拆分

虽然添加了完整的 phase 标注，但函数仍未拆分。

暂缓原因：
- 用户明确要求"不要一次重构"
- Phase 5-13（parameter adjustment pipeline）是纯 trace builders，可以安全提取
- Phase 3-4 和 14-15 直接 mutate state，拆分需要更小心
- 建议下一阶段先提取 Phase 5-13 为独立函数

### 暂缓：evaluateEventAttention 接入 processEvent

评估结果：应该接入，但当前不接入。

暂缓原因：
- 接入需要修改 `PhysicsStepResult` 类型（增加 attentionEvaluation 字段）
- 需要更新所有使用 `processEvent` 的测试和 API route
- 属于功能增强而非基础设施修复
- 建议作为 V4.0 的独立 feature 处理

### 暂缓：Calibration helpers 接入 API

四个 calibration hint builders 仍然只在测试中使用。

暂缓原因：
- 需要扩展 calibration report endpoint
- 当前 calibration route 只展示 replay calibration
- 建议在 V4.0 统一设计 calibration dashboard 时接入

### 暂缓：GET route auth 配置化

GET route 当前在 CHARACTEROS_API_KEY 设置后仍然开放。

暂缓原因：
- 用户明确"GET route 可以暂时开放"
- 未来可以加 `CHARACTEROS_API_KEY_PROTECT_GET=true` 来配置
- 当前只在 decision GET（消耗 LLM）上强制保护

## 下一阶段建议

### V4.0 优先级

```text
1. Repository 迁移策略
   - 是否从 JSON 文件迁移到 SQLite
   - 迁移后可以实现真正的导入 ACID 事务
   - 迁移后文件锁问题自然消失

2. Phase 5-13 纯函数提取
   - 从 runContinuousTick 提取 parameter adjustment pipeline
   - 不改变行为，只改善代码结构

3. evaluateEventAttention 接入 processEvent
   - 在 PhysicsStepResult 中增加 attentionEvaluation
   - 让每次事件处理都产生注意力评估

4. Calibration report 扩展
   - 将四个 calibration hint builders 接入统一的 calibration endpoint
   - 配合 replay calibration 形成完整的校准面板

5. API rate limiting
   - 当前只有 API key 认证，没有频率限制
   - 轻量方案：内存计数 + 429 响应
```

### 长期原则

```text
- 继续 API-only，不做前端
- 继续单角色，不做多角色/世界模拟
- 继续打磨基础设施，不扩张理论模块
```
