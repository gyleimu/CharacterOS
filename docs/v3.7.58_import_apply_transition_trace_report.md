# CharacterOS V3.7.58 Import Apply Transition Trace Report

## 目标

本阶段继续收敛无前端状态下的角色导入链路。

V3.7.57 已经让导入验证具备高层 `auditSummary`。V3.7.58 进一步打磨真正发生状态替换的 apply 阶段：当一个导入包通过验证并带有正确确认短语时，系统现在会记录替换前后的角色状态完整性报告。

这让“导入成功”不再只是一个布尔结果，而是一次可审计的状态转换。

## 新增能力

`CharacterImportApplyTrace` 新增：

```text
appliedAt
beforeStateIntegrity
afterStateIntegrity
```

语义如下：

- `appliedAt`：导入真正应用的时间。
- `beforeStateIntegrity`：目标角色被替换前的完整性检查结果。
- `afterStateIntegrity`：导入状态反序列化后的完整性检查结果。

如果导入被阻止，例如缺少确认短语、导入计划不是 ready、完整性失败、digest 不匹配，则不会产生应用转换快照。

## 行为变化

成功导入时：

```text
authorize import
-> inspect current target state
-> deserialize imported state
-> inspect imported state
-> replace state and adjustment history
-> return apply transition trace
```

失败导入时：

```text
return blocked trace
no state replacement
no appliedAt
no transition integrity snapshots
```

## 为什么重要

CharacterOS 暂时删除了用户可见前端，核心调试主要依赖 API、测试和报告。

导入是高风险操作，会替换一个角色的内部生命史、记忆星系、参数历史和状态结构。因此 apply 阶段必须能回答：

- 替换前角色状态是什么样？
- 替换后角色状态是否完整？
- 这次替换到底有没有真正发生？
- blocked 请求是否保持了无副作用？

V3.7.58 让这些问题可以直接从 trace 中得到答案。
