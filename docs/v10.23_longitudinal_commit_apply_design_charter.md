# CharacterOS V10.23 - Longitudinal Commit Apply Design Charter

**Date:** 2026-06-28
**Version:** V10.23
**Status:** Design Only, No Core Code
**Scope:** Safe apply design for longitudinal single-character commit

---

## 1. Executive Summary

V10.23 defines the apply path for longitudinal simulation commits.

V10.21 created the private `finalStateForCommit` handoff. V10.22 exposed a public commit preview without leaking `finalState`. V10.23 keeps persistence closed and specifies the exact gates required before a future API can turn a preview into durable character history.

**Verdict:** apply remains deferred. The next implementation should add a service-layer commit store and audit primitives before any route writes the simulated final state.

---

## 2. Why Apply Is Sensitive

Longitudinal commit is not a normal editor patch.

It can convert simulated future life into durable memory, and therefore can change how a character behaves later. It touches the deepest CharacterOS promise:

```text
Characters should continue changing even when nobody is watching.
```

That promise requires restraint. Continuous life must not become uncontrolled mutation.

## 4. Non-Goals

V10.23 does not:

- Implement the apply route.
- Persist simulated final state.
- Execute self-action candidates.
- Create background autonomous loops.
- Add multi-character simulation.
- Add world simulation.
- Add graph database storage.
- Add new benchmark cases.

## 5. Apply Principle

```text
Preview shows what could become history.
Apply requires consent, freshness, audit, and rollback.
```

An apply path must never accept a raw state from the client.

The client may submit:

- `simulationId`
- `requestDigest`
- `baseStateFingerprint`
- selected commit options
- operator confirmation

The server must recompute or retrieve the private handoff. It must not trust a client-provided final state.

---

## 6. Required Apply Gates

### Gate 1: Auth

Apply must require auth in all environments where `CHARACTEROS_API_KEY` is configured.

Local development may still run without auth, matching existing route policy, but tests must cover both modes.

### Gate 2: Explicit Confirmation

Apply must require a confirmation string, for example:

```text
APPLY_LONGITUDINAL_COMMIT
```

This prevents accidental writes from preview clients.

### Gate 3: Request Digest Match

The apply request must include the preview `requestDigest`.

The server must rebuild or retrieve the handoff and reject the apply if the digest differs.

### Gate 4: Base State Freshness

The current persisted character state must match `baseStateFingerprint`.

If the character changed after preview, apply must return 409 Conflict.

### Gate 5: Governance Pass

`LongitudinalCommitGovernanceDecision.status` must be `pass` or explicitly allowed warning.

`block` must never apply.

### Gate 6: Integrity Pass

The final state must pass structural integrity inspection.

Warnings may be returned, but integrity errors block apply.

### Gate 7: Self-Action Safety

Any commit surface change with source `self_action_candidate` blocks apply.

Self-actions remain candidates. They are not actions taken by the character.

### Gate 8: Rollback Plan Availability

Apply must produce a rollback plan before writing.

The rollback plan must at minimum include generated memory ids and stale-write policy.

---

## 7. Apply Request Contract

Recommended future route:

```text
POST /api/characters/[characterId]/life/simulate/commit/apply
```

Recommended body:

```json
{
  "simulationId": "longsim_...",
  "requestDigest": {
    "algorithm": "sha256",
    "canonicalization": "characteros-longitudinal-json-v1",
    "value": "..."
  },
  "baseStateFingerprint": {
    "algorithm": "sha256",
    "canonicalization": "characteros-longitudinal-json-v1",
    "value": "..."
  },
  "confirmation": "APPLY_LONGITUDINAL_COMMIT",
  "allowWarnings": false
}
```

The request should not include `finalState`.

---

## 8. Apply Response Contract

Recommended response:

```json
{
  "characterId": "lin_fan",
  "applied": true,
  "audit": {
    "id": "longcommit_audit_...",
    "simulationId": "longsim_...",
    "changedPathCount": 2,
    "generatedMemoryCount": 2,
    "governanceStatus": "pass"
  },
  "rollback": {
    "id": "rollback_...",
    "type": "remove_generated_memories",
    "generatedMemoryCount": 2,
    "staleWritePolicy": "block_if_changed"
  },
  "warnings": [],
  "reasons": []
}
```

The response should not include full state.

---

## 9. Service-Layer Requirement

Before a public apply route exists, the service layer needs an internal operation:

```ts
previewLongitudinalCommit(...)
applyLongitudinalCommit(...)
getLongitudinalCommitHistory(...)
rollbackLongitudinalCommit(...)
```

The route should not directly call `replaceState` with a simulated state. The service must own:

- stale-write checks
- state replacement
- audit recording
- rollback recording
- conflict handling

---

## 10. Audit Requirements

An applied longitudinal commit audit entry should include:

- audit id
- character id
- simulation id
- timestamp
- request digest
- base state fingerprint
- final state fingerprint
- changed paths
- generated memory ids
- commit policy
- governance status
- warning summary
- rollback id

It should not store the full final state in public history.

---

## 11. Rollback Requirements

Initial rollback should be conservative:

```text
remove generated memories only
```

Rollback must block if the current state fingerprint differs from the post-commit `finalStateFingerprint`.

This avoids deleting memories after the character has already evolved further.

---

## 12. Conflict Status Codes

Recommended status behavior:

| Status | Condition |
|---|---|
| 200 | Applied |
| 400 | Invalid JSON |
| 401 | Unauthorized |
| 404 | Character or preview not found |
| 409 | Base state fingerprint mismatch |
| 422 | Invalid request, governance block, integrity error |

---

## 13. Test Plan

Future implementation must test:

- auth required
- explicit confirmation required
- unknown character 404
- stale base state 409
- digest mismatch 422
- governance block 422
- integrity block 422
- no client-supplied final state accepted
- successful apply writes state exactly once
- successful apply records audit
- response excludes full state
- rollback removes generated memories only
- rollback blocks after further mutation
- repeated apply of same simulation is idempotent or blocked
- V2.1 benchmark remains stable

---

## 14. Recommended Implementation Sequence

1. V10.24 - Longitudinal commit audit repository and service types.
2. V10.25 - Service-layer apply/rollback core, no route.
3. V10.26 - Apply API behind auth and confirmation.
4. V10.27 - Rollback API.
5. V10.28 - Governance QA and V2.1 regression.

This slower route is intentional. Commit apply is the first place where simulated continuous life becomes durable life.

## 16. Final Statement

V10.23 does not make the character live more loudly.

It makes sure that when simulated life becomes history, the system knows who consented, what changed, why it changed, and how to undo the narrow part that can be undone.

That is the correct next step for CharacterOS.
