# CharacterOS V10.46 — Mind Galaxy Integration Boundary Design

**Date:** 2026-06-28
**Version:** V10.46
**Status:** Design Document — No Code
**Scope:** Define how the Mind Galaxy Viewer transitions from preview prototype to formal CharacterOS product entry

---

## 1. What Is the Current Preview?

The `previews/mind-galaxy-preview.html` is an **observation instrument prototype**.

### It IS:
- A Canvas 2D rendering of a character's personality structure as a nebula-like galaxy
- A single-character, read-only viewer
- Zoomable from outer static nebula (L0) to individual influence factors with slow drift vectors (L4)
- Deterministic — same data produces identical visual output
- Self-contained — one HTML file + one JS file, no server needed
- A validation tool — developers can see the V10.33–V10.36 data pipeline rendered visually

### It IS NOT:
- An editor — no node dragging, no parameter tuning, no save button
- A control panel — no commit/apply/rollback triggers
- A chat interface — no text input, no conversation
- A product page — no branding, no navigation, no user accounts
- A dashboard — no metrics panels, no time-series charts, no tables
- A multi-character tool — one character's mind at a time

### Its relationship to CharacterOS:
```
CharacterPhysicsState → Graph Pipeline → Galaxy View Snapshot → Canvas Render → Preview
     (core)              (V7/V8/V10.33)    (V10.33)            (V10.34/38)    (V10.38-45)
```

The preview is the **visual terminal** of the pipeline. It proves the data layers work. It does not yet serve as a product surface.

## 2. Three Candidate Integration Paths

### Path A: Static Artifact Mode

**What**: Continue exporting JSON + HTML. The viewer is a portable artifact.

**How**:
- `scripts/export-mind-galaxy-preview-data.ts` generates `mind-galaxy-real-data.json`
- HTML file opens locally or via static server
- No CharacterOS runtime needed after export

**Pros**:
- Zero server dependency — works offline
- No auth, no permissions, no deployment
- Ideal for sharing "here is what this character's mind looks like right now"
- Already working in V10.42+

**Cons**:
- Static snapshot — no live state
- Must re-export to see changes
- No interactive time-travel or state comparison

**Best for**: Offline observation, teaching, documentation, character state snapshots

---

### Path B: Read-Only Product Surface

**What**: A Next.js read-only page served by the CharacterOS API.

**How**:
- New `GET /product/[characterId]/galaxy` endpoint returns `GalaxyViewData`
- Next page renders Canvas 2D from the response
- Character selection via URL parameter
- No write paths, no editor, no commit buttons

**Pros**:
- Live state — always shows current character
- Integrated with CharacterOS runtime
- URL-addressable — share a link
- Natural step toward product surface

**Cons**:
- Requires Next.js page (React component, routing, server)
- Introduces deployment complexity
- Auth considerations for multi-user
- Performance — graph generation is CPU-bound per request
- Risk of feature creep (editors, controls, comparisons)

**Best for**: Live product surface, shared observation, "check on your character"

---

### Path C: Debug Instrument Mode

**What**: A developer tool with parameter controls for inspecting the pipeline.

**How**:
- Extended preview with controls already built (seed, scale, timeMs, drift mode)
- Additional debug panels: show raw graph data, inspect individual node metadata, compare snapshots
- Command-line flags for batch export
- JSON schema validation for exported artifacts

**Pros**:
- Leverages existing preview infrastructure
- No product surface complexity
- Immediate value for CharacterOS developers
- Teaches the pipeline internals through visual inspection

**Cons**:
- Not a product — developers only
- UI complexity scales with debug features
- Risk of over-engineering before the core is stable

**Best for**: Development, debugging, calibration, teaching the pipeline

---

## 3. Recommended Path: A + C First, B Later

### Phase 1 (Now): Path A + C — Artifact + Debug

1. **Freeze the preview data contract** (V10.47)
2. **Stabilize static artifact export** (V10.48)
3. **Add debug instrument controls** (V10.49)
4. **Declare Mind Galaxy stable** (V10.50)

### Phase 2 (Later): Path B — Product Surface

Only after:
- Product surface DTOs are finalized (from V10.31 design charter)
- Character Daily Surface API exists
- The product surface design has been validated against user needs
- Deployment and auth patterns are established

### Why not B now?

1. **Product pages introduce UI framework complexity** (React, routing, state management) that distracts from the core life simulation
2. **The core is still stabilizing** — V10 continuous life is complete, but the product experience layer hasn't been designed
3. **The "poetry-cloud" philosophy** from V10.31 says the product surface should be text-first, narrative, minimal. A full Next.js galaxy viewer page would compete with that direction
4. **Path A already serves the observation purpose** — export JSON, open HTML, observe. No server needed
5. **Path C already serves the debugging purpose** — the preview is already a debug tool

### What must NEVER be in a public DTO

- `CharacterPhysicsState` — raw state
- `finalState` — private handoff field
- Full `memories[]` — content, vectors, timestamps
- Raw `clusters` Map entries
- `proceduralRoutines` dump
- `serializedState`
- `coordinate.values` — personality coordinates
- `biologicalNature`
- `rewardState`, `homeostasisState`, `boredomState` internals
- Any `auditDraft`, `rollbackPlan`, `commitPolicy`
- File paths, API keys, LLM configuration

## 5. Next.js Page Boundaries (Future)

If a Next.js product page is built in the future, these boundaries are **non-negotiable**:

### MUST
- Read-only — no state mutation from the viewer
- Single character — `[characterId]` route param
- Canvas 2D rendering only
- Respect `prefers-reduced-motion`
- Support deterministic fixture mode (no server needed)
- Show clear loading/error states
- Cache graph snapshots (don't regenerate on every render)

### MUST NOT
- No editor — no drag, no parameter sliders, no save
- No write button — no commit/apply/rollback from viewer
- No auto-commit — no background persistence
- No autonomous scheduler trigger — no "run life simulation" button
- No multi-character comparison — one mind at a time
- No hidden state mutation — all data flows are observable
- No LLM integration — the viewer is not a chatbot
- No WebGL/Three.js — keep Canvas 2D
- No mobile native app — responsive web only, if needed

## 6. Mobile / Desktop Strategy

### Current: Desktop-first

The preview is designed for desktop browsers (mouse wheel, drag, hover). Canvas size is fixed at 900×700.

### Future: Responsive shell only

If a product page is built:
- **Canvas**: Resize to viewport, recalculate world-to-screen transforms
- **Touch**: Pinch-to-zoom, single-finger pan (no new core code)
- **Layout**: Stack controls vertically on narrow screens
- **No native app**: No iOS/Android builds. No React Native. No PWA requirement

### Why not mobile now

- The Mind Galaxy is a dense visual — benefits from larger screens
- Touch gesture handling adds complexity without improving the core
- Mobile-first would require responsive layout work that detracts from life simulation

## 7. WebGL / Three.js Position

### Current: Canvas 2D only

All rendering in V10.34–V10.45 uses Canvas 2D. This is sufficient for:
- < 2000 nodes (current LinFan: 40 nodes)
- Nebula glow effects (radial gradients)
- Simple geometric shapes (circles, lines, text)
- Drift vector arrows

### When to evaluate WebGL/Three.js

| Trigger | Threshold |
|---------|-----------|
| Node count | > 5000 nodes causing < 30fps in Canvas 2D |
| Visual complexity | Volumetric nebula rendering, 3D spatial relationships |
| Performance | Frame budget < 16ms at L4 with real data |
| Product need | Users request true 3D navigation |

### What WebGL would enable (future)
- True 3D particle nebula with volumetric lighting
- GPU-accelerated force-directed layout
- Smooth LOD transitions at 60fps with 10k+ nodes
- Actual Z-depth for cluster layering

### What WebGL would NOT change
- The data pipeline (V10.33–V10.37) — renderer is a consumer, not a producer
- The zoom/LOD system (V10.35) — same bands, same visibility rules
- The drift animation data (V10.36) — same elliptic orbits
- The philosophy — still an observation instrument

### Current stance
**Canvas 2D is sufficient. WebGL/Three.js is not forbidden forever, but is deferred until a concrete performance or visual need exceeds Canvas 2D's capabilities. CharacterOS is about believable life, not about the rendering engine.**

## 10. Core Conclusions

1. **The preview is an observation instrument, not a product.** It proves the pipeline works. It is not yet a user-facing surface.

2. **Path A (Static Artifact) + Path C (Debug Instrument) come first.** Path B (Product Surface) is deferred until the product experience layer is designed and the core is stable.

3. **DTO boundaries are clear.** Public DTOs expose graph projections. They never expose raw CharacterPhysicsState, clusters, memories, or internal audit data.

4. **Canvas 2D is sufficient for now.** WebGL/Three.js is a future option gated by concrete performance thresholds, not a current priority.

5. **The Mind Galaxy Viewer is part of CharacterOS, not the whole product.** It is the visual observation layer. The core remains the life simulation engine.
