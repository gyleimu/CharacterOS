## 3. Adapter Shell 设计

### 每个 adapter 声明

```typescript
interface TemporalProcessAdapter {
  processId: string;                          // 匹配 metadata id
  sourcePhase: number;                        // 匹配 metadata phase
  wrapsExistingFunction?: string;             // 未来委托的 V3 函数名
  mutationPolicy: "none" | "delegates_to_v3" | "future";
  implementationStatus: TemporalProcessAdapterStatus;
  notes: string;                              // 迁移说明
  risks: string[];                            // 已知风险
}
```

### 当前状态

```text
所有 17 个 adapter 的 implementationStatus = "adapter_shell"
零个 adapter 的 implementationStatus = "delegated" 或 "native"
零个 adapter 调用任何 V3 函数
零个 adapter 执行任何 mutation
```

### V4.2 不做什么

```text
不委托任何 V3 函数                   ← 只声明，不调用
不执行任何 mutation                  ← mutationPolicy 只是计划
不替换 runContinuousTick             ← V3 行为完全不变
不改变 V3 测试                        ← 256 个测试全部通过
```

## 5. 风险分析

### 跨-phase 依赖

| 依赖 | 描述 | 缓解 |
|------|------|------|
| Phase 2 → Phase 3 | metaState.forgettingSpeed 必须在 decay 之前更新 | 保留顺序 |
| Phase 3 → Phase 4 | homeostasis 有意覆盖 decay_and_recovery 的 boundary/reward/meta | 保留顺序 |
| Phase 4 → Phase 5 | recovery_trace 读取 homeostasis 后的值 | 保留顺序 |
| Phase 3 → Phase 15 | belief_evolution 读取已 decay 的 memories | 保留顺序 |

**结论：V4 拆分时必须保留 phase 顺序约束。**

### 最大风险的 adapter

| Adapter | 风险 |
|---------|------|
| decay_and_recovery | 最大的 composite phase，4 个子操作。拆分可能改变顺序。 |
| homeostasis | 覆盖 Phase 3 输出，移除会导致 double-recovery。 |
| time_perception | 应该是共享 context，不是独立 process。V4 可能需要重新设计。 |

### 最低风险的 adapter

| Adapter | 原因 |
|---------|------|
| meta_drift | 单一 V3 函数，已测试，低耦合 |
| boredom | 单一 V3 函数，只写 boredomState |
| belief_evolution | 单一 V3 函数，只写 beliefStates |
| 所有 trace phases | 不改变状态，纯函数，零风险 |
