## 3. Core Readiness Rules

`evaluateLongitudinalCommitApplyReadiness()` checks:

- confirmation equals `APPLY_LONGITUDINAL_COMMIT`
- current state fingerprint matches preview base state fingerprint
- governance is not `block`
- governance warnings require `allowWarnings=true`

Readiness statuses:

- `ready`
- `blocked`
- `conflict`

---

## 4. Service Apply Behavior

`CharacterPhysicsService.applyLongitudinalCommit(handoff, options)`:

1. Verifies character exists.
2. Computes current persisted state fingerprint.
3. Evaluates readiness.
4. Records blocked/conflict audit if readiness fails.
5. Writes `handoff.finalState` only when readiness is `ready`.
6. Records applied audit.
7. Returns compact apply result.

The method accepts only the private server-side handoff. It does not accept a client-provided final state.

## 6. Safety Boundaries

V10.25 still does not:

- add public commit apply API
- accept `finalState` from clients
- execute self-action candidates
- start background life loops
- bypass governance
- bypass stale-write checks
- persist full final state in audit history

---

## 7. Test Coverage

Core readiness tests:

- ready when confirmation/fingerprint/governance pass
- blocked when confirmation is missing
- conflict when state fingerprint changed
- blocked when governance blocks

Service tests:

- applies ready handoff and records applied audit
- blocks without confirmation and does not write state
- conflicts after state changed post-preview
- returns not_found for missing character
- blocks governance-disabled handoff
