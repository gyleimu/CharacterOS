# V11.0: CharacterOS Explorer — Design Charter

**Date:** 2026-07-02
**Version:** V11.0 Design Charter
**Previous:** V10.80 (Post-RC Repository Hardening)
**Status:** Design Phase — No Implementation

---

## 1. V11 Definition

### What V11 Is

**CharacterOS Explorer** is a single-character personality exploration platform.

A user can:
- Input events that happen to the character
- Observe the character's current psychological state
- Understand *why* the character changed
- Explore the Mind Galaxy (advanced view)
- Verify simulation authenticity with Reality Audit
- Travel back in time to see how personality evolved

### What V11 Is NOT

| Not this | Reason |
|----------|--------|
| ❌ Demo / product showcase | Explorer is interactive, not read-only |
| ❌ Chat Agent | No dialogue, no "talk to the character" |
| ❌ Multi-character | Single character core, single character platform |
| ❌ UI polish of V10 | New architecture, not reskinning |
| ❌ Autonomous scheduler | User drives events, system does not self-run |
| ❌ Mobile / native app | Web platform only |
| ❌ Server deployment | Local-first, single-machine |
| ❌ Medical / psychological diagnosis | Simulation, not clinical tool |

### Core Thesis

> **V11 should let a user explore one personality's life — not chat with a character.**

## 3. Six Modules

### 3.1 Event Studio

**User Goal:** Input an event and see what will happen before committing.

**Core Actions:**
1. User types/pastes natural language event description
2. User optionally sets: time, location, characters involved, intensity
3. System parses event → `ParsedExperienceEvent`
4. System previews: category, emotion, impact score, coordinate delta
5. System previews: expected channel activation (Reality Audit light)
6. User reviews and confirms → event applied to character
7. Repeat: user can specify "apply N times" for accumulation testing

**Reuse:**
- `parseExperienceEvent()` — rule or LLM parser
- `POST /api/characters/[id]/physics/parse` — existing parse endpoint
- `runImpactCalibrationAudit()` — channel activation preview
- `POST /api/characters/[id]/physics` — event application
- Commit preview/apply/rollback — existing longitudinal API

**New DTOs:**
- `EventStudioPreview` — parse result + impact preview + reality check
- `EventStudioApplyRequest` — confirmed event + repeat count
- `EventStudioApplyResult` — before/after state summary

**Read/Write Boundary:**
- Preview: **read-only** (clone state, simulate, discard)
- Apply: **write** (requires confirmation, records in history)

**Safety:**
- Preview always runs on a clone — never mutates live state
- Apply requires explicit confirmation
- Reality audit warnings shown in preview before apply
- Repeat events show accumulation preview (not just single-step)

**Does Not Do:**
- ❌ Auto-generate events (no autonomous scheduler)
- ❌ Batch import from external sources
- ❌ Undo without rollback (rollback is explicit)

### 3.2 Character State

**User Goal:** See who the character is *right now* — a human-readable summary, not raw variables.

**Core Actions:**
1. View personality summary (trust level, fear level, openness, attachment style)
2. View current emotion, stress level, boundary phase
3. View top beliefs (what the character currently believes)
4. View active needs (what the character currently lacks)
5. View current desires/goals
6. View behavioral tendencies (what they're likely to do next)
7. View "state at a glance" dashboard card

**Reuse:**
- `DerivedCharacterState` (from `deriveCharacterState`)
- `GET /api/characters/[id]/physics` — existing state endpoint
- `coordinateToRecord()` — personality values
- `PsychologicalBoundary` phase/stress — boundary status

**New DTOs:**
- `CharacterStateSurface` — human-readable summary
  ```
  {
    identity: { name, description },
    personalitySummary: { trust, fear, openness, attachment, ... },
    emotionalState: { primary, valence, arousal },
    boundaryStatus: { phase, stressLevel, integrity },
    topBeliefs: [{ content, strength }],
    activeNeeds: [{ name, intensity }],
    currentDesires: [{ content, intensity }],
    behavioralTendency: { likelyAction, strategyLabel },
    lastEvent: { description, time, impact }
  }
  ```

**Read/Write Boundary:**
- **Read-only** — no state mutation from viewing

**Safety:**
- Does not expose raw coordinate values (e.g., 0.2647)
- Shows qualitative labels: "低信任" / "高恐惧" / "压力超载"
- Shows source grounding: "因为三天前王雪失联..."

**Does Not Do:**
- ❌ Expose hundreds of raw variables
- ❌ Allow direct parameter editing (use existing adjustment API separately)
- ❌ Show internal IDs, cluster hashes, particle IDs

### 3.3 Explainability

**User Goal:** Understand *why* the character changed. "Why am I more anxious today than 7 days ago?"

**Core Actions:**
1. View event chain: "These events happened → these things changed"
2. View memory chain: "This memory was formed → it influenced these beliefs"
3. View belief chain: "This belief strengthened → it activated these needs"
4. View need/desire chain: "This need emerged → it drove this desire"
5. View personality delta: "Trust decreased by X because..."
6. View decision delta: "Before this event, the character would have done A; now they do B"
7. Time-range explainability: "Past 7 days / 30 days / since last milestone"

**Reuse:**
- `RealityAuditExplanationTrace` — existing delta path tracing
- `DecisionInfluenceLayerResult` — decision surface delta
- `buildDecisionInfluenceLayer()` — influence vector computation
- `diffMemories / diffBeliefs / diffCoordinate / diffNeeds` — existing diff functions

**New DTOs:**
- `ExplainabilityTimeline` — time-ordered chain of cause→effect
  ```
  {
    timeRange: { from, to },
    eventChain: [{ event, memory, beliefImpact, needImpact }],
    personalityDelta: { before, after, perDimensionDelta },
    decisionDelta: { beforeStrategy, afterStrategy, influenceVectors },
    topContributors: [{ factor, contribution, evidence }]
  }
  ```
- `WhyAmIAnxious` — specialized anxiety trace
  ```
  {
    currentAnxietyLevel,
    contributingEvents: [{ event, daysAgo, contributionWeight }],
    reinforcedBeliefs: [{ belief, howItContributes }],
    activatedNeeds: [{ need, intensity }]
  }
  ```

**Read/Write Boundary:**
- **Read-only** — analysis only, no state change

**Safety:**
- Every explanation cites a grounded delta path (V10.68 requirement)
- No fabricated explanations — if a delta has no clear source, it says "unclear"
- Reality audit verdict shown alongside explanation

**Does Not Do:**
- ❌ Generate narrative justifications (no LLM storytelling)
- ❌ Make up causal links that don't exist in the data
- ❌ Hide uncertainty — if the system can't explain something, it admits it

### 3.4 Mind Galaxy

**User Goal:** Advanced view for authors, developers, and researchers who want to see the underlying structure.

**Core Actions:**
1. Navigate to Advanced → Mind Galaxy (not the default view)
2. See cluster visualization: which life experiences form which psychological clusters
3. See gravitational forces: which clusters are pulling personality in which directions
4. See cluster metrics: mass, density, stability, age
5. See personality trajectory: where the core is and where it's drifting
6. Toggle between: cluster view, force view, trajectory view

**Reuse:**
- Mind Galaxy graph projection (`src/core/graph/`)
- `GET /api/characters/[id]/graph` — existing graph endpoint
- `outputs/characteros-demo/mind-galaxy/` — existing artifact
- `ImpactCluster` metrics — mass, density, stability
- `PersonalityCoordinate` — core position

**New DTOs:**
- `MindGalaxySurface` — galaxy state for visualization
  ```
  {
    clusters: [{ category, mass, stability, particleCount, dominantDirection }],
    forces: [{ clusterId, magnitude, trustForce, fearForce }],
    corePosition: { trust, fear, openness, ... },
    trajectory: [{ timestamp, corePosition }]
  }
  ```

**Read/Write Boundary:**
- **Read-only** — observation only

**Safety:**
- Clearly labeled as "Advanced / Research View"
- Not the default user entry point
- Includes disclaimer: "This is a simulation model, not a clinical diagnostic tool"

**Does Not Do:**
- ❌ 3D visualization (2D graph projection only)
- ❌ Real-time animation (static snapshot with refresh)
- ❌ Direct cluster manipulation (no drag-and-drop editing)

### 3.5 Reality Audit

**User Goal:** Understand that this is a simulation, not medical truth.

**Core Actions:**
1. See audit verdict: PASS / WARN / FAIL for the current state
2. See state diff: what changed from last event
3. See decision diff: how the character's choices changed
4. See explanation grounding: are the explanations backed by real data?
5. See channel calibration: is the impact going to the right channels?
6. See regression risks: are any V10 fixes being violated?

**Reuse:**
- `runRealityAudit()` — full audit for current state
- `runCoreRealityRegressionGate()` — regression gate
- `RealityAuditResult` — full audit output
- `ImpactCalibrationAuditResult` — channel calibration
- `DecisionResponsiveness` audit

**New DTOs:**
- `RealityAuditSurface` — user-facing audit summary
  ```
  {
    verdict: "PASS" | "WARN" | "FAIL",
    stateChanged: boolean,
    decisionResponded: boolean,
    explanationGrounded: boolean,
    boundarySafe: boolean,
    neutralStable: boolean,
    warnings: [{ message, severity, source }],
    disclaimers: ["这是模拟结果，不是医学诊断", ...]
  }
  ```

**Read/Write Boundary:**
- **Read-only** — audit only, no state change

**Safety:**
- **Primary purpose is safety** — prevents users from misinterpreting simulation as fact
- Every page that shows character state must show audit verdict
- Disclaimers are not hidden in footnotes — they are co-located with state display
- If audit is WARN or FAIL, the state display shows a prominent warning banner

**Does Not Do:**
- ❌ Hide warnings to make the system look more reliable
- ❌ Present simulation output as objective truth
- ❌ Allow users to dismiss audit warnings permanently

### 3.6 Time Machine

**User Goal:** Travel back in time. "How did I become who I am today?"

**Core Actions:**
1. See timeline: Day 1 / Day 7 / Day 30 / Day 90 / Today
2. Select a point in time → view character state at that moment
3. See what events happened between then and now
4. See how personality, beliefs, needs, and galaxy changed
5. Restore a historical view (read-only — does not change current state)
6. Compare two points in time side by side

**Snapshot Strategy:**
- Every event application creates an implicit snapshot (transition history)
- Explicit snapshots can be created at milestones
- Snapshots store: full `CharacterPhysicsState` serialized
- Snapshots are immutable — restoring creates a *view*, not a *rollback*
- Rollback is a separate explicit action (existing rollback API)

**How To Restore Galaxy/State at Time T:**
1. Find the snapshot closest to (but not after) time T
2. If T is between snapshots, load the earlier snapshot and replay events up to T
3. Replayed events are applied to a *clone* — live state is never mutated
4. The restored view is presented as "Historical View: Day 30"

**Storage Implications:**
- Each snapshot ≈ 50–200KB (serialized CharacterPhysicsState)
- 100 snapshots ≈ 5–20MB — manageable for local storage
- Snapshot compression: only store deltas between consecutive states (optional optimization)
- Snapshot retention policy: keep milestones, thin intermediate snapshots

**Reuse:**
- `serializeCharacterPhysicsState` / `deserializeCharacterPhysicsState` — state serialization
- Export/import pipeline — snapshot save/load
- Transition history (`GET /api/characters/[id]/import/history`) — existing history pattern
- `runContinuousTick` — forward simulation between events
- Trace replay artifact format — snapshot format inspiration

**New DTOs:**
- `TimeMachineTimeline` — list of available snapshots
  ```
  {
    snapshots: [{ id, timestamp, label, eventCount, milestone }],
    currentPosition: { snapshotId, isLive }
  }
  ```
- `TimeMachineView` — state at a historical point
  ```
  {
    snapshotId, timestamp, isHistoricalView: true,
    characterState: CharacterStateSurface,
    galaxyState: MindGalaxySurface,
    eventsSince: [{ description, time, impact }],
    personalityDeltaFromNow: { trust, fear, openness, ... }
  }
  ```
- `TimeMachineComparison` — side-by-side diff
  ```
  {
    pointA: { timestamp, state },
    pointB: { timestamp, state },
    deltas: { personality, beliefs, needs, boundary },
    keyEventsBetween: [...]
  }
  ```

**Read/Write Boundary:**
- Historical view: **read-only** (no mutation)
- Rollback: **write** (explicit confirmation, existing API)
- Create snapshot: **write** (adds to snapshot list)

**Safety:**
- Historical view is clearly labeled "Historical View — Read Only"
- Rollback requires explicit confirmation and records audit trail
- Current (live) state is always clearly distinguished from historical views
- Cannot accidentally overwrite current state by viewing history

**Does Not Do:**
- ❌ Auto-snapshot on every tick (too much storage)
- ❌ Undo without audit trail
- ❌ Branch into alternate timelines
- ❌ Delete history (append-only)

## 7. Final Judgment

> **V11 should let a user explore one personality's life — not chat with a character.**

The Explorer is an observation platform. The user is a witness to a simulated mind, not a participant in a conversation. Every module is designed around understanding and verifying, not interacting or role-playing.

The Reality Audit is the conscience of the system. The Time Machine is its memory. The Explainability module is its honesty. The Mind Galaxy is its depth. The Event Studio is its input. The Character State is its surface.

Together, they form a complete single-character personality exploration platform — faithful to the V10 core, honest about its limitations, and ready to be built.
