## 1. 两次 Delegation 的模式稳定性

### Adapter 文件结构对比

| 维度 | Boredom (V4.7) | Belief Evolution (V4.8) | 一致性 |
|------|---------------|------------------------|--------|
| 文件行数 | 102 | 93 | ✓ 同量级 |
| 输入类型 | BoredomTemporalAdapterInput | BeliefEvolutionTemporalAdapterInput | ✓ 同模式（mirror V3 params） |
| 输出类型 | BoredomTemporalAdapterResult | BeliefEvolutionTemporalAdapterResult | ✓ 同模式（trace + phase + temporalResult） |
| V3 委托 | `updateBoredomForTick(...)` | `evolveBeliefsForTick(...)` | ✓ 直接委托 |
| phase.name | `"boredom"` | `"belief_evolution"` | ✓ 匹配 V3 phase name |
| phase.changedStates | `["boredomState"]` | `["beliefStates"]` | ✓ 匹配 V3 |
| phase.reasons | `[...trace.reasons]` | `[...trace.reasons]` | ✓ 透传 |
| temporalResult.before | 5 字段直接映射 | count + avgStrength 聚合 | ⚠️ 聚合方式不同（见下方分析） |
| temporalResult.after | 5 字段直接映射 | count + avgStrength 聚合 | ⚠️ 聚合方式不同 |

**temporalResult.before/after 差异分析：**

Boredom adapter 的 `before/after` 是 boredomState 的 5 个字段的浅拷贝。Belief evolution adapter 的 `before/after` 聚合为 `{count, avgStrength}`。这是有意的设计差异——beliefStates 是可变长度数组，不适合逐条映射到 TemporalProcessResult 的 `Record<string, unknown>` 中。

**结论：这种灵活性是 adapter 模式的优势，不是缺陷。** 每个 adapter 可以决定自己的 temporalResult shape。checklist 不需要强制统一 before/after 格式。

### Phase Replacement Pattern

```text
V4.7 (boredom):
  原: const boredom = updateBoredomForTick({...})
      state.boredomState = boredom.after
      phases.push({ name:"boredom", ... })

  新: const boredomResult = executeBoredomTemporalAdapter({...})
      state.boredomState = boredomResult.trace.after
      phases.push(boredomResult.phase)

V4.8 (belief_evolution):
  原: const beliefEvolution = evolveBeliefsForTick({...})
      state.beliefStates = beliefEvolution.after
      phases.push({ name:"belief_evolution", ... })

  新: const beliefEvolutionResult = executeBeliefEvolutionTemporalAdapter({...})
      state.beliefStates = beliefEvolutionResult.trace.after
      phases.push(beliefEvolutionResult.phase)
```

**模式：** 100% 一致。未来 delegation 可以直接复制此模板。

### Registry Status Update Pattern

```text
V4.7: boredom: "adapter_shell" → "delegated"
V4.8: belief_evolution: "adapter_shell" → "delegated"

每次 delegation 只改一行（一个 adapter 的 implementationStatus）。
```

```text
回滚时间：< 5 分钟每个 delegation。
```

## 3. 当前 Execution Layer 风险分析

### R1: Duplicated Mutation — LOW

```text
当前风险：低。
原因：每次 delegation 都注释/替换了 V3 inline 代码。
      没有保留旧代码 + 新 adapter 同时执行的情况。
缓解：代码 review 可以一眼看出替换是否完整。
      如果 adapter 写 state.field = result.trace.after
      而旧代码仍然写 state.field = oldResult.after，会产生 double-write。
      当前代码中没有这种情况。
```

### R2: Phase Order Drift — LOW

```text
当前风险：低。
原因：delegation 不改变 phase 在 tick 中的位置。
      Adapter 调用在 V3 代码中的位置 == 原 inline 代码的位置。
      没有引入动态排序或 registry-based ordering。
缓解：D4 测试验证了 phase 在 phases 数组中的 index。
```

### R3: Adapter Over-Abstraction — LOW

```text
当前风险：低。
原因：每个 adapter 只有 ~95 行。
      没有引入基类、工厂模式、装饰器等复杂抽象。
      Adapter 只是薄包装层（wrap + return）。
缓解：如果 adapter 行数超过 150 行，或者引入继承层次，应该停止。
      当前两个 adapter 都在 100 行左右。
```

### R4: Trace Mismatch — LOW

```text
当前风险：低。
原因：adapter 返回的 phase 对象与原 inline phases.push 参数完全一致。
      trace 字段（boredom/beliefEvolution）通过 adapterResult.trace 透传。
缓解：D2 测试验证了 V3 trace 的 sub-fields 完整。
      V4.7.1 补测了 reasons flow-through 和 inspiration 等价性。
```

### R5: State Reference Mutation — LOW

```text
当前风险：低。
原因：adapter 本身不写 state。调用方负责 state.field = result.trace.after。
      V3 函数（updateBoredomForTick, evolveBeliefsForTick）已经设计为
      接收值参数并返回新值，不修改传入的引用。
缓解：D7 测试验证了 adapter 不修改 other fields。
```

### R6: Test Brittleness — LOW

```text
当前风险：低。
原因：delegation 测试不依赖精确数值。
      等价性测试用 toEqual 比较整个 trace 对象。
      顺序测试用 phase index 而非内容。
缓解：如果未来 V3 参数调优改变了 boredom/belief 的输出值，
      delegation 测试仍然通过（因为 adapter 和直接调用使用相同的输入）。
```

### R7: Registry Count Drift — LOW

```text
当前风险：低。
原因：每次 delegation 后都更新了 registry 测试的硬编码计数。
      当前：15 adapter_shell, 2 delegated, 0 native, 0 metadata_only。
缓解：summarize() 方法自动统计，不依赖硬编码。
      测试中的硬编码计数只是双重验证。
```

## 4. 剩余 3 个 Candidate 评估

### 4.1 meta_drift (Phase 2)

```text
V3 函数：updateMetaStateForTick(meta, daysElapsed, stressLoad, boundaryIntegrity)
写入字段：metaState (13 sub-fields)
下游消费：Phase 3 (decay_and_recovery) 读取 metaState.forgettingSpeed

Risk: MEDIUM
原因：
- metaState 有 13 个 sub-fields，是所有 mutation phase 中字段最多的
- 被 Phase 3 消费（forgettingSpeed），phase order 依赖强
- V3 函数已存在且独立，可以直接委托

Why next: 低风险的单一函数委托。Phase 2 在所有 mutation 之前执行，
          没有覆盖语义。比 decay_and_recovery 和 homeostasis 简单得多。

Why not next: 13 个 sub-fields 的 before/after 在 temporalResult 中
              需要聚合（类似 belief evolution），需要决定聚合方式。

Required tests: 与 boredom/belief 相同的 13-15 tests。
                额外需要：验证 metaState.forgettingSpeed 正确传递给 Phase 3。

Rollback complexity: 低。与 boredom/belief 相同的 6 行恢复模式。

推荐：V4.10 的首选。是剩余 3 个中最简单的。
```

### 4.2 decay_and_recovery (Phase 3)

```text
V3 函数：4 个子操作（decayMemory + recoverBoundary + recoverRewardBaseline
         + decayProceduralRoutine），全部 inline
写入字段：memories, proceduralRoutines, rewardState, boundary, metaState (5 个)
下游消费：Phase 4 (homeostasis) 覆盖 boundary/reward/meta

Risk: HIGH
原因：
- 不是单一函数委托。是 4 个独立子操作的内联组合。
- 拆分需要决定：每个子操作是独立 TemporalProcess 还是保持 composite？
- 如果是独立的：需要 4 个 adapter，需要定义它们之间的依赖顺序。
- 如果是 composite：adapter 内部仍然需要执行 4 个子操作。
- Phase 4 覆盖其输出，覆盖语义必须保留。

Why not next: 
- 复杂度远超 boredom/belief_evolution
- 没有单一的 V3 函数可以委托
- 需要先决定 composite vs split 的架构选择
- 当前 delegation 模板不适合 composite phase

Required tests: 不确定（取决于 composite vs split 的决策）

Rollback complexity: 中等。需要恢复 4 个子操作的 inline 代码。

推荐：继续暂缓。在 meta_drift 和 homeostasis 都成功 delegation 后，
      有了更多复合委托的经验，再回来处理 decay_and_recovery。
```

### 4.3 homeostasis (Phase 4)

```text
V3 函数：applyHomeostasis(homeostasis, meta, boundary, reward, daysElapsed)
写入字段：homeostasisState, metaState, boundary, rewardState (4 个)
下游消费：Phase 5-13 (trace builders) 读取 regulated values
覆盖语义：有意覆盖 Phase 3 的 boundary/reward/meta 输出

Risk: MEDIUM
原因：
- 单一 V3 函数，可以委托
- 但是有覆盖语义——Phase 4 的 regulatedBoundary 等值必须覆盖 Phase 3 的输出
- 如果 adapter 返回的 after 没有被正确写回 state，会出现 "double recovery"
  （Phase 3 恢复 + Phase 4 应该在此基础上精调，但如果 Phase 4 被跳过，
   state 保留 Phase 3 的输出，缺少精调）
- 覆盖语义在当前 inline 代码中是隐式的（state.boundary = homeostasis.regulatedBoundary）
- 委托时必须同样显式写回

Why next (after meta_drift):
- 单一 V3 函数，与 boredom/belief 模式相同
- 是 V4 的第一公民原则——最有价值的 delegation

Why not next:
- 覆盖语义需要在测试中显式验证
- 需要验证 Phase 3 的输出确实被 Phase 4 覆盖

Required tests:
- 与 boredom/belief 相同的 13-15 tests
- 额外：验证 Phase 4 输出覆盖 Phase 3 输出
  测试方法：snapshot boundary/meta/reward 在 Phase 3 之后、Phase 4 之后的值
  确认 Phase 4 之后的值 ≠ Phase 3 之后的值（不等于意味着覆盖发生了）

Rollback complexity: 低。与 boredom/belief 相同的 6 行恢复模式。

推荐：V4.11 的第二候选。在 meta_drift (V4.10) 成功之后执行。
```
