V10.21 implements the type surface designed in V10.20 without opening commit persistence.

- deterministic request digest
- deterministic state fingerprint
- commit surface extraction
- governance decision
- audit draft
- rollback plan

No API route was added. No `replaceState()` path was added. No longitudinal commit apply exists yet.

## 5. Governance Rules Implemented

| Rule | Behavior |
|---|---|
| `commitPolicy.enabled` must be true | Otherwise governance status is `block` |
| Self-action candidate changes are blocked | Any surfaced `self_action_candidate` change blocks |
| Final state integrity must pass | Uses `inspectCharacterStateIntegrity()` |
| Public preview strips final state | Tested by JSON inspection |
| Rollback is narrow | Generated memory ids only |

Self-action candidates remain signals only.

---

## 6. Commit Surface

`buildLongitudinalCommitSurface()` extracts:

- applied flag
- total/committed steps
- generated memory count
- compact before/after summaries
- surfaced applied changes
- surfaced skipped changes

For generated memory ids, the helper resolves the real id from `commitResult.state.memories[index]` when available. This matters because current V10.9 `LifeTickCommitChange.to` records memory content, not the id.

---

## 7. Fingerprint Policy

`computeLongitudinalStateFingerprint()` projects stable persisted fields:

- identity
- memories
- beliefs
- coordinate / velocity
- boundary
- meta state
- reward/homeostasis/boredom state
- procedural routines
- particles
- clusters
- learning rate

The fingerprint intentionally excludes derived state and graph layout. It is meant for stale-write protection, not export identity.

Known caveat: generated memory timestamps are still created by `new Date().toISOString()` inside `lifeTickPersistence.ts`. V10.22/V10.23 should resolve deterministic generated timestamps before opening apply persistence.

---

## 8. Rollback Model

V10.21 rollback is intentionally narrow:

```typescript
type = "remove_generated_memories"
staleWritePolicy = "block_if_changed"
```

It only knows how to remove generated memory ids. It does not reverse scalar state changes, execute actions, or alter life state fields.

This is appropriate because first longitudinal commits should only append generated memory seeds.

## 9. Privacy Boundary

Private object:

- may include `finalState`
- must stay inside core/service layer

Public preview:

- includes fingerprints
- includes commit surface
- includes audit summary
- includes rollback summary
- includes governance decision
- does not include full `CharacterPhysicsState`

Tests assert the public preview does not contain `"finalState"`.

## 13. Known Limitations

- No apply route yet.

- No persistent longitudinal audit repository yet.
- Rollback only covers generated memories.
- Generated memory timestamps need deterministic design before apply opens.
- Current `/life/simulate` route remains dry-run/deferred; V10.21 helpers are not wired into an API.

Recommended next step: **V10.22 Longitudinal Commit Preview**.

V10.22 should expose a preview route or service-level preview function that:

- requires auth according to write-preview policy
- builds `LongitudinalFinalStateForCommit`
