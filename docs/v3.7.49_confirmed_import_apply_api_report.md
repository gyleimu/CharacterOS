# CharacterOS V3.7.49 Confirmed Import Apply API Report

## Goal

After removing the visible frontend, CharacterOS still needs a safe way for a human operator to participate in data adjustment.

This update adds a conservative import application path:

```text
export package
-> validate package
-> build import plan
-> require explicit confirmation phrase
-> replace state and adjustment history
```

This is not automatic migration.

This is an audited manual operation.

## Core Authorization

Added:

```text
src/core/export/characterImportApply.ts
```

It provides:

```text
authorizeCharacterImportApplication
buildImportConfirmationPhrase
```

The confirmation phrase is:

```text
replace:<characterId>
```

The import is blocked when:

```text
package validation fails
import plan is blocked
import plan needs review
confirmation phrase is missing
confirmation phrase does not match
deserialization fails
```

High-risk packages remain blocked even if the caller provides the confirmation phrase.

## Service Integration

Updated:

```text
src/services/characterPhysicsService.ts
```

The service now exposes:

```text
importCharacterPackage(characterId, candidatePackage, confirmation)
```

When applied, it replaces:

```text
character physics state
parameter adjustment history
```

The operation uses the existing import plan and does not bypass risk checks.

## Repository Support

Updated:

```text
src/db/repositories/parameterAdjustmentHistoryRepository.ts
```

Added:

```text
replace(characterId, entries)
```

Imported history entries are retargeted to the destination character id.

This matters because cross-character import is allowed as medium risk with explicit confirmation, but the destination repository must remain internally consistent.

## API

Added:

```text
POST /api/characters/[characterId]/import/apply
```

Request:

```json
{
  "package": "...CharacterPhysicsExportResponse...",
  "confirmation": "replace:lin-fan-dashboard"
}
```

Response:

```text
trace
state
mutatesState: true
```

Status behavior:

```text
200 -> applied
409 -> blocked
```

The validate endpoint remains read-only:

```text
POST /api/characters/[characterId]/import/validate
mutatesState: false
```

## Architecture Meaning

CharacterOS now has a frontend-free manual data control path.

The human can still participate through API-level import/export and adjustment operations, but every mutation remains explicit, auditable and guarded.

This keeps the project aligned with the current direction:

```text
core first
API second
frontend later
visualization last
```
