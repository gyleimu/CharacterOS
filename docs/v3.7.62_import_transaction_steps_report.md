# CharacterOS V3.7.62 Import Transaction Steps Report

## 目标

本阶段继续打磨角色导入应用链路。

V3.7.61 已经让 `/import/apply` 返回 trace 与 Import Transition History 直接关联。V3.7.62 在 trace 内加入显式事务步骤，让一次导入尝试不只是“成功/失败”，而是能看到每个关键阶段的执行顺序和状态。

## 新增字段

`CharacterImportApplyTrace` 新增：

```text
transactionSteps
```

每个 step 包含：

```text
name
status
message
details
```

`status` 当前支持：

```text
completed
blocked
failed
```

## 成功导入步骤

成功应用导入时，步骤顺序为：

```text
authorization_checked
before_state_integrity_inspected
state_deserialized
after_state_integrity_inspected
state_replaced
adjustment_history_replaced
history_recorded
```

这些步骤全部为 `completed`。

## Blocked 导入步骤

缺少确认短语或导入计划不是 ready 时，步骤为：

```text
authorization_checked
history_recorded
```

其中：

```text
authorization_checked = blocked
history_recorded = completed
```

这样 blocked 请求也会被审计，但不会替换角色状态。

## 失败导入步骤

如果授权通过但应用过程中失败，系统会追加：

```text
apply_failed
history_recorded
```

当前失败会返回 blocked trace，并记录失败原因。未来接数据库事务后，可以在这里继续扩展 rollback / compensation steps。

## 为什么重要

导入角色包是高风险操作，会改变：

- character state
- memory galaxy
- impact clusters
- belief states
- adjustment history

因此它不能只是一个黑箱替换动作。V3.7.62 让导入应用具备最小事务骨架：

```text
authorize
inspect
deserialize
replace
record
```

这为后续 Prisma / SQLite / PostgreSQL 事务化持久化打基础，同时仍然保持当前实现轻量、可测、API-only。
