### Writes（4 个 state 字段）

| 写入 | 来源 | 覆盖 Phase 3? |
|------|------|:---:|
| `state.homeostasisState` | `homeostasis.after` | 否（Phase 3 不写此字段） |
| `state.metaState` | `homeostasis.regulatedMetaState` | **是** — 覆盖 Phase 3 的 `metaState.after` |
| `state.boundary` | `homeostasis.regulatedBoundary` | **是** — 覆盖 Phase 3 的 boundary recovery 结果 |
| `state.rewardState` | `homeostasis.regulatedRewardState` | **是** — 覆盖 Phase 3 的 reward recovery 结果 |

### Overwrite Semantics

```text
Phase 3 (decay_and_recovery):
  state.metaState = metaDriftResult.trace.after   ← 粗调：写 metaState
  state.boundary = recoverBoundary(...)            ← 粗调：写 boundary（stress 恢复）
  state.rewardState = recoverRewardBaseline(...)   ← 粗调：写 reward（dopamine 恢复）

Phase 4 (homeostasis):
  state.metaState = homeostasis.regulatedMetaState     ← 精调：覆盖 metaState
  state.boundary = homeostasis.regulatedBoundary        ← 精调：覆盖 boundary
  state.rewardState = homeostasis.regulatedRewardState  ← 精调：覆盖 reward
  state.homeostasisState = homeostasis.after            ← 新写：homeostasisState

这是有意的设计：Phase 3 做粗调（朝 baseline 恢复），Phase 4 做精调（homeostatic regulation）。
Phase 4 的 regulated 值基于 Phase 3 的输出进行计算，然后覆盖它们。
```

### 是否需要 D10

**需要。Homeostasis 有覆盖语义，前 3 次 delegation 没有。**

```text
D10: Overwrite Semantics Equivalence

Homeostasis adapter delegation 完成后，必须验证：
- state.metaState 被 regulatedMetaState 覆盖（≠ Phase 3 写入的值）
- state.boundary 被 regulatedBoundary 覆盖（≠ Phase 3 写入的值）
- state.rewardState 被 regulatedRewardState 覆盖（≠ Phase 3 写入的值）
- state.homeostasisState 被 homeostasis.after 写入（Phase 3 不写此字段）

具体测试方法：
- 在 Phase 3 之后、Phase 4 之前 snapshot meta/boundary/reward
- 在 Phase 4 之后再次 snapshot
- 确认 Phase 4 之后的 meta ≠ Phase 3 之后的 meta
- 确认 Phase 4 之后的 boundary ≠ Phase 3 之后的 boundary
- 确认 Phase 4 之后的 reward ≠ Phase 3 之后的 reward
- 如果不等于，说明覆盖发生了（覆盖 ≠ 相同值跳过）

测试应该同时验证：
- 覆盖的值 == adapter 返回的 regulated{...} 值
- 而不是某种中间值

D10 不替代 D1-D9。它是 D3 的补充——专门针对覆盖语义。
```

## 3. Homeostasis Adapter 风险

### R1: Multi-State Write Risk — MEDIUM

```text
Homeostasis 写 4 个 state 字段，是所有 delegated phase 中最多的。
Boredom 写 1 个字段，Belief 写 1 个字段，Meta Drift 写 1 个字段。

如果 adapter 返回了正确的 regulated 值但调用方没有正确写回全部 4 个字段，
会导致部分覆盖——例如 metaState 被覆盖了但 boundary 没有。

缓解：在 V3 tick 中保持 4 个写回语句（与 inline 代码相同的位置）。
```

### R2: Overwrite Phase 3 Result Risk — HIGH

```text
如果 adapter 在 Phase 3 之前执行（phase order error），
它会读取 Phase 2 的 metaState（而不是 Phase 3 recovery 后的值），
计算出的 regulated 值将不正确。

如果 adapter 的 regulated 值没有被正确写回，
Phase 3 的粗调值将成为最终值——缺少精调。

缓解：D4（phase order）确保 adapter 在 Phase 3 之后执行。
      D10 确保覆盖确实发生了。
```

### R3: Trace Mismatch Risk — LOW

```text
与之前 3 次 delegation 相同。adapter 返回的 phase 对象应该与 inline phases.push 参数完全一致。
```

### R4: Before/After Reference Mutation Risk — LOW

```text
applyHomeostasis 内部已经做了 { ...params.homeostasis } 浅拷贝作为 before。
返回的 after/regulated 是新对象。无引用共享风险。
```

### R5: Reasons Order Risk — LOW

```text
buildReasons 返回的 reasons 数组顺序是确定的（按 if 分支顺序）。
adapter 透传 reasons 即可。
```

### R6: Homeostasis First-Class Semantics Risk — MEDIUM

```text
Homeostasis 是 V4 的第一公民原则。
Delegation 时如果丢失了 "homeostasis 是调节层" 的语义，
它可能会被误认为是普通的 4-field writer。

缓解：在 adapter 文档和注释中明确说明覆盖语义。
      在 adapter 中保留 "homeostasis" 作为 processId。
```

### D10 Overwrite Test 详细设计

```typescript
it("Phase 4 regulated values overwrite Phase 3 recovery values", () => {
  const state = stressedState(); // boundary stress high, needs recovery
  
  // Snapshot after Phase 3 (simulated by running a tick that only does recovery)
  // Actually, we can't easily split the tick. Alternative approach:
  
  // 1. Run full V3 tick
  const trace = runContinuousTick(state, { daysElapsed: 7 });
  
  // 2. Verify Phase 3 (decay_and_recovery) changed meta/boundary/reward
  const phase3 = trace.phases[2]!;
  expect(phase3.name).toBe("decay_and_recovery");
  expect(phase3.changedStates).toContain("boundary");
  
  // 3. Verify Phase 4 (homeostasis) also changed the SAME fields
  const phase4 = trace.phases[3]!;
  expect(phase4.name).toBe("homeostasis");
  expect(phase4.changedStates).toContain("boundary");
  
  // 4. The key invariant: Phase 4's trace.homeostasis.regulatedBoundary
  //    should NOT be identical to Phase 3's recovered boundary.
  //    (If they were identical, homeostasis did nothing — overwrite semantics lost.)
  //    They SHOULD be different because Phase 4 does additional regulation.
  const regulatedBoundary = trace.homeostasis.regulatedBoundary;
  // compare with state.boundary after tick — should equal regulatedBoundary
  expect(state.boundary.stressLoad).toBe(regulatedBoundary.stressLoad);
  
  // 5. Phase 4 also writes homeostasisState (not written by Phase 3)
  expect(phase4.changedStates).toContain("homeostasisState");
});
```
