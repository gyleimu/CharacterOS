# CharacterOS V10.31 — Minimal Product Surface Design Charter

**Date:** 2026-06-28
**Version:** V10.31
**Status:** Design Charter — No Code
**Scope:** Define the minimal observable/actionable surface of CharacterOS for a single-character experience

---

## 1. Philosophy

CharacterOS V10 已经完成单角色 Continuous Life 闭环。内核已经可以：
- 接受事件 → 记忆 → 信念 → 需求 → 欲望 → 决策
- 推进连续生命：疲劳、睡眠、梦境、无聊、随机思维、自主动作候选
- 运行纵向模拟：720 步上限，确定性种子链，紧凑摘要
- 分化决策管道：图式、需求、欲望、策略、行动面
- Commit 闭环：preview → apply → rollback，指纹验证，审计追踪

但所有这些能力目前只能通过 25 条原始 API 调用。没有"产品表面"——没有一个人可以直接"看"和"感受"角色的方式。

"诗云式产品形态"的目标是：**给这个内核一层最小、最薄的观察表面。** 不是 Dashboard，不是 UI 框架，不是前端应用。是一个人可以打开浏览器、命令行、或任何 HTTP 客户端，用极少几个动作就能"看见"和"与角色互动"的 API 表面。

### What "诗云式" (Poetry-Cloud) means here

- **诗 (Poetry)**: 角色的输出应该像诗一样凝练——不是 raw JSON dump，而是精选的、有叙事感的文本片段
- **云 (Cloud)**: 所有状态在云端，用户不需要安装、不需要保存、不需要理解内核。打开即看。

### Core design principles

1. **Read surfaces dominate**. 80% of the product surface is observation. 20% is action.
2. **One character at a time**. No cross-character views. No relationship panels.
3. **Text-first, not visual**. Every surface can render as plain text, JSON, or minimal HTML. No canvas, no WebGL, no heavy CSS.
4. **Dry-run is default**. Any action that could mutate state requires explicit confirmation. Observation never mutates.
5. **No autonomous loop**. The character lives only when the user asks it to live.
6. **No background processes**. Every request is synchronous. No timers, no queues, no cron.
7. **Reuse existing APIs**. New endpoints only when existing ones can't serve the product surface.

## 2. User Actions — What Can You Do?

### Observation (Read — No Mutation, No Auth)

| # | Action | Description | API Reuse |
|---|--------|-------------|-----------|
| O1 | **看角色今日状态** | 当前心情、能量、疲劳、睡眠阶段、无聊度、信任/恐惧/控制力 | New: `GET /product/[characterId]/today` |

| O4 | **看角色生命变化** | 运行一段纵向模拟，只看摘要变化（紧凑摘要、人格漂移趋势） | Reuse: `POST /life/simulate` (dry-run) |

| # | Action | Description | API Reuse |
|---|--------|-------------|-----------|

## 3. Product API Surface Design

### 3.1 Character Today Surface — `GET /product/[characterId]/today`

**Purpose:** One-shot read of everything a user needs to "see" the character right now.

**Reuses:**
- `GET /physics` (state)
- `GET /decision` (current decision)

**Response shape (design):**

```typescript
interface CharacterTodayResponse {
  characterId: string;
  characterName: string;
  generatedAt: string; // ISO timestamp

  // Mood & energy
  mood: {
    energy: number;          // [0,1]
    fatigue: number;         // [0,1]
    sleepPhase: string;      // "awake" | "light_sleep" | "deep_sleep"
    sleepQuality: number;    // [0,1]
    boredom: number;         // [0,1]
    restlessness: number;    // [0,1]
    summary: string;         // "林凡今天精力充沛，睡得很好，有些无聊但不算焦躁。"
  };

  // Personality snapshot
  personality: {
    trust: number;
    fear: number;
    control: number;
    openness: number;
    conscientiousness: number;
    boundaryStress: number;
    boundaryIntegrity: number;
    resilience: number;
    selfControl: number;
  };

  // Current thoughts
  recentThoughts: {
    lastRandomThought?: string;     // ≤ 200 chars
    lastDreamFragment?: string;     // ≤ 200 chars
    lastInspirationSeed?: string;   // ≤ 200 chars
    topSelfActionTendency?: string; // "倾向于检查手机消息"
  };

  // Current decision
  currentDecision?: {
    action: string;                // "压住情绪，先追问原因。"
    strategy: string;              // "安全寻求"
    confidence: number;            // [0,1]
    coreReason: string;            // "因为过去的背叛经历..."
    activatedSchemas: string[];    // ["背叛图式", "安全寻求"]
    topNeeds: string[];            // ["安全感", "可控感"]
  };

  // Memory landscape
  memoryLandscape: {
    totalMemories: number;
    totalBeliefs: number;
    dominantClusterCategory?: string; // "abandonment"
    dominantBelief?: string;          // ≤ 80 chars
  };

  // Life stats
  lifeStats: {
    totalEventsProcessed: number;
    totalTicksApplied: number;
    totalCommitsApplied: number;
    totalCommitsRolledBack: number;
  };

  warnings: string[];
}
```

### 3.2 Inner Log Surface — `GET /product/[characterId]/inner-log`

**Purpose:** Time-ordered feed of the character's internal experiences: dreams, random thoughts, inspirations, self-action tendencies.

**Reuses:**
- `POST /life/simulate` (dry-run to generate recent inner events)
- Life tick runner (internal subsystems)

**Response shape (design):**

```typescript
interface InnerLogResponse {
  characterId: string;
  characterName: string;
  generatedAt: string;
  simulatedHoursCovered: number; // e.g., 72 (last 3 days)

  entries: InnerLogEntry[];
}

interface InnerLogEntry {
  timestamp: string;        // ISO, simulated time
  elapsedHour: number;      // hours into the simulation
  type: "dream" | "random_thought" | "inspiration" | "self_action_tendency" | "mood_shift";
  content: string;          // ≤ 200 chars
  tone?: string;            // "calm" | "anxious" | "hopeful" | "neutral"
  intensity?: number;       // [0,1]
}
```

### 3.3 Character Life Change Surface — `POST /product/[characterId]/life-preview`

**Purpose:** Run a longitudinal simulation and return a human-readable summary of how the character changed.

**Reuses:**
- `POST /life/simulate` (with `includeDecision: true, includeExplanation: true`)

**Response shape (design):**

```typescript
interface LifePreviewResponse {
  characterId: string;
  characterName: string;
  simulatedHours: number;
  stepCount: number;

  before: CompactStateSummary;
  after: CompactStateSummary;

  personalityDrift: {
    trustDelta: number;
    fearDelta: number;
    controlDelta: number;
    summary: string; // "信任轻微下降(-0.02)，恐惧轻微上升(+0.03)，控制力基本稳定。"
  };

  sleepPattern: {
    awakeSteps: number;
    lightSleepSteps: number;
    deepSleepSteps: number;
  };

  dominantStrategies: Array<{
    strategy: string;
    count: number;
  }>;

  notableMoments: Array<{
    hour: number;
    description: string; // "第16小时: 在浅睡中做了一个关于过去的焦虑梦。"
  }>;

  decisionEvolution: string; // "角色从'安全寻求/条件性行动'逐步过渡到'成长修复/条件性行动'。"

  warnings: string[];
}
```

### 3.4 Minimal Interaction Surface

**Send Event** — Reuses `POST /physics/parse` + `POST /physics` directly. No new endpoint needed. The product surface just wraps them in a single call.

**Advance Time** — Reuses `POST /physics/tick` directly.

**Commit Dreams** — Reuses `POST /life/simulate/commit/preview` + `POST /life/simulate/commit/apply` workflow.

## 5. Read vs Write Boundaries

### Read Surface (No Auth, No Mutation)
- `GET /product/[characterId]/today` — aggregate state snapshot
- `GET /product/[characterId]/inner-log` — internal experience feed
- `POST /product/[characterId]/life-preview` — dry-run simulation summary (POST because it accepts parameters)
- `GET /physics` — raw state (existing, strip sensitive fields for product use)
- `GET /decision` — current decision (existing)

### Write Surface (Requires CHARACTEROS_API_KEY when configured)
- `POST /physics/parse` → `POST /physics` — send event
- `POST /physics/tick` — advance time
- `POST /life/simulate` (with commitPolicy) — dry-run with internal commit tracking
- `POST /life/simulate/commit/preview` → `POST /life/simulate/commit/apply` — commit workflow

### Write Surface (Requires Explicit Confirmation)
- `POST /life/simulate/commit/apply` — requires `confirmation: "APPLY_LONGITUDINAL_COMMIT"`
- `POST /life/simulate/commit/rollback` — requires `confirmation: "ROLLBACK_LONGITUDINAL_COMMIT"`
- `POST /editor/apply` — requires auth
- `POST /import/apply` — requires auth + confirmation

## 6. How to Display Character Content

### Text rendering principles

Each product surface should return content that can be rendered in **three tiers**:

**Tier 1 — JSON (machine-readable)**: Full structured response. The default for API consumers.

**Tier 2 — Markdown (human-readable)**: A `narrative` field in each response that translates the structured data into natural Chinese prose. Example:
> 林凡现在处于浅睡阶段。他今天精力中等(0.65)，疲劳度较高(0.42)。最近做了一个关于被抛弃的焦虑梦。他的信任感偏低(0.31)，恐惧感偏高(0.58)。当前最可能的行动倾向是"检查手机消息"。

**Tier 3 — Plain text (terminal-friendly)**: A `summary` field with ≤ 500 chars, suitable for CLI output or SMS-length delivery.

### What NOT to display

- Raw `CharacterPhysicsState` (coordinate, clusters, particles, proceduralRoutines)
- Internal trace IDs, simulation IDs, audit IDs
- File paths, repository internals
- LLM configuration, API keys
- Benchmark results
- Import/export package JSON

## 7. Explicitly Forbidden (V10.31 and near-term)

| Forbidden | Reason |
|-----------|--------|
| Multi-character views | Single-character philosophy |
| Relationship panels | Not implemented. V20+ |
| World simulation | Not in scope |
| Autonomous background scheduler | User explicitly deferred |
| Mobile native app | Not in scope |
| 3D visualization / WebGL | V3.7.48 removed frontend |
| Complex React SPA | Keep product surface API-only |
| WebSocket / SSE / streaming | Synchronous request/response only |
| LLM integration in product surface | LLM is internal engine, not product feature |
| Character creation UI | Existing `resetCharacter` + `import/apply` sufficient |
| Event editor UI | `POST /physics/parse` + `POST /physics` sufficient |
| Dashboard revival | Explicitly removed. Not restoring |
| Push notifications / email | No background processes |
| Authentication/authorization beyond API key | Current CHARACTEROS_API_KEY is sufficient |

## 8. Phased Roadmap

### V10.31 ✅ Design Charter (this document)

### V10.32 Product DTO Types
- Define `CharacterTodayResponse`, `InnerLogResponse`, `LifePreviewResponse` in `src/appContracts/productSurface.ts`
- Define shared narrative/summary helper interfaces
- Type-only. No route implementations.

### V10.33 Character Daily Surface API
- Implement `GET /product/[characterId]/today`
- Thin aggregation over `getState()` + `deriveCharacterState()` + decision
- Add narrative generation (rule-based, no LLM)
- Tests: returns compact data, no state leak, READ-ONLY

### V10.34 Inner Log API
- Implement `GET /product/[characterId]/inner-log`
- Runs a short dry-run longitudinal simulation (e.g., 72 hours) to generate inner events
- Returns time-ordered feed of dreams, thoughts, inspirations
- Tests: deterministic, no state mutation, entries bounded in size

### V10.35 Life Preview API
- Implement `POST /product/[characterId]/life-preview`
- Thin wrapper over `POST /life/simulate` with narrative formatting
- Returns personality drift summary, sleep patterns, notable moments
- Tests: dry-run only, narrative bounded, decision evolution coherent

### V10.36 Product Surface QA
- End-to-end test: create character → get today → run life preview → check inner log
- Verify all 3 product endpoints respect read/write boundaries
- Verify no state leak in any product response
- Verify narrative/summary fields populated
- Documentation sync
