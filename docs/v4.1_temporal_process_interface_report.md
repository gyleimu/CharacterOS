## 1. 新增文件

### `src/core/temporal/temporalProcess.ts`

TemporalProcess 最小类型系统。

**导出：**
- `TemporalProcessId` — string 别名
- `TemporalProcessLabel` — string 别名
- `TemporalProcessCategory` — `"mutation" | "trace" | "context"`
- `TemporalProcess` — 核心元数据接口（id, label, phase, category, reads, writes, description）
- `TemporalProcessInput` — 未来 V4.2+ 的输入类型（占位）
- `TemporalProcessResult` — 未来 V4.2+ 的输出类型（占位）

**设计决策：**
- `phase` 是 1-based number，匹配 V3 的 17 个 phase
- `category` 区分 mutation（写状态）/ trace（只读艺术）/ context（共享上下文）
- 接口层面不包含 `run()` 方法 — V4.1 元数据 only
- `TemporalProcessInput` / `TemporalProcessResult` 已声明但不使用

### `src/core/temporal/v3TickPhaseMetadata.ts`

V3 `runContinuousTick` 的 17 个 phase 的声明式元数据。

**导出：**
- `V3_TICK_PHASES` — `readonly TemporalProcess[]`（17 项，`Object.freeze`）
- `V3_TICK_PHASE_COUNT` — `17`（literal type）
- `getV3TickPhase(id)` — 按 id 查找
- `v3TickPhaseIds()` — 按执行顺序返回所有 phase id

**Phase 顺序：**
与 `continuousTick.ts` 实际执行顺序严格一致：
```
 1. snapshot
 2. meta_drift
 3. decay_and_recovery
 4. homeostasis
 5. recovery_trace
 6. parameter_network
 7. baseline_drift
 8. parameter_accumulation
 9. parameter_adjustment_draft
10. parameter_adjustment_preview
11. parameter_adjustment_audit
12. parameter_adjustment_patch
13. parameter_adjustment_snapshot
14. boredom
15. belief_evolution
16. attention_and_reflection
17. time_perception
```

**分类：**
- mutation: 4 phases（meta_drift, decay_and_recovery, homeostasis, boredom, belief_evolution）
- trace: 12 phases（snapshot, recovery_trace, parameter_network, baseline_drift, parameter_accumulation, parameter_adjustment_draft/preview/audit/patch/snapshot, attention_and_reflection）
- context: 1 phase（time_perception）

注意：Phase 3 (decay_and_recovery) 和 Phase 4 (homeostasis) 都写 boundary/rewardState/metaState。Phase 4 有意覆盖 Phase 3 的输出 — 这是 V3 的 design intent，metadata 正确反映。

## 3. V3 ContinuousTickPhaseName 与 Metadata 的对比

V3 的 `ContinuousTickPhaseName` 是一个 TypeScript union type。其成员**声明顺序**与代码**执行顺序**不完全一致（`parameter_network` 在 type 中位于 `belief_evolution` 之后，但在代码中位于 recovery_trace 之后即 Phase 6）。

**V4.1 的 metadata 遵循代码执行顺序，而非 type 声明顺序。**

| Phase | 代码执行 | ContinuousTickPhaseName type 中的位置 | Metadata |
|-------|---------|--------------------------------------|----------|
| 6 | parameter_network | 15（错位） | 6 ✓ |
| 7 | baseline_drift | 6（错位） | 7 ✓ |
| 15 | belief_evolution | 14（错位） | 15 ✓ |

这是 V3 的一个 cosmetic issue — union type 成员顺序不影响运行时行为。V4.1 metadata 选择以代码为准。
