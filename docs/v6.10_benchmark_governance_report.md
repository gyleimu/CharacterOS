## 3. Benchmark 哲学

```text
1. Benchmark 是 garden，不是 wall。
   它帮助我们发现 regression，但不阻止我们做正确的改变。
   一个 failing benchmark 可能是 fixture 假设过时了，而不是系统坏了。

2. 测方向，不测精确人格值。
   "trust should decrease after abandonment" — 方向断言。
   "trust after 14 days = 0.347" — 脆弱断言，不应使用。
   方向断言在参数调整和角色差异下仍然成立。

3. Benchmark 不评判文学质量。
   "这篇 narrative 好不好" 不是 benchmark 问题。
   "decision.mostLikelyAction 是否非空" 才是。

4. Benchmark 不替代人工审查。
   Benchmark 是自动化 canary — 快速发现方向性回归。
   但它不能代替认真阅读 trace 和 decision narrative。

5. Benchmark 不等于真实心理学真理。
   CharacterOS 模拟的是 emergent psychological dynamics，
   不是 gold-standard 心理测量。Benchmark 不等于 validity study。
```

## 4. Governance Rules

### 规则 1: 不因单个 benchmark 失败就硬改人格

```text
如果 memory_decay benchmark 失败了：
  - 先检查：参数是否被无意修改？state mutation 顺序是否改变？
  - 再判断：fixture 的 expected direction 是否仍然反映系统设计的意图？
  - 最后才考虑：是否需要调整人格参数？

行为：
  - failing benchmark → investigate, explain, document
  - 不自动 revert
  - 不为了 "通过 benchmark" 而把参数往 fixture 方向调
```

### 规则 2: 不为了 pass benchmark 牺牲心理一致性

```text
如果 adjusting metaState.selfControl 让 belief evolution benchmark pass，
但同时让行为决策变得 "机械"（失去 fear/trust 的 nuance）：
  - 这不是 acceptable trade-off
  - Parameter "calibration to benchmark" is anti-governance
  - benchmark 服务于心理一致性，不是反过来
```

### 规则 3: 不把 LLM 输出作为唯一判据

```text
当前 V6 benchmark 不使用 LLM。
如果未来使用 LLM 辅助评估（V7+）：
  - LLM 输出必须配 numeric 指标
  - LLM fail alone ≠ benchmark fail
  - LLM 生成的解释 ≠ ground truth
```

### 规则 4: 新 fixture 必须说明 expected direction 的心理理由

```text
每个 expected direction 的 reason 字段必须解释：
  - 为什么这个方向是合理的（psychological justification）
  - 为什么使用这个 tolerance mode
  - 在什么情况下这个 assertion 应该被 "视为过时"

例如：
  "trust should decrease after abandonment event because
   abandonment sensitivity amplifies impact on trust dimension.
   This is directional: magnitude varies by character blueprint.
   If the trust mechanism is redesigned, this fixture may need update."
```

### 规则 5: fixture 必须优先使用 directional / bounded / invariant

```text
优先顺序：
  1. directional（方向，不求 magnitude）
  2. bounded（安全范围，不求精确值）
  3. monotonic（连续方向，不求线性度）
  4. invariant（结构性不变量）
  5. exact（仅限结构性不变量，如 D10 overwrite）

永远不要：
  - exact numeric target（"trust = 0.42"）
  - LLM output match（"narrative should contain '...'"）
  - cross-character equality（"trust(A) == trust(B)"）
```

### 规则 6: benchmark 失败要先判断 root cause

```text
Benchmark fail → 三种 root cause：

  A. 系统 bug
     - state mutation 顺序改变
     - 函数签名变化导致参数丢失
     - 浮点精度退化
     → fix the bug, benchmark should re-pass

  B. fixture 假设错误
     - 系统行为改变了，但改变是合理的
     - fixture 的 expected direction 不再反映当前设计
     → update the fixture, document why

  C. 模型理论需要修正
     - 长期观察发现原假设不对
     - 例如 "belief 确实不会在 21 days 内增加"
     → update documentation, adjust fixture bound
```

## 5. Risk Register

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| R1 | Overfitting benchmark | high | 只测方向，不测精确值。fixture 应通用化，不绑定特定参数值。 |
| R2 | False confidence | medium | benchmark pass ≠ system is "correct"。它只说 "方向没回归"。 |
| R3 | Culturally narrow fixtures | medium | 所有中文事件 + 林帆角色。V7 可加多角色 + 英文 fixture。 |
| R4 | Fragile text assertion | medium | behavior_decision fixture 只测 bounded numeric + non-empty text。不做 string exact match。 |
| R5 | Slow variable misunderstanding | medium | belief_evolution 在 21d 内只动 0.001。如果 fixture 期望大幅变化，那是 fixture 错了。 |
| R6 | Benchmark-driven personality flattening | high | 如果为了 pass benchmark 而统一调整参数，不同角色的 niche 行为会消失。Cross-character benchmark 应允许不同的 magnitude，只检查 direction 一致。 |
| R7 | Graph edge weights over-trusting benchmark | medium | V7 graph 不应把 benchmark pass/fail 直接当做 causality weight。Benchmark 提供 "direction 在这个阶段是否保持一致"，不是 "这条 edge 的 effect size = X"。 |

## 6. Adding New Benchmark — Policy

```text
新增 benchmark 的标准流程：

1. Write Scenario
   - 描述输入（initial state, events, ticks）
   - 描述观察对象（哪个 state field / trace / subprocess）
   - 不假设精确输出值

2. Identify Metric Source
   - 从 V6 metric sources 列表中选择
   - 如果需要新 source，先在 benchmarkTypes.ts 中注册

3. Choose Tolerance
   - directional（首选） → bounded → monotonic → invariant → exact
   - 绝大多数 fixture 应使用 directional

4. Justify Expected Direction
   - 为什么 trust 应该 decrease？
   - 为什么 belief strength 应该 ≥ 0.1？
   - 这个 expected direction 在什么情况下会失效？

5. Run Baseline
   - 在 clean state 上运行 benchmark
   - 记录 baseline result（用于未来对比）

6. Add Regression Test
   - tests/core/benchmark/ 下新增或更新测试
   - 测试应验证 benchmark pass 并且 metrics 有意义

7. Document Limitations
   - fixture notes 中声明 known limitations
   - 例如："belief strength 在 21d 内几乎不变，使用 bounded_below 而非 increase"
```

## 7. API Governance

```text
GET /api/benchmark/report

  readonly     — 不修改任何 state
  no LLM       — 不调用任何 LLM
  no mutation  — 不写数据库
  no API key   — 公开可访问（development health check）
  idempotent   — 相同输入 → 相同输出
  suitable for — Codex health check, Claude project health check, CI pipeline

Response shape:
  generatedAt    — ISO timestamp
  summary        — totalCases, passed/failed/skipped/errored counts
  results        — per-fixture verdict, assertion counts, duration, warnings
  warnings       — D10 notes, slow variable notes, text assertion caveats
  reasons        — standard benchmark report reasons

Do NOT add to this API:
  - POST / mutation endpoints
  - LLM evaluation
  - Cross-character comparison (V7)
  - Graph visualization (V8)
  - History storage (V7+)
```

    - directional consistency: 在多少次 benchmark run 中 direction 保持一致
