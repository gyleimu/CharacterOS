## 1. 新增文件

### `src/core/temporal/unifiedTickTrace.ts`（230 行）

**类型定义：**

| 类型 | 用途 |
|------|------|
| `UnifiedTickProcessTrace` | 单个 process 的 trace（14 个字段） |
| `UnifiedTickMutationSummary` | 跨 process 的 mutation 统计 |
| `UnifiedTickCoverageSummary` | metadata ↔ adapter 覆盖率 |
| `UnifiedTickTrace` | 顶层 trace（version, source, processTraces, 两个 summary, warnings, reasons） |

**Builder：**

```typescript
buildUnifiedTickTrace({ v3Trace, adapters? }) → UnifiedTickTrace
```

输入一个 V3 `ContinuousTickTrace`，输出 V4 结构化 trace。零 side effect，零 state mutation。

**映射逻辑：**
- 遍历 `v3Trace.phases`（V3 的 17 个 `ContinuousTickPhaseTrace`）
- 对每个 phase：查找 metadata（label, reads, writes）+ adapter（mutationPolicy, status）
- 构建 `UnifiedTickProcessTrace`，其中 `observedOnly = adapterStatus !== "delegated" && adapterStatus !== "native"`
- 汇总 mutation summary + coverage summary

## 3. 设计决策

### 为什么是观察层

```text
UnifiedTickTrace 不改变任何行为。
它是对 V3 ContinuousTickTrace 的重新解释。

V3 trace 已经包含所有信息。
UnifiedTickTrace 只是把这些信息重新组织成 V4 的结构。
```

### 为什么 phase 顺序来自 metadata 而非 V3 trace

V3 trace 的 `phases` 数组已经按执行顺序排列。Builder 直接遍历该数组，因此 phase 顺序自然与 V3 一致。Metadata 用于补充 label/reads/writes，不改变顺序。

### 为什么 observedOnly = true 对所有 17 个 process

V4.2 的所有 adapter 都是 shell（`implementationStatus = "adapter_shell"`）。没有 adapter 升级到 "delegated" 或 "native"。因此所有 process 都是 observed-only。

### 为什么 warnings 包含 mismatch 信息

如果 metadata 和 adapter 不一致（缺少 adapter、adapter 无对应 metadata），builder 会生成 warnings。这不是错误 — 只是诊断信息。
