# CharacterOS V3.7.60 Import Transition History Report

## 目标

本阶段继续强化无前端状态下的高风险操作审计能力。

V3.7.59 已经让单次成功导入具备 `transitionId` 和 `transitionSummary`。V3.7.60 将这些一次性 trace 沉淀为角色级导入尝试历史：每次调用 `/import/apply`，无论成功还是被阻止，都会记录一条 Import Transition History Entry。

## 记录范围

会记录：

- 缺少确认短语的 blocked apply attempt。
- 导入计划不是 ready 的 blocked apply attempt。
- 成功替换角色状态的 applied import。
- 反序列化失败产生的 blocked attempt。

不会记录：

- `/import/validate` 纯验证请求。
- 没有进入 apply 入口的普通读取请求。

## Summary

`CharacterImportTransitionHistorySummary` 当前包含：

```text
totalEntries
appliedCount
blockedCount
latestStatus
latestTransitionId
latestCreatedAt
reasons
```

它用于在没有前端页面时快速判断某个角色最近是否发生过导入替换，以及是否有被阻止的导入尝试。

## 生命周期

`CharacterPhysicsService.resetCharacter()` 现在会同时清理：

```text
character state
parameter adjustment history
import transition history
```

这样 reset 仍然代表“重置这个角色的本地运行态”。

成功导入不会清空目标角色已有的 import history，因为导入历史描述的是目标角色在当前系统中的操作审计，而不是被导入包内部的一部分。

## 持久化

Next 单例现在使用轻量 JSON 文件仓库：

```text
data/import_transition_history.json
```

它和现有状态文件一样属于本地运行数据，未来接 Prisma / SQLite 时只需要替换 repository 实现。
