# V12.0: Character Agent SDK — Design Charter

**Date:** 2026-07-02
**Version:** V12.0 Design Charter
**Previous:** V11.13 (Post-RC Hardening)
**Status:** Design Phase — No Implementation

---

## 1. V12 Definition

### What V12 Is

**Character Agent SDK** is an embeddable interface layer. It lets external applications connect CharacterOS Core to chat, novel writing, journaling, plugins, and tool flows — without modifying the core personality engine.

### What V12 Is NOT

| Not this | Reason |
|----------|--------|
| ❌ Chatbot product | SDK is the plumbing, not the product |
| ❌ Multi-character | V20 is future work |
| ❌ Autonomous agent | No background loop, no self-directed action |
| ❌ LLM wrapper | LLM is an optional output adapter, not the core |
| ❌ Server deployment | Local-first, single-machine SDK |
| ❌ Medical/psychological tool | Simulation, not clinical |

### Core Thesis

> **V12 should make CharacterOS embeddable, not turn it into a chatbot.**

## 4. Critical Boundaries

| Boundary | Rule |
|----------|------|
| LLM → Core | LLM cannot mutate CharacterPhysicsState directly |
| LLM → State | LLM cannot invent personality claims not in state |
| Reply → Context | Reply must be grounded in CharacterOS context |
| Writeback | Requires policy: never / preview_only / auto_apply_safe / require_confirmation |
| Default policy | `preview_only` (user sees what would happen, decides) |
| Diagnosis | Medical/psychological diagnosis language prohibited |
| Multi-character | Relationship engine prohibited in V12 |

## 5. SDK Modules

### 5.1 AgentSession

Manages one agent session — the lifecycle of a conversation or interaction.

```
AgentSession {
  sessionId, characterId,
  policy: { writeback, safety, llm },
  memoryWriteMode: "session_only" | "persist",
  safetyMode: "strict" | "permissive"
}
```

### 5.2 AgentInputAdapter

Normalizes external input into a standard agent turn.

| Input Type | Adapter |
|-----------|---------|
| Chat message | `adaptChatMessage(text, sender, timestamp)` |
| Journal entry | `adaptJournalEntry(text, date, mood)` |
| Story event | `adaptStoryEvent(narrative, characters, setting)` |
| Local log | `adaptLocalLog(entry, source, metadata)` |
| Plugin input | `adaptPluginInput(payload, pluginId, schema)` |

### 5.3 EventCandidateExtractor

Converts agent input into `EventStudioDraft` candidates.

- **Deterministic first:** rule-based extraction from keywords + structure
- **LLM extraction later:** optional, gated behind configuration
- **Multiple candidates:** one input may produce 0–N event candidates
- **Each candidate:** intensity, tags, category hint, confidence

### 5.4 AgentPolicyGate

Decides what happens to each event candidate.

```
Decision {
  action: "block" | "preview_only" | "apply",
  reason: string,
  realityAuditCheck: PASS | WARN | FAIL,
  safetyCheck: PASS | WARN | FAIL,
  confidenceRequired: number
}
```

**Block rules:**
- Diagnostic/medical language in input → block
- Multi-character relationship claim → block
- Reality Audit FAIL → block (unless override)
- Confidence below threshold → block or preview_only

### 5.5 AgentContextBuilder

Assembles context for reply generation from V11 modules.

```
AgentGroundingBundle {
  stateSurface: CharacterStateSurface,        // V11.4
  explainability: ExplainabilityTimeline,     // V11.5
  timeMachine?: TimeMachineRestoreView,       // V11.7
  recentEvents: EventStudioAuditEntry[],      // V11.3
  personalitySummary: PersonalitySummary,     // V11.4
  safetyNotices: string[]
}
```

**Rule:** No raw `CharacterPhysicsState` in the grounding bundle.

### 5.6 ReplyPlanner

Builds a structured reply plan — NOT an LLM-generated message.

```
AgentReplyPlan {
  facts: string[],              // Grounded facts from state
  emotionalContext: string,     // Current emotional state summary
  uncertainties: string[],      // What the system is unsure about
  suggestedDirections: string[], // Possible reply directions
  safetyConstraints: string[],  // Must-include safety notices
  groundingRefs: EvidenceRef[]  // Links to source data
}
```

**Rule:** ReplyPlanner is deterministic. LLM is a separate optional layer.

### 5.7 AgentOutputAdapter

Formats the reply plan for different output channels.

| Output Type | Adapter |
|------------|---------|
| Chat reply | Structured message with context |
| JSON API | Machine-readable response |
| Story narration | Narrative prose (from reply plan) |
| Plugin response | Plugin-specific schema |

### 5.8 Writeback Policy

Decides whether the conversation itself becomes a character event.

```
WritebackPlan {
  action: "never" | "auto_apply" | "require_confirmation",
  eventCandidate?: EventStudioDraft,
  auditEntry?: EventStudioAuditEntry,
  reason: string
}
```

- `never` — conversation does not affect character
- `auto_apply` — safe events auto-applied (with audit)
- `require_confirmation` — user must explicitly confirm

**Default:** `require_confirmation` (never auto-apply without user consent)

## 7. Safety Model

| Layer | Rule |
|-------|------|
| Input | Block diagnostic/medical language |
| Extraction | Never extract multi-character relationship events |
| Policy | Reality Audit WARN → flag; FAIL → block |
| Context | Never expose raw state to external consumer |
| Reply | Always include simulation-not-diagnosis notice |
| Writeback | Never auto-apply without policy + audit |
| Trace | Every apply produces audit entry + rollback reference |
| LLM | LLM output is suggestion, not ground truth; never mutates core |
| Consent | User must opt in to any writeback policy above preview_only |

## 9. Explicit Non-Goals

| Non-goal | Reason |
|----------|--------|
| ❌ Multi-character | V20 is future work |
| ❌ Relationship engine | Requires multi-character |
| ❌ Autonomous background loop | User or external app drives |
| ❌ Production chat UI | SDK is interface, not product |
| ❌ Real LLM integration | Optional adapter, deferred |
| ❌ User accounts | Single-machine design |
| ❌ Server deployment | Local SDK |
| ❌ Medical/psychological diagnosis | Simulation, not clinical |
| ❌ Unsupervised state mutation | Always policy-gated + audited |

---

## 10. Final Position

> **V12 should make CharacterOS embeddable — a personality engine that any application can connect to. It should not turn CharacterOS into a chatbot.**

The SDK is an interface, not a personality. The agent is a pipeline, not an identity. The reply is grounded, not generated. The writeback is audited, not hidden. The safety is designed in, not added on.
