## Boundary Guarantees (Confirmed)

| Guarantee | How |
|---|---|
| LifeDecisionContext is modifier, not owner | Life signals only adjust strategy weights (×0.5–1.15 range); schemas from persona/state untouched |
| Self-action candidates are signals, not executed actions | `topSelfActionCandidateType`/`topSelfActionCandidateScore` are read-only indicators |
| Life dry-run is read-only | `runLifeTickDryRun` produces projection; no state written |
| Default API behavior unchanged | Without `?includeLifeContext=true`, response identical to V10.13 |
| No infinite loop | `runLifeTickDryRun` called once per request; no daemon/loop |
| No autonomous simulation | Dry-run is explicit, caller-initiated, single-tick |
| No multi-character system | Single `CharacterPhysicsState` per request |
| No self-action execution | Neither `commitLifeTickProjection` nor `applyActionNoise` called in life decision path |

## Known Limitations

| Limitation | Severity | Mitigation |
|---|---|---|
| Life context always starts from default subsystem states | Low | Caller can provide elapsedHours to simulate passage of time |
| No persistent life state across API calls | Low | Future: store life state and replay from last persisted tick |
| Life modifiers use hardcoded thresholds (0.55, 0.45, etc.) | Low | Thresholds are configurable in code; not exposed as API params |
| Random thought signal varies with seed | Very low | Deterministic — same seed = same signal; documented behavior |
