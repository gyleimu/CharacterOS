## 1. tagNormalization 核心链路接入

### 审计发现

V3.9 创建了 `src/core/event/tagNormalization.ts` 模块和 21 个测试，但 **未接入任何核心链路**。`normalizeTags` 在整个 `src/` 目录中零引用。

受影响的系统：

| 系统 | 文件 | tag 匹配方式 | 使用 canonical tags? |
|------|------|-------------|---------------------|
| eventParser | `eventParser.ts` | `categoryKeywords` 硬编码中文 | 仅中文，英文 tag 不匹配 |
| attentionSystem | `attentionSystem.ts` | `tagChannels` 硬编码中文 | 仅中文，英文 tag 不匹配 |
| worldModel | `worldModel.ts` | `tags.has("失联")` 等直接比较 | 仅中文 |
| proceduralMemory | `proceduralMemory.ts` | `matchRatio()` exact match | 仅匹配当前 tag |
| physicsEngine | `physicsEngine.ts` | `classifyCategory()` hardcoded | 仅中文 |

### 已执行修复

**修复策略：在 tag 入口点归一化，而非修改每个消费端。**

这保证所有下游系统自动收到 canonical Chinese tags，无需逐一修改 attention / worldModel / proceduralMemory。

修改文件：
- `src/core/event/eventParser.ts` — `parseExperienceEvent()` 入口调用 `normalizeTags()`
- `src/core/event/llmEventParser.ts` — `normalizeLLMParsedEvent()` 输出归一化 tags
- `src/services/characterPhysicsService.ts` — `processEvent()` 和 `simulateEvents()` service boundary 归一化

**覆盖的路径：**
1. Rule-based parse → tags normalized at `parseExperienceEvent` ✓
2. LLM parse → tags normalized at `normalizeLLMParsedEvent` ✓
3. Direct API POST to `/physics` → tags normalized at `processEvent` service boundary ✓
4. Batch simulation POST to `/physics/simulate` → tags normalized at `simulateEvents` boundary ✓

**中文 tag 保留：**
- 中文 tag 经 `normalizeTags` 后保持不变（canonical form 就是中文）
- 英文 tag 被映射为中文 canonical form
- 展示层始终使用 canonical tags，对中文用户无影响

### 验证

- 现有 eventParser 测试全部通过（使用中文 tag，通过 normalizeTags 不变）
- tagNormalization 测试全部通过（21 tests）
- 无破坏性变更

## 3. Import Partial Failure Tracking 审计

### 审计发现

V3.9 的 import mutation tracking 代码结构正确：

**Pre-mutation snapshot：**
- 在 `state_replaced` 和 `adjustment_history_replaced` 之前捕获 `preMutationState`
- 存入 `trace.stateRollbackSnapshot` 供人工恢复
- 注释明确说明这不是自动 rollback

**Per-step tracking：**
- `state_replaced` 被独立 try/catch 包裹
- 失败时：`mutationOutcome.stateReplaced = false`，返回 blocked trace + snapshot
- `adjustment_history_replaced` 被独立 try/catch 包裹
- 失败时（state 已替换）：`mutationOutcome.description` 明确说明 "partial mutation — manual rollback may be needed"

**TransactionSummary 准确性：**

| 场景 | terminalStatus | stateMutated | adjHistoryMutated | historyRecorded |
|------|---------------|-------------|-------------------|-----------------|
| auth 失败 | blocked | false | false | true |
| state replace 失败 | blocked (failed) | false | false | true |
| state 成功, adjHistory 失败 | blocked (partial) | true | false | true |
| 全部成功 | completed | true | true | true |

**Partial mutation warning：**
`summarizeImportTransactionSteps` 中的 `buildTransactionSummaryReasons` 会在 stateMutated=true 且 adjustmentHistoryMutated=false 时追加 WARNING。

### 已知限制（已明确文档化）

```text
当前仍不是 ACID 事务。

原因：
- 文件 JSON 仓库不支持跨文件原子提交
- 没有 WAL / rollback journal
- state replace 和 adjustmentHistory replace 是两个独立操作

缓解措施：
- pre-mutation snapshot 保留完整旧状态供人工恢复
- mutationOutcome 清晰描述哪些 mutation 成功/失败
- 代码注释和文档明确说明这不是 ACID
```

### 历史记录

- `recordImportTransition` 正确设置 `mutationOutcome.historyRecorded = true`
- History entry 持久化完整的 trace（包含 mutationOutcome）
- API response 返回完整 trace 供调用方判断

## 4. Deep Validation vs Serialization Schema 一致性

### SerializedCharacterPhysicsState 字段验证覆盖

| 字段 | 必填 | 验证深度 | 状态 |
|------|------|---------|------|

| `identity` | 可选 | — | ⚠️ 未验证 |

| `rewardState` | 可选 | — | ⚠️ 未验证 |
| `homeostasisState` | 可选 | — | ⚠️ 未验证 |
| `boredomState` | 可选 | — | ⚠️ 未验证 |

### 评估

- **必填字段核心验证足够**：coordinate, personality, clusters, memories 的深度验证能捕获大多数结构化错误
- **可选字段未验证**：这些字段在 `SerializedCharacterPhysicsState` 中都是 optional，缺失不报错是正确行为。但如果存在但格式错误（如 `rewardState.dopamineLevel` 不是 number），当前不会检测
- **particles/derived/galaxy 仅 shallow**：particles 结构简单（ImpactParticle），derived 和 galaxy 是 computed fields，在 deserialization 时会被重建，shallow check 足够

### 合法性检查

验证不会过严导致旧包无法导入。原因：
- 旧包缺少新字段时，这些字段是 undefined → 验证跳过（因为 optional fields 不验证）
- 旧包有 coordinate 但没有扩展维度（trust/attachment/fear/control）时，这些 key 值是 undefined → 验证只检查"if present and not number"，不会报错
- 旧包 clusters 只有 id/category/mass → 这些字段被验证，旧包应该都有

### 建议

当前验证深度合理。未来如果需要更严格的 validation：
- 加 `CHARACTEROS_EXPORT_VERSION` 版本号
- V1.1 需要所有可选字段也通过验证
- V1.0 的旧包可以继续使用当前宽松规则

### Phase 间冲突分析

**Phase 3 (decay_and_recovery) → Phase 4 (homeostasis)：**
- Phase 3 直接写 `state.boundary`, `state.rewardState`, `state.metaState`
- Phase 4 读这些字段并可能再次修改（`homeostasis.regulatedBoundary`, `regulatedMetaState`, `regulatedRewardState`）
- **关系：Phase 4 覆盖 Phase 3 的 boundary/reward/meta 写入。** 这是有意设计——decay/recovery 先做粗调，homeostasis 再做精调。
- **无冲突**：homeostasis 读取的是 Phase 3 后的最新值，然后在其基础上做 homeostatic regulation

**Phase 14 (boredom) / Phase 15 (belief_evolution)：**
- Phase 14 写 `state.boredomState`
- Phase 15 写 `state.beliefStates`
- **无冲突**：操作不同字段

**Phase 4 (homeostasis) → Phase 5-13 (trace pipeline)：**
- Phase 4 是最后一次直接 state mutation（在 trace pipeline 之前）
- Phase 5-13 只构建 trace artifacts，不写 state
- **正确顺序**：所有 mutation 在 trace building 之前完成

**测试覆盖：**
`tests/core/continuousTick.test.ts` 有 16 个测试，覆盖了主要路径（tick with decay, homeostasis, reward recovery, boredom update, belief evolution, time perception 等）。
