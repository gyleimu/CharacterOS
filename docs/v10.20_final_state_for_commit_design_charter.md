# CharacterOS V10.20 — finalStateForCommit Design Charter

**Date:** 2026-06-28
**Version:** V10.20
**Status:** Design Only, No Core Code
**Scope:** Safe commit design for longitudinal single-character simulation

---

## 1. Executive Summary

V10.20 defines the missing persistence surface between V10.17 longitudinal simulation and V10.18 simulation API.

The system can already simulate a single character across time, optionally run internal commit projection on a working clone, and report what would change. It deliberately does not persist the final cloned state through the API. V10.20 keeps that boundary closed and designs the safe path for opening it later.

**Verdict:** commit persistence remains deferred. The next implementation phase should add types and preview surfaces before any apply route writes state.

---

## 2. Why V10.20 Is Design-Only

Longitudinal simulation commit is more sensitive than ordinary editor patch apply:

- It can convert simulated future traces into durable character history.
- It may append generated memories from dreams, random thoughts, or inspiration.
- It can span many simulated steps.
- It sits close to the boundary between observation and autonomous life.
- It could accidentally make self-action candidates look like executed actions if the surface is sloppy.

Therefore V10.20 does not open persistence. It defines the contract that future versions must obey.

---

## 3. Current State

### Existing Safe Pieces

| Version | Capability | Boundary |
|---|---|---|
| V10.9 | `commitLifeTickProjection` | Applies selected generated artifacts to a clone only |
| V10.17 | `runLongitudinalSimulation` | Dry-run-first harness; optional internal commit on working clone |
| V10.18 | `/life/simulate` API | Read-only by default; commit persistence deferred |
| V10.19 | Governance audit | Fixed 404, request bounds, cap warning |

### Current Gap

`runLongitudinalSimulation()` may internally advance a working clone when `commitPolicy.enabled=true`, but `LongitudinalSimulationResult` exposes only compact summaries and per-step commit results. The API cannot safely persist because it lacks a formal, private, governed `finalStateForCommit` surface.

---

## 4. Non-Goals

V10.20 explicitly does not:

- Implement final state persistence.
- Add a commit API.
- Add benchmark cases.
- Execute self-action candidates.
- Add a background loop.
- Add world simulation.
- Add multi-character simulation.
- Add a graph database or external event scheduler.
- Persist full life traces.

The goal is to design the lock before cutting the key.

---

## 5. Core Principle

```text
Dry-run is observation.
Commit is consent.
finalStateForCommit is a guarded handoff, not an API payload.
```

`finalStateForCommit` should be an internal service-layer handoff object. Public responses may expose commit preview summaries, audit records, warnings, and digests, but not the full `CharacterPhysicsState`.

## 7. Commit Surface

`LongitudinalCommitSurface` should describe what changed without exposing the full state.

```typescript
interface LongitudinalCommitSurface {
  applied: boolean;
  totalSteps: number;
  committedSteps: number;
  generatedMemoryCount: number;
  changes: LongitudinalCommitSurfaceChange[];
  skipped: LongitudinalCommitSurfaceChange[];
  beforeSummary: CompactStateSummary;
  afterSummary: CompactStateSummary;
}

interface LongitudinalCommitSurfaceChange {
  stepIndex: number;
  source: "dream" | "random_thought" | "inspiration_seed";
  path: string;
  generatedId?: string;
  from: unknown;
  to: unknown;
  reason: string;
}
```

### Allowed Initial Sources

| Source | Allowed In First Apply? | Destination | Notes |
|---|---:|---|---|
| Dream fragments | Yes | `state.memories[]` append | Low importance memory seed |
| Random thoughts | Yes | `state.memories[]` append | Low importance memory seed |
| Inspiration seeds | Yes | `state.memories[]` append | Low importance memory seed |
| Energy/fatigue | No | Requires schema migration | No `lifeState` field yet |
| Sleep/wake | No | Requires schema migration | No `lifeState` field yet |
| Boredom expansion | No | Requires schema migration | No `lifeState` field yet |
| Self-action candidates | Never | None | Signal only, not executed |

## 8. Governance Gates

Every future apply must pass all gates:

| Gate | Rule | Failure Behavior |
|---|---|---|
| Auth | Commit apply requires auth | 401/403 |
| Explicit consent | `commitPolicy.enabled=true` required | 422 |
| Idempotency | Caller provides or receives `simulationId`/digest | Duplicate returns existing result or conflict |
| Base state fingerprint | Current state must match preview base | 409 stale state |
| Request digest | Apply request must match preview request | 409 mismatch |
| No self-action execution | `commitSelfActionCandidates` must be false | 422 |
| Generated memory cap | Per-step and total caps enforced | 422 or partial skipped |
| Integrity check | Final state passes structural integrity | 422 |
| Audit write | Audit entry is created with commit | Block if audit cannot be recorded |
| Rollback plan | Rollback plan exists before apply | Block if absent |

Commit must be harder than preview. That is intentional.

---

## 9. State Fingerprints

`baseStateFingerprint` and `finalStateFingerprint` should be deterministic digests over a stable state projection.

Suggested fields:

- `identity.id`
- `memories[].id/content/importance/emotion/timeStamp`
- `beliefStates[].id/content/strength/evidenceCount`
- coordinate values
- boundary values
- metaState values
- reward/homeostasis summaries

Do not include non-deterministic timestamps unless already part of persisted memory identity. If generated memory timestamps remain time-based, V10.21 should first make generated memory seed timestamps deterministic from simulation seed.

## 10. Audit Model

### Audit Draft

```typescript
interface LongitudinalCommitAuditDraft {
  id: string;
  characterId: string;
  simulationId: string;
  timestamp: string;
  requestDigest: string;
  baseStateFingerprint: string;
  finalStateFingerprint: string;
  commitPolicy: LongitudinalCommitPolicy;
  changedPaths: string[];
  generatedMemoryIds: string[];
  beforeSummary: CompactStateSummary;
  afterSummary: CompactStateSummary;
  governance: LongitudinalCommitGovernanceDecision;
  warnings: string[];
  reasons: string[];
}
```

### Audit Storage

V10.20 recommends a dedicated longitudinal audit log rather than reusing editor patch history directly.

Reasons:

- Editor patch history is field-edit oriented.
- Longitudinal commit is generated-artifact oriented.
- Rollback semantics differ: initial version should remove generated memory ids, not reverse arbitrary scalar edits.

Future versions may expose both histories through a unified timeline, but the write models should remain separate.

---

## 11. Rollback Plan

The first rollback model should be narrow:

```typescript
interface LongitudinalRollbackPlan {
  id: string;
  simulationId: string;
  type: "remove_generated_memories";
  generatedMemoryIds: string[];
  baseStateFingerprint: string;
  finalStateFingerprint: string;
  staleWritePolicy: "block_if_changed";
  warnings: string[];
  reasons: string[];
}
```

### Rollback Rules

| Rule | Behavior |
|---|---|
| Generated memory ids still exist | Remove them in reverse chronological order |
| Generated memory was edited | Block rollback |
| New later memories depend on generated memory | Block or require explicit override in future version |
| Current state fingerprint differs | Block as stale |
| Self-action trace present | Should never occur; block and report corruption |

Rollback should continue history, not erase history.

## 12. API Roadmap

### V10.21 — Types Only

Add:

- `LongitudinalFinalStateForCommit`
- `LongitudinalCommitSurface`
- `LongitudinalCommitPreview`
- `LongitudinalCommitAuditDraft`
- `LongitudinalCommitAuditEntry`
- `LongitudinalRollbackPlan`
- fingerprint helpers

No route writes.

### V10.22 — Commit Preview

Add:

- `POST /api/characters/[characterId]/life/simulate/commit/preview`

Behavior:

- Requires auth or follows project policy for write previews.
- Runs simulation with commit policy on a clone.
- Returns `LongitudinalCommitPreview`.
- Stores no state.
- Optionally stores short-lived preview token only if needed later.

### V10.23 — Commit Apply

Add:

- `POST /api/characters/[characterId]/life/simulate/commit/apply`

Behavior:

- Requires auth.
- Requires preview digest or equivalent request replay.
- Re-runs simulation from current state or consumes private preview state only if safe.
- Checks base fingerprint.
- Persists through `characterPhysicsService.replaceState()`.
- Records longitudinal audit.
- Returns no full state.

### V10.24 — Commit Rollback

Add:

- `POST /api/characters/[characterId]/life/simulate/commit/rollback`

Behavior:

- Requires auth.
- Uses rollback plan and stale-write protection.
- Records a new audit entry.

## 13. DTO Privacy

Public API must never expose:

- Full `CharacterPhysicsState`
- `clusters` map
- `particles`
- `proceduralRoutines`
- full memory vectors
- full life tick trace arrays
- private `finalState`

Public API may expose:

- compact summaries
- generated memory ids and snippets
- change counts
- warning/reason strings
- governance decisions
- fingerprints/digests

---

## 14. Benchmark and Integrity Gate

Full V2.1 benchmark should not run inside every commit apply request. It is too expensive and unrelated to a single character write. Instead:

- Core structural integrity check should run synchronously.
- Commit surface invariants should run synchronously.
- Optional offline regression should remain part of release QA.
- Future "high risk commit" mode may require a sampled benchmark or fixture subset.

V10.20 design keeps benchmark as governance evidence, not request-time friction.

## 15. Failure Modes

| Failure | Required Behavior |
|---|---|
| Invalid request | 422 |
| Unknown character | 404 |
| Auth missing for apply | 401/403 |
| Preview base state stale | 409 |
| Commit surface includes self-action | 422 |
| Integrity error after commit projection | 422 |
| Audit write fails | Block apply |
| Rollback plan missing | Block apply |
| finalState accidentally appears in public DTO | Test failure |

## 16. Testing Plan

V10.21 should add type/helper tests:

- fingerprint deterministic for same state
- fingerprint changes when generated memory is appended
- public preview strips `finalState`
- commit surface contains changes/skipped only
- self-action candidate cannot enter commit surface
- rollback plan contains generated memory ids
- stale fingerprint blocks apply decision

V10.22 should add preview route tests:

- valid preview returns commit surface
- preview no mutation
- preview no full state leak
- same request same seed same digest
- commit deferred warning replaced by preview-ready status

V10.23 should add apply route tests:

- apply requires auth
- apply persists generated memories
- apply records audit
- apply blocks stale state
- apply blocks self-action candidate commits
- apply returns compact response only

V10.24 should add rollback tests:

- rollback removes generated memory ids
- rollback preserves audit history
- rollback blocks stale state
- rollback blocks edited generated memory

## 17. Known Limitations

- No `lifeState` field exists yet. Energy/fatigue, sleep/wake, and boredom writeback remain blocked.
- Generated memory seed timestamps currently come from `new Date().toISOString()` in `lifeTickPersistence.ts`; deterministic commit design should address this before persistence opens.
- Rollback can initially cover appended generated memories only.
- Longitudinal commit history should be separate from editor patch history until a unified timeline is designed.
- No API commit persistence should be enabled until V10.23 or equivalent apply phase.

---

## 18. Success Criteria for Future Apply

Commit persistence may be opened only when all are true:

- `finalStateForCommit` private surface exists.
- Public DTO stripping is tested.
- Auth gate is enforced.
- Base state fingerprint stale-write protection exists.
- Audit write is mandatory.
- Rollback plan is generated before apply.
- Self-action candidates remain unexecuted and unpersisted.
- Generated memory caps are enforced.
- Structural integrity passes.
- `npm run build`, `npm test`, and `npm run next:build` pass.

## 20. Final Statement

V10.20 keeps the persistence door closed while designing the hinges, lock, audit trail, and rollback handle.

The mainline remains intact:

```text
Single Character
→ Experiences shape memory and belief
→ Continuous life produces signals
→ Life context influences decisions
→ Longitudinal simulation observes possible futures
→ Commit remains explicit, audited, reversible, and deferred until finalStateForCommit is implemented safely
```
