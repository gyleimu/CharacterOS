## 1. Module

## 2. Writeback Plan vs Writeback Execution

| Writeback Plan (V12.7) | Writeback Execution (V12.8+) |
|------------------------|------------------------------|
| Generates plan structure | Calls EventStudioApply |
| Selects candidate | Mutates state (with audit) |
| Drafts audit entry | Creates audit trail |
| `prepare_pending` / `ready_for_apply` | `applied` |
| No mutation | Confirmation-gated |

## 3. Status Lifecycle

```
blocked → (never proceeds)
preview_pending → (preview exists, user sees it)
confirmation_pending → (waiting for user "apply")
ready_for_apply → (all checks passed, can apply)
applied → (V12.8+ — actual writeback execution)
```

## 4. Candidate Selection

Score = confidence - safety_penalty + preview_bonus

- `possible_diagnosis_claim`: -0.5 penalty
- Preview PASS: +0.15 bonus
- Preview FAIL: -0.5 penalty
- Score < 0.2: not selected
- Multiple candidates: highest score wins

## 5. Audit Draft Fields

```
writebackId, sessionId, turnId, selectedCandidateId,
policyDecision, confirmationRequired, previewRefs,
realityAuditVerdict, safetyFlags, reasons, status
```

## 6. Confirmation Model

- `applyRequiresConfirmation: true` by default
- Only `autoApplySkipConfirmation: true` + `apply_allowed` can bypass
- Block, preview_only, confirmation_required: never bypass
