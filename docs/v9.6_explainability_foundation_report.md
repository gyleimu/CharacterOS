## 1. Motivation

### Why Now

CharacterOS has crossed a critical threshold in V9. Prior to V9, the system was read-only — state was generated from blueprints and evolved via simulation ticks, but was never directly edited by users.

V9.1–V9.5 introduced:

- `POST /editor/preview` — preview a patch before applying
- `POST /editor/apply` — mutate character state
- `GET /editor/history` — audit trail
- Integrity policy, risk assessment, domain mapping, clamp detection

With modification capability comes the obligation to **explain** every mutation. Without explainability, CharacterOS becomes an uncontrollable black box.

### What Explainability Is (and Is Not)

- **Is:** A structured, machine-readable trace of *why* something happened
- **Is:** A debugging tool for developers and authors
- **Is:** Deterministic — same input → same explanation (except timestamps)
- **Is:** Pure data — no UI, no natural language generation, no LLM

- **Is not:** A "Why panel" with conversational explanations
- **Is not:** Marketing copy or natural language narrative
- **Is not:** Complete psychological interpretation of state changes

## 4. Type System

### ExplanationScope

```typescript
type ExplanationScope =
  | "patch" | "decision" | "memory_activation" | "cluster_influence"
  | "belief_evolution" | "need_change" | "desire_change"
  | "behavior_bias" | "state_transition" | "integrity_policy";
```

### ExplanationTrace

The central data structure. Every explainability function returns one.

```typescript
interface ExplanationTrace {
  id: string;                    // Unique trace id
  scope: ExplanationScope;       // Which system generated this
  title: string;                 // Short human-readable title
  summary: string;               // One-sentence summary
  reasons: ExplanationReason[];  // Why things happened
  facts: ExplanationFact[];      // Supporting data (deduplicated)
  warnings: string[];            // Standalone warning messages
  createdAt: string;             // ISO timestamp
}
```

### ExplanationReason

A single atomic reason within a trace.

```typescript
interface ExplanationReason {
  id: string;
  message: string;
  scope: ExplanationScope;
  severity: "info" | "warning" | "danger" | "blocked";
  confidence: "low" | "medium" | "high";
  supportingFacts: ExplanationFact[];
}
```

### ExplanationFact

A structured fact that can support one or more reasons.

```typescript
interface ExplanationFact {
  id: string;
  label: string;
  value: unknown;
  source: "state" | "patch" | "validation" | "integrity" | "graph" | "benchmark" | "derived";
}
```

### DecisionTrace (Skeleton)

```typescript

## 5. Functions

### explainPatchPreview(input) → ExplanationTrace

**When:** POST /editor/preview

**Input:**
- `patch` — the proposed patch
- `validation` — validation result
- `affectedDomains` — domains the patch touches
- `riskSummary` — risk assessment
- `clampedChanges` — any out-of-range values being clamped
- `integrity?` — post-edit integrity report

**What it explains:**
1. How many changes in the patch
2. Which domains are affected
3. Whether validation passed
4. Risk severity and count breakdown
5. Why each value was clamped (if any)
6. Post-edit integrity check result
7. Overall verdict: allowed / allowed with warnings / blocked

**Deterministic:** Same input always produces the same reasons and facts (except id and createdAt).

### explainPatchApply(input) → ExplanationTrace

**When:** POST /editor/apply

**Input:**
- `patch` — the applied/blocked patch
- `applied` — whether it was actually applied
- `validation` — validation result
- `affectedDomains` — domains affected
- `riskSummary` — risk assessment
- `clampedChanges` — clamped values
- `integrityPolicy` — integrity policy decision

**What it explains:**
1. Whether the patch was applied or blocked
2. If blocked, why (validation / risk / integrity)
3. Integrity policy status (pass / soft_warning / block)
4. All clamped changes with before/after values
5. Audit log relevance
6. Domain impact

### explainStateTransition(summary) → ExplanationTrace

**When:** State changes (preview projection or actual apply)

**Input:**
- `changedPaths` — which paths changed
- `beforeSummary` — values before
- `afterSummary` — values after
- `affectedDomains` — derived domain mapping

**What it explains:**
1. How many paths changed
2. Per-path before → after values
3. Which domains were affected

**Does NOT infer:** psychological meaning, personality impact, or behavioral consequences.

    "reasons": [ ... ],
    "facts": [ ... ],
    "warnings": [ ... ],
    "createdAt": "..."
  }
}
```

### Apply Response (new `explanation` field)

Same structure, added to the success response body when a patch is applied or blocked.

### Safety

Both integrations use try/catch — if explanation generation fails for any reason, the API still returns a valid response with `explanation: null`.

## 9. Design Decisions

### Deterministic, not AI-generated

All explanations are generated from structured data with simple template strings. This ensures:
- Repeatability — same patch always produces same explanation
- Testability — assertions can verify exact reason content
- Performance — no LLM latency or token cost
- Reliability — no hallucination risk

### Pure data, no UI

The `ExplanationTrace` and related types are pure data structures. Rendering them (as tables, panels, diffs, etc.) is a UI layer concern for V9.7+.

### Non-blocking API integration

The `explanation` field is added to API responses inside a try/catch. If explanation generation fails (e.g., due to unexpected input shapes), the API continues to work — the field simply becomes `null`. This respects the principle that explainability is a debugging tool, not a critical path dependency.

### Scope expansion designed for gradual rollout

All 10 `ExplanationScope` values are defined in V9.6, but only `patch` and `state_transition` are fully implemented. The remaining 8 scopes share the same type infrastructure and can be implemented incrementally without schema changes.

## 11. Philosophy

> Complex systems cannot only produce results. They must explain their changes.

CharacterOS explainability is not marketing copy. It is a debugging tool. It helps developers and authors understand:

- Why did this character become this way?
- Why did this decision happen?
- Why was this edit allowed?
- Why does this risk exist?

If CharacterOS is to simulate life, it must also preserve comprehensible traces.
