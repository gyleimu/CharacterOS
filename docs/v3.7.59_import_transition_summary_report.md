# CharacterOS V3.7.59 Import Transition Summary Report

## 目标

本阶段继续打磨角色导入应用链路。

V3.7.58 已经能记录导入替换前后的完整性报告。V3.7.59 在此基础上增加一个更适合 API、日志和人工审计使用的摘要层：成功应用导入时，trace 会返回稳定的 `transitionId` 和简洁的 `transitionSummary`。

## 新增能力

`CharacterImportApplyTrace` 新增：

```text
transitionId
transitionSummary
```

`transitionSummary` 包含：

```text
packageDigest
before
after
delta
beforeValid
afterValid
errorDelta
warningDelta
```

其中 `delta` 会比较：

```text
memoryCount
particleCount
clusterCount
beliefCount
proceduralRoutineCount
```

## Transition ID

`transitionId` 使用以下信息生成：

```text
targetCharacterId
sourceCharacterId
packageDigest
beforeStateIntegrity.summary
afterStateIntegrity.summary
before/after validity
before/after error and warning counts
```

输出形式：

```text
import_<16位sha256前缀>
```

它不是数据库主键，也不依赖时间戳。它用于描述“这一次状态替换的结构性身份”，方便未来接入导入事件日志、持久化事务或人工审计列表。

## 行为边界

成功导入：

```text
has transitionId
has appliedAt
has transitionSummary
has beforeStateIntegrity
has afterStateIntegrity
```

失败导入：

```text
no transitionId
no appliedAt
no transitionSummary
no afterStateIntegrity
no state mutation
```

## 为什么重要

CharacterOS 当前正在删除可视化依赖，回到纯核心架构。

在没有前端页面的阶段，所有高风险操作都必须能被 API trace 解释清楚。导入一个角色包本质上是在替换一段人格历史、记忆星系和参数调整历史，因此系统需要能用很少的字段回答：

- 这次替换是哪一次？
- 替换来自哪个包？
- 替换前后结构规模如何变化？
- 替换后是否仍然完整？

V3.7.59 让导入应用链路开始具备事件日志雏形。
