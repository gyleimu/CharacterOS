## Objective

Define the minimal type system and scheduler model for Continuous Life in CharacterOS. V10.1 is a **foundation-only** phase: it establishes what a "life tick" is at the type level, without executing any state change.

---

## Life Tick vs. V4 Continuous Tick

| Dimension | V4 Continuous Tick | V10 Life Tick |
|---|---|---|
| Purpose | Time-based recovery / decay / homeostasis | One slice of lived time |
| Input | `CharacterPhysicsState` (mutated in-place) | `LifeTickRequest` (pure) |
| Output | `ContinuousTickTrace` (full state delta) | `LifeTickPlan` + `LifeTickTrace` skeleton |
| State | Mutates memories, boundary, reward, beliefs, etc. | No state mutation (V10.1) |
| Phases | 17 ordered phases (snapshot...time_perception) | 10 ordered phases (passive_recovery...trace_summary) |
| Scope | Physics engine | Life simulation (non-agentic, non-LLM) |

**Key distinction:** V4 is a physics engine. V10 is a life simulation layer built on top. V10.1 only defines the interface between them.

## Architecture

### LifeTickRequest
```
characterId, elapsedHours (1–168), observed, seed?, requestedAt, mode
```
`mode` is `"dry_run"` (plan only) or `"commit_later"` (reserved for future).  
`observed=false` means the character changes without being watched.

### LifeTickPlan
```
id, characterId, elapsedHours, phaseSequence, timeScale, seed, dryRun, warnings, reasons
```
- `phaseSequence` always equals `LIFE_TICK_PHASES` (10 phases in stable order)
- `timeScale` derived: ≤6h → `"short"`, ≤48h → `"daily"`, >48h → `"multi_day"`

### LifeTickTrace Skeleton
```
id, characterId, planId, elapsedHours, observed, phaseTraces[],
generatedInternalEvents[], selfActionCandidates[], stateChanged: false
```
- `stateChanged` is **always `false`** in V10.1 — no execution
- `LifePhaseTrace.executed` is **always `false`** in V10.1

### Phase Sequence (10 phases)
1. `passive_recovery` — baseline recovery
2. `energy_fatigue` — energy depletion
3. `attention_drift` — attention wandering
4. `boredom_pressure` — boredom accumulation
5. `memory_resurfacing` — spontaneous memory recall
6. `random_thought` — unbidden thoughts
7. `inspiration_check` — creative insight probability
8. `self_action_candidate` — generate self-action options
9. `quiet_drift` — no significant change
10. `trace_summary` — summarize what happened

### InternalEvent Types (9)
`dream_fragment`, `random_thought`, `memory_resurfacing`, `boredom_spike`, `inspiration_flash`, `fatigue_wave`, `loneliness_wave`, `emotional_settling`, `quiet_realization`

- All have `evaluated: false` in V10.1

### SelfActionCandidate Types (9)
`check_phone`, `avoid_message`, `write_note`, `go_for_walk`, `sleep`, `revisit_memory`, `seek_contact`, `withdraw`, `do_nothing`

- All have `evaluated: false` in V10.1
- `probability` field indicates likelihood, but no selection is made

### SeededRandom
- Simple LCG (Linear Congruential Generator): multiplier=1664525, increment=1013904223, modulus=2^32
- String seed → djb2 hash → initial LCG state
- `next()` returns `[0, 1)` — deterministic, reproducible
- **No `Math.random` anywhere in the Life scheduler**

### Scheduler Policy
- `validateLifeTickRequest`: returns `{ valid, errors[], warnings[] }`
- `buildLifeTickPlan`: pure function, same input → same output
- No state read, no state write
- Throws on invalid request
- `seed` preserved if provided; generated deterministically if absent

## Why V10.1 Does Not Modify State

V10.1 is the **grammar** of life, not the living. Before a character can live through a tick, the system must agree on:

1. **What a tick is** (type system)  
2. **How a tick is requested** (request → plan)  
3. **How a tick is traced** (trace skeleton)  
4. **What events can occur** (internal event types)  
5. **What actions are candidates** (self-action types)  
6. **How randomness is seeded** (deterministic RNG)

State mutation begins in V10.2+ — energy/fatigue, recovery curves, and actual phase execution.

## Design Philosophy

In CharacterOS, continuous life must be:

- **Explicit** — every tick is intentionally requested
- **Traceable** — every tick produces a trace
- **Seeded** — all randomness is reproducible
- **Bounded** — elapsed time has limits (max 1 week)
- **Single-character** — no multi-agent simulation
- **Non-agentic** — no autonomous action
- **Non-LLM** — no generative model calls

The character can change without being observed.  
But the system must know: **when**, **why**, **through which phases**, **what was candidate**, and **what was not executed**.

> First the beat. Then the life.
