## 5. LOD Opacity Curves

### How they work

Each node type has a "home level" (the zoom level where it first becomes visible). When zoom enters that level's band, the node's opacity transitions from 0→1 over the first 20% of the band.

```
Opacity
  1.0 ┤                    ╭──────────────
      │                   ╱
      │                  ╱
  0.5 ┤                ╱
      │              ╱
      │            ╱
  0.0 ┤──────────╱
      └─────────┼─────────┼────────────── Scale
              band.min   band.fullOpacityAt   band.max
               ← 20% transition zone →
```

### Node type home levels

| Home Level | Node Types |
|------------|-----------|
| L0 | personality_core |
| L1 | impact_cluster, temporal_process |
| L2 | memory, impact_particle, belief, need, desire |
| L3 | behavior_bias, internal_state_variable, benchmark_signal |
| L4 | (all fully visible, drift vectors added) |

### Opacity behavior

- **Before home level**: opacity = 0 (invisible)
- **At home level, in transition zone**: opacity = progress from 0→1
- **At home level, after transition**: opacity = 1
- **After home level**: opacity = 1 (fully visible)

## 6. Viewport Transform Math

```
Screen → World:
  worldX = (screenX - panX) / scale
  worldY = (screenY - panY) / scale

World → Screen (inverse):
  screenX = worldX * scale + panX
  screenY = worldY * scale + panY
```

Round-trip is lossless (within floating-point precision).

## 7. Zoom Transition Interpolation

Linear interpolation between two clamped scales:

```
scale(t) = clamp(from + (to - from) * t)
```

- `t = 0` → `fromScale`
- `t = 1` → `toScale`
- `t` outside [0, 1] is clamped
- Both `from` and `to` are clamped before interpolation

This is the DATA for a transition — the future V10.36 animation system calls this every frame with an easing-modified `t`.

## 9. Relationship to V10.34 Renderer

The Zoom/LOD system feeds the render command builder:

```text
GalaxyZoomState
  │
  ├──▶ computeGalaxyLodVisibility(type, state) → opacity per node/edge
  │
  └──▶ V10.34 render commands can use opacity for fade-in
       (currently binary visible/hidden — V10.36 can add per-command opacity)
```

No changes to V10.34 are required. The renderer's `getVisibleGalaxyNodes/Edges` already uses the zoom level for binary visibility. The opacity system here adds the smooth transition layer that can be applied to command opacity in V10.36.
