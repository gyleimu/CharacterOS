**Date:** 2026-06-28

---

## 1. Executive Summary

V10.28 responds to the V10.20-V10.27 code review.

The work focuses on correctness and robustness in the longitudinal commit lifecycle:

```text
preview -> apply -> rollback
```

No new product capability was added. The changes harden existing commit behavior.

---

## 2. Findings Addressed

| Finding | Action |
|---|---|
| Rollback conflict semantics mixed with blockers | Cleaned readiness semantics so applied-audit fingerprint drift returns `conflict` and is tested |
| Failed apply retry could preserve misleading audit mutation | Fixed service behavior so existing `applied` audit entries are preserved and not rewritten by blocked retry attempts |
| State write and audit write partial-failure risk | Added pre-write audit materialization for apply and best-effort state compensation around apply/rollback write sequences |
| Fingerprint omitted `biologicalNature` and `personality` | Added both fields and compile-time `Record<keyof CharacterPhysicsState, unknown>` coverage guard |
| Route limit constants repeated | Added shared longitudinal simulation limit constants |
| Preview/apply route helper duplication | Extracted shared commit route helpers for simulation request construction and final committed state extraction |
| Corrupt audit JSON was silently discarded | Corrupt audit files are now renamed to `.corrupt-*` before falling back to an empty store |

---

## 3. Important Boundary

V10.28 improves normal write-failure handling, but it does not claim full cross-file ACID transactions.

`physics_states.json` and `longitudinal_commit_audit.json` remain separate files. The service now reduces partial-write risk through:

- audit materialization before apply state replacement
- applied audit update after state replacement
- best-effort state restoration if the later audit write throws
- corrupt audit file preservation instead of silent overwrite

Crash-level atomicity across two files remains a known limitation until a transactional repository layer exists.
