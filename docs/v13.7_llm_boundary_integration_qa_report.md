# V13.7 LLM Boundary Integration QA

## Integration Path

```text
AgentReplyPlan
+ AgentGroundingBundle
+ AgentSessionConfig
+ AgentPolicyDecision
-> Safety Preflight
-> Sanitized LlmBoundaryRequest
-> Grounded LlmBoundaryPrompt
-> Offline Mock ProviderConfig
```

The integration produces boundary DTOs only. It performs no provider call, generates no final prose, and has no state mutation or writeback authority.

## Safety Gates

- `allowLlm` requires explicit caller opt-in.
- Session mode must be `planned_boundary_only`.
- Reply plan must explicitly allow the LLM boundary.
- Policy must not be blocked.
- Safety preflight must pass.
- Provider config always uses `networkAllowed=false` at this stage.
- Raw-state keys, coordinate assignments, memory payloads, secrets, writeback claims, and diagnosis claims are detected and redacted.

## Determinism

Identical Agent inputs produce identical request and prompt identifiers. No runtime clock or random source participates in this integration path.

## Test Coverage

The integration tests cover default-off behavior, complete opt-in gating, blocked policy, disabled session mode, raw-state detection, coordinate redaction, secret redaction, diagnosis and writeback blocking, deterministic replay, input immutability, turn-result adaptation, Chinese grounding preservation, and contract summary output.
