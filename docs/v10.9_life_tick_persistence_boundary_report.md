# V10.9: Life Tick Persistence Boundary — Report

**Date:** 2026-06-24
**Phase:** V10.9 — Explicit, Conservative Write-Back
**Status:** ✓ Complete

## Objective

Establish the persistence boundary between V10 life tick projections and `CharacterPhysicsState`. V10.9 defines *what can be written back, what cannot, and what requires schema migration.*

The boundary, not the write, is the accomplishment.

---

## Why Persistence Is Separate from Dry-Run

| Layer | V10.8 (Dry-Run) | V10.9 (Commit) |
|---|---|---|
| Purpose | Observe what *would* happen | Choose what to *keep* |
| State access | Read-only | Clone + selective write |
| Default | Everything projected | Nothing written |
| Control | No caller choice | Per-domain explicit opt-in |
| Risk | Zero (no side effects) | Guarded (clone, audit trail) |

**Dry-run is observation. Commit is consent.** No life projection writes into character state unless the caller explicitly asks for it.

## Architecture

### Commit Options (all default: false)

| Option | Domain | Default |
|---|---|---|
| `allowEnergyFatigue` | Energy/fatigue state | false → skipped (no schema field) |
| `allowSleepWake` | Sleep/wake state | false → skipped (no schema field) |
| `allowBoredomExpansion` | Boredom expansion state | false → skipped (no schema field) |
| `allowDreamMemorySeed` | Dream fragments → memory seeds | false |
| `allowInspirationSeed` | Inspiration seeds → memory seeds | false |
| `allowRandomThoughtMemorySeed` | Random thoughts → memory seeds | false |
| `allowSelfActionCandidateMemorySeed` | Top candidate → trace seed | false |
| `maxGeneratedMemories` | Cap on generated memories | 3 |

### Commit Change

```typescript
{ path: string; from: unknown; to: unknown; reason: string }
```

Every change — applied or skipped — is tracked with full audit trail.

### Commit Result

```typescript
{
  applied: boolean;      // true only if at least one change was written
  state: CharacterPhysicsState;  // cloned state (input never mutated)
  changes: LifeTickCommitChange[];
  skipped: LifeTickCommitChange[];
  warnings: string[];
  reasons: string[];
}
```

## Persistence Policy

### What CAN be written (when explicitly enabled)

| Source | Destination | Format |
|---|---|---|
| Dream fragment | `state.memories[]` | Internal memory seed: `"Dream residue: {description} (tone: {tone})"` |
| Inspiration seed | `state.memories[]` | Internal memory seed: `"Inspiration seed: {trigger} (type: {type})"` |
| Random thought | `state.memories[]` | Internal memory seed: `"Thought: \"{phrase}\" (kind: {kind})"` |
| Top self-action candidate | `state.memories[]` | Trace seed: `"Action tendency: {type} (score: {score})"` |

### What CANNOT be written (requires schema migration)

| Domain | Reason |
|---|---|
| Energy/fatigue state | No `lifeState` field on `CharacterPhysicsState` |
| Sleep/wake state | No `lifeState` field on `CharacterPhysicsState` |
| Boredom expansion state | No `lifeState` field on `CharacterPhysicsState` |

These are **skipped** with clear warnings. Even when `allowEnergyFatigue=true`, the system explains *why* it cannot write — the schema doesn't support it yet.

### What NEVER happens

- Self-action candidates are **never executed**
- `CharacterPhysicsState` is **never mutated** (clone only)
- Memories are **never replaced** — only appended
- V3-V9 behavior is **preserved** — no regression

---

## Design Philosophy

> The beauty of V10.9 is not "it finally writes back." It is "it knows what NOT to write back."

A persistence boundary that says "no" is more trustworthy than one that eagerly writes everything. V10.9 establishes:

- **Per-domain consent** — every write requires explicit opt-in
- **Full audit trail** — every applied and skipped change is tracked
- **Self-action firewall** — candidates are never executed, regardless of options
- **Schema honesty** — when a field doesn't exist, the system says so instead of silently failing or forcing migration

## Known Limitations

- Energy/fatigue, sleep/wake, boredom expansion cannot be written to `CharacterPhysicsState` — these require a future `lifeState` field or separate `CharacterLifeState` object
- Memory seeds are small and neutral (importance ~0.1–0.2) — they won't dominate the memory pool
- No bulk commit — each domain must be opted in individually
- No two-phase commit — applied changes are immediate on the clone
