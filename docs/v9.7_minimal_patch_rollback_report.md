## 1. Motivation

V9.4 gave CharacterOS the ability to mutate character state via the apply API. With mutation comes the need to undo. However, CharacterOS does not store full state snapshots — audit entries contain only `beforeSummary` and `afterSummary` for the specific paths that were changed.

V9.7 implements the simplest possible rollback that respects this constraint: a reverse patch generated from the audit entry, flowing through the same validate → apply → audit pipeline as any other edit.

### What Rollback Is (and Is Not)

- **Is:** A new, recorded edit that reverses a prior edit
- **Is:** A reverse patch flowing through the standard pipeline
- **Is:** Protected by stale-write detection

- **Is not:** A time machine that restores arbitrary past state
- **Is not:** A full snapshot restore
- **Is not:** A multi-step undo chain
- **Is not:** A version tree or branch system

## 2. Design

### Core Function: `createRollbackPatchFromAuditEntry`

```typescript
export function createRollbackPatchFromAuditEntry(
  entry: PatchAuditEntry
): CharacterEditPatch
```

**Rules:**

1. For each path in `entry.changedPaths`, create a reverse `PatchChange`:
   - `from` = `entry.afterSummary[path]` (value after original patch)
   - `to` = `entry.beforeSummary[path]` (value before original patch)
   - `reason` = `Rollback audit entry ${entry.id}`

2. If any path is missing from `beforeSummary` or `afterSummary`, throw a controlled error — rollback is impossible without sufficient data.

3. Generate new `patch.id` and `description = Rollback ${entry.patchId}`.

4. All changes flow through `validatePatchAgainstState`, which provides stale-write protection:
   - `from` must match current state value
   - If another edit has modified the same path since the original patch, the rollback fails with 422

### API Route

```
POST /api/characters/[characterId]/editor/rollback
```

**Request:**
```json
{
  "auditEntryId": "audit_...",
  "dryRun": false
}
```

**Response (success):**
```json
{
  "valid": true,
  "rolledBack": true,
  "dryRun": false,
  "auditEntryId": "audit_...",
  "rollbackPatch": { "id": "patch_...", "description": "Rollback patch_...", "changes": 1 },
  "validation": { ... },
  "integrity": { ... },
  "integrityPolicy": { ... },
  "changedPaths": [ ... ],
  "affectedDomains": [ ... ],
  "riskSummary": { ... },
  "clampedChanges": [ ... ],
  "projectedStateSummary": { ... },
  "warnings": [ ... ],
  "reasons": [ ... ],
  "audit": { "id": "audit_...", "recorded": true, "timestamp": "..." },
  "explanation": { ... }
}
```

### Pipeline

```
requireAuth
  ↓
load state (404 if missing)
  ↓
find audit entry by auditEntryId (404 if not found)
  ↓
createRollbackPatchFromAuditEntry (422 if insufficient data)
  ↓
validatePatchAgainstState(state, rollbackPatch)
  ↓
422 if invalid or risk severity = error
  ↓
applyValidatedPatch to clone (500 if failed)
  ↓
integrity check + policy evaluation
  ↓
If dryRun → return with rolledBack=false, no state mutation
  ↓
replaceState + recordPatchAudit
  ↓
return rolledBack=true with explanation
```

## 4. Audit Behavior

Rollback creates a **new** audit entry. It does not delete or modify the original entry.

Audit log after apply + rollback:

| Entry | Type | Applied |
|---|---|---|
| `audit_1` | Original apply | true |
| `audit_2` | Rollback of audit_1 | true |

The rollback entry includes a warning referencing the original audit entry, preserving the full edit lineage.

**Philosophy:** History is not deleted — it is continued. If a user undoes an edit, the system remembers both that the edit happened AND that it was undone.

## 5. Limitations

| Limitation | Reason |
|---|---|
| Single-step rollback only | No snapshot storage, no multi-undo chain |
| Stale writes block rollback | If another edit touches the same path, rollback fails (by design) |
| No partial rollback | Entire original patch must be reversible |
| No non-numeric paths | Only paths with numeric values in beforeSummary/afterSummary |
| No UI | Deferred to V9.8+ |

## 8. Design Decisions

### Reverse patch, not snapshot restore

Audit entries store only `beforeSummary` and `afterSummary` for changed paths — not full state. A reverse patch is the natural rollback primitive: it uses the same type system, the same validation, and the same audit recording.

### Stale-write protection on rollback

The rollback patch goes through `validatePatchAgainstState`, ensuring `from` matches current state. If another edit has modified the same path since the original patch, the rollback fails. This prevents silent data corruption from concurrent edits.

### No partial rollback

If any path in the original patch is missing from the summary, the entire rollback is rejected. Partial rollback would leave the character in an inconsistent state relative to the audit trail.

### Rollback is a new edit

Rollback does not erase history. It adds a new entry to the audit log. The character's edit history is a linear log — not a tree of branches — and rollback is simply the next entry in that log.
