## Objective

Wire all V10.1–V10.7 subsystems into a single deterministic, non-mutating life tick execution. This is the first time the character "lives" a complete tick — but only in projection. No state is written. The foot hovers; it has not yet touched the ground.

---

## Why Dry-Run Before Persistence

Before CharacterOS can commit a life tick to persistent state, the system must prove:

1. **All subsystems compose correctly** — each phase reads from prior projections without conflicts
2. **Determinism holds** — same inputs → same projections every time
3. **No leaks** — `CharacterPhysicsState` is never mutated, even transitively
4. **Trace is complete** — every phase produces reasons, warnings, and change keys
5. **The chain is visible** — from request → plan → projected state → trace

Dry-run is the dress rehearsal. Persistence (V10.9) comes after the choreography is proven.

## Integrated Subsystem Order

```
 1. scheduler          — validate request, build LifeTickPlan
 2. energy_fatigue     — tick energy/fatigue from CharacterPhysicsState context
 3. sleep_wake         — tick sleep/wake using projected energy/fatigue
 4. dream              — select sources + generate dream fragment if sleeping
 5. boredom_inspiration — tick boredom expansion + generate inspiration seeds
 6. random_thought     — select sources + generate thought from all prior signals
 7. self_action_candidate — score all candidates from full projected state
 8. summary            — aggregate warnings, reasons, phase traces
```

**Upstream → downstream data flow:**

```
CharacterPhysicsState (READ ONLY)
  → energyFatigueContext → projectedEnergyFatigue
    → sleepWakeContext → projectedSleepWake
      → dreamContext → dreamFragments
    → boredomContext → projectedBoredom → inspirationSeeds
      → randomThought → (optional)
        → selfActionCandidateContext → candidates
```

## Key Guarantees

- **`applied` is always `false`** — no state persistence
- **No mutation** — `CharacterPhysicsState` is read-only; all operations are on cloned or projected state
- **Deterministic** — same request + same character state + same seed → identical result
- **Seed flow** — base seed flows through all subsystems with deterministic derivations (`:dream`, `:insp`, `:rt`)
- **Graceful degradation** — invalid requests produce an error result instead of throwing
- **Full trace** — every phase produces `LifePhaseTrace` entries with reasons/warnings

## Known Limitations

- `stimulationLevel` and `socialContactLevel` are fixed defaults (0.4) — no world-model source yet
- Character starts from scratch each tick (default subsystem states) — no persistent life state carried forward
- `attention_drift`, `quiet_drift` phases are mapped to sleep/dream but not independently modeled
- No option to override initial subsystem states (always starts from defaults)
- `localHour` defaults to `requestedAt` hour — no timezone handling

## Next Steps: V10.9 — Life Tick Persistence Boundary

Recommended progression:

1. Define `CharacterLifeState` — a separate persistent state object for life-layer fields
2. Implement `commitLifeTick` — write projected changes back under explicit caller control
3. Add `mode: "commit_later"` support — store plan for later commitment
4. Validate that persisted state stays within bounds
5. Add `lifeStateId` to `LifeTickTrace` for audit trail
6. Still no autonomous execution — caller always controls when ticks happen and when they commit
