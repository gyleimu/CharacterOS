# CharacterOS V3.8 Project Progress Report

## 当前版本

```text
CharacterOS V3.8 Import Transaction Boundary
```

V3.8 不是新的人格理论扩张，也不是前端恢复版本。

本阶段目标是把 V3.7 后半段建立的导入/导出、状态完整性、导入审计和导入事务链路收束成一个可 review 的基础设施阶段。

## 当前主线判断

CharacterOS 当前方向仍然正确：

```text
single character
-> Character Physics Core
-> state integrity
-> export/import audit
-> transaction trace
-> API-only observability
```

项目没有恢复用户可见前端，也没有重新开始做 3D 可视化。当前重点仍然是核心架构、纯逻辑、测试和 API。

## V3.8 完成内容

### 1. Import Transaction Steps

`CharacterImportApplyTrace` 现在包含：

```text
transactionSteps
```

成功导入步骤：

```text
authorization_checked
before_state_integrity_inspected
state_deserialized
after_state_integrity_inspected
state_replaced
adjustment_history_replaced
history_recorded
```

blocked 导入步骤：

```text
authorization_checked
history_recorded
```

这让导入不再是黑箱替换，而是具备最小事务骨架。

### 2. Import Transaction Summary

`CharacterImportApplyTrace` 新增：

```text
transactionSummary
```

包含：

```text
terminalStatus
totalSteps
completedSteps
blockedSteps
failedSteps
stateMutated
adjustmentHistoryMutated
historyRecorded
lastStep
reasons
```

这让 API 调用方不必手动遍历 steps，就可以判断：

- 这次导入是否完成。
- 是否被授权阻止。
- 是否发生失败。
- 是否真的替换了角色状态。
- 是否替换了参数调整历史。
- 是否写入导入历史。

### 3. Import Transition History

新增角色级导入尝试历史：

```text
GET /api/characters/[characterId]/import/history
```

每次 `/import/apply` 都会记录一条历史，无论 applied 还是 blocked。

历史 entry 与 apply trace 直接关联：

```text
trace.historyEntryId
trace.historyRecordedAt
historyEntry.id
historyEntry.createdAt
historyEntry.trace.historyEntryId
```

### 4. Import Package Integrity

导入链路现在包含多层护栏：

```text
package shape validation
state deserialization
state integrity inspection
embedded stateIntegrity snapshot comparison
packageDigest comparison
auditSummary
explicit confirmation phrase
transaction trace
transition history
```

其中 digest mismatch、integrity mismatch、高风险 adjustment governance 都不能通过普通确认短语直接应用。

### 5. File Repository Lock Stabilization

新增共享锁 helper：

```text
src/db/repositories/fileLock.ts
```

用于：

```text
FileCharacterPhysicsRepository
FileParameterAdjustmentHistoryRepository
FileCharacterImportTransitionHistoryRepository
```

改进点：

- Windows 下 lock 目录释放时重试清理。
- 遇到没有 owner 的 stale lock 会尝试清理。
- lock owner 现在写入当前进程 pid，用于识别死亡进程遗留的 stale lock。
- owner pid 仍然存活或无法确认死亡时，不主动删除 lock。
- 统一本地 JSON 仓库锁逻辑，减少重复实现。

### 6. Test Environment Isolation

`src/server/characterPhysicsServiceSingleton.ts` 现在在 Vitest 环境下使用内存仓库。

生产/开发仍使用文件仓库：

```text
data/physics_states.json
data/parameter_adjustment_history.json
data/import_transition_history.json
```

这样 API route 测试仍然经过真实 route 和 service，但不会并发污染同一组本地 JSON 文件。

## 已知限制

当前仍然不是完整数据库事务。

`importCharacterPackage()` 中的状态替换、调整历史替换、导入历史写入仍然是顺序执行的普通操作。V3.8 的 transaction trace 是事务骨架和审计表达，不是 ACID 事务。

当前仍然没有前端。

这是有意的。核心逻辑稳定之前，继续保持 API-only。

当前仍然是单角色内核。

没有多角色、关系网络、世界模拟，也没有恢复可视化。

## 当前评价

CharacterOS 已经从“人物心理决策引擎”推进到一个更稳的核心系统阶段：

```text
Character Blueprint
-> Character Physics State
-> State Integrity
-> Export Package
-> Import Plan
-> Import Apply
-> Transaction Trace
-> Transition History
```

这条链路对未来很重要，因为 CharacterOS 后续会越来越依赖可保存、可迁移、可回放、可审计的角色状态。

V3.8 的意义是：

```text
角色状态不只是能运行。
角色状态开始能被安全迁移和审计。
```
